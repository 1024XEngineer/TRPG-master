## 0. 一句话定义

一个**模组驱动的 AI 主持平台**:玩家上传 TRPG 模组,平台解析成结构化数据,AI 担任 DM(守秘人)主持游戏。核心是把**大模型的创造性叙事**与**确定性的规则/数值运算**解耦。

本文档以 **COC(克苏鲁的呼唤,第七版)** 为唯一示例贯穿说明。

---

## 1. 核心设计原则(准绳)

1. **确定性运算走引擎,创造性叙事走 LLM。** 数值、检定、骰点、资源增减一律由确定性代码执行;只有需要创造力的叙事、演绎、临场裁决才调用大模型。
2. **规则是数据,不是硬编码。** 引擎是"规则解释器",读取规则定义来运算。换规则系统 = 换数据,不改引擎代码。
3. **State 是唯一真相源。** LLM 永远不直接改 state,只能通过工具调用提议变更,由引擎裁决后写入。
4. **事实靠 state,叙事靠 memory,两者分离。**
5. **能结构化的就别用 LLM。** 凡能收敛成结构化输入/运算的都走确定性路径,把 LLM 调用压到最少。
6. **模组 > 设定库 > LLM 即兴。** 内容来源优先级:模组明确写的绝对优先;模组引用标准元素但没详述的查设定库补全;都没有的才由 LLM 按风格发挥。

## 3. 三个核心概念

一局游戏所需的内容可以拆成三部分:规则、设定、模组。三者在系统里走的路径不同,必须分清。

| 概念  | 举例                      | 系统中的形态         | 谁负责       |
| --- | ----------------------- | -------------- | --------- |
| 规则  | d100 检定、技能、战斗、SAN、魔法    | A/B/C 三类(见 §4) | 平台构建      |
| 设定  | 神话生物、禁书、1920 年代背景、装备、术语 | B/C 类          | 平台构建      |
| 模组  | 具体的地点、NPC、线索、结局         | 剧本 IR          | 用户上传 + 官方 |

**关键推论一:规则书需要拆为"规则"和"设定"。** 检定/战斗/SAN 是规则,神话生物、装备表、术语是设定。二者性质不同(见 §4 的 A/B/C 拆分)。

**关键推论二:COC 规则 ≠ 克苏鲁设定。** COC 是规则,默认搭配克苏鲁神话,但可以拆开。《半步之遥》用 COC 规则却跑民国佛教业报,与神话无关。所以平台顶层按**规则系统**(COC)分区,后续优先模组设定，或使用默认的克苏鲁。

**关键推论三:一局游戏实际需要的,是"规则 + 该模组 + 少量设定",不是整本书。** 模组是自包含的——KP 跑本子靠"脑子里的规则 + 手边这个本子",不重读整本规则书。系统同理:DM 运行时只需"引擎(规则)+ 模组 IR + 按需检索的设定",**不需要把整本规则书喂给 LLM**。

## 4. COC 规则书内容的 A/B/C 三类拆分 【本文档核心】

一本 COC 规则书里的内容**不是一类东西**,而是三种性质完全不同的东西。**混在一起处理是最大的认知错误。** 必须先拆成三类,每类走不同的技术路径。

### 4.1 拆分总览

| 类别            | 性质        | 举例(COC 规则书章节)                     | 技术路径                     | 是否喂给 LLM         |
| ------------- | --------- | --------------------------------- | ------------------------ | ---------------- |
| **A · 规则机制**  | "怎么运算"    | 检定系统、战斗系统、追逐轮、SAN 机制、疯狂发作、伤害计算    | **引擎代码/数据**(Rule Module) | **否**            |
| **B · 结构化数据** | "世界的事实清单" | 技能表、装备/武器表、神话生物属性、术语表、职业表         | **结构化数据库** + 硬约束         | 否(引擎查表;部分按需检索)   |
| **C · 风格基调**  | "氛围和品味"   | 宇宙恐怖、1920 年代语气、"不可名状"美学、调查而非战斗的基调 | **提示词 + 少量范例**           | **是**(激活大模型内置知识) |

**三条重点注解:**

> **注2 · 第十四章怪物"一章横跨 B 和 C"。** 每个怪物 = 属性块(B,给引擎算战斗/SAN 损失)+ 形象描述(C,给 LLM 演绎恐怖)。解析入库时**两部分拆开存**:属性进结构化字段,描述进给 LLM 的文本字段。复用 §5.4"属性块型 + 机制/描述型"实体设计。

> **注3 · 附录 III / 表17 / 角色卡模板 是最该先"照抄"的 B 类数据。** 装备表、武器表是现成结构化清单;**角色卡模板直接是人物卡 schema 的官方蓝本**——建卡和按钮系统对着它设计字段即可,不必自己发明。

### 4.3 三类各自的处理方式

**A 类 · 规则机制 → Rule Module(引擎)**

- 形式化成引擎能执行的数据 + 代码,**不喂 LLM**。
- 理由:让 LLM 记规则、临场算 SAN 会算错、会漂移;引擎执行则精确、瞬时、可控。
- 例:"1/1D6 理智损失"由引擎读规则掷骰扣 SAN,而非让 LLM 模拟这个规则。
- **只需把规则形式化一次(做成 Module),不需每次对话把规则书喂进上下文。**

**B 类 · 结构化数据 → 数据库 + 硬约束**

- 做成结构化数据表,承担两个作用:
  1. **约束**:玩家/LLM 不能凭空造出库外的东西(防超纲)。例:玩家想在 1920 年代掏出 AK47 → 查装备库不存在 → 引擎拒绝。**这种约束必须靠结构化数据,提示词是软约束会漏。**
  2. **补全**:模组没详述的标准元素,从库里查补进 IR。例:模组只写"出现一只深潜者" → 从生物库查其属性和描述。
- 大模型对 B 类事实**不可靠**(生物属性、物品价格会幻觉),所以精确事实一律走数据库,不靠 LLM 记忆。

**C 类 · 风格基调 → 提示词 + 范例**

- 大模型**内置**克苏鲁的"风格知识"(宇宙恐怖基调、洛式语气、1920 年代氛围),可用提示词有效激活:"你是一位克苏鲁 COC 守秘人,营造宇宙恐怖氛围,强调人类渺小与未知恐惧,语言克制阴郁……"
- 但大模型的设定知识**不精确、不受控**,只能用于"风格",涉及具体事实时必须查 B。
- 几乎零成本,第一天就能有。

### 4.4 特别说明:技能表横跨 A 和 B(最易混淆,但最重要)

技能表看似是 B(有哪些技能),但它同时是 A 的必要输入(检定要用基础值),还是输入层按钮系统的数据源:

```
"教育检定" 这一次检定:
  ├── "COC 有'教育'这项技能" ......... B(清单)
  ├── "教育基础值是多少" ............. B(数据)→ 引擎检定要用
  └── "d100 ≤ 教育值 判成败" .......... A(规则)→ 引擎执行
```

**因此技能表是刚需,必须第一版做。** 且模组按惯例**不会写**技能定义(默认平台提供),没有它连优质模组也跑不动——见 §7。

### 4.5 内容来源优先级(模组 vs 设定库 vs LLM)

当模组、设定库、LLM 三者可能冲突时,裁决顺序:

```
1. 模组明确写了的 → 绝对遵照模组(作者设定最高)
   例:《某模组》说"外域恐怖是黑色水母状",就按这个,不管标准克苏鲁怎么描述
2. 模组没写但引用了标准元素 → 查 B 类设定库补全
   例:模组只写"一只深潜者"没详述 → 从生物库查
3. 模组和设定库都没有 → LLM 按 C 类风格即兴
   例:某路人 NPC 说什么话 → LLM 按 1920 年代 + 克苏鲁氛围自由演

核心:模组 > 设定库 > LLM 即兴。越具体的来源优先级越高。
设定库是"填空默认值",不是"覆盖模组的权威"(模组作者常魔改标准设定)。
```

### 4.6 D 类 · 玩家可读引导(Player-Facing Guidance)

A/B/C 三类服务的都是"系统内部"(引擎、查表、LLM)。但还有一类内容服务的是**人类玩家**——直接展示给玩家阅读的**简略引导文本**。它是第四类,记为 **D 类**。

**D 类特点:** 不参与任何运算、不喂给 LLM 做叙事,是从 A/B/C 提炼出的**人话摘要**,帮玩家快速上手。纯展示层内容。

#### D 类内部再分四子类(来源、时机、复用性不同,必须分开)

| 子类          | 内容                           | 提炼自       | 展示时机        | 复用范围          |
| ----------- | ---------------------------- | --------- | ----------- | ------------- |
| **D1 背景引导** | 世界观、年代、基调(1920 年代、宇宙恐怖是什么)   | C 类(世界观)  | 进入分区 / 开局前  | **平台级,跨模组复用** |
| **D2 规则引导** | 怎么玩:d100 怎么掷、SAN 是什么、检定成败怎么看 | A 类(规则机制) | 首次游玩 / 随时可查 | **平台级,跨模组复用** |
| **D3 模组导语** | 这个模组讲什么、你扮演谁、开场情境            | 模组 IR     | 开局时         | 每个模组独有        |
| **D4 情境提示** | 当前能做什么、可用技能按钮的说明             | 运行时状态     | 游戏中随时       | 运行时动态生成       |

**为什么必须分开(回答"背景和规则要不要分"):** 因为来源和更新时机不同。

- **D1 背景 + D2 规则**是**平台级、跨模组复用**的——所有 COC 模组共用同一套"什么是 1920 年代""d100 怎么掷"。写一次,所有模组通用。
- **D3 模组导语**每个模组独有,跟着模组 IR 走。
- **D4 情境提示**运行时动态生成(如根据人物卡技能生成按钮说明)。

若不分开、混成一坨,会导致每个模组都要重写背景和规则说明,浪费且不一致。

#### 关键纪律:D 类是"投影",不是"另写一份"

D 类不应是团队另外手写的独立文档,而应从 A/B/C **投影/摘要**而来,否则会出现"规则改了但玩家指引没同步"的不一致。

```
A 类规则(引擎里的 d100 检定逻辑)
    └──投影/摘要──→ D2 玩家规则引导("掷 d100,骰值 ≤ 你的技能值就成功")
                    ← 同一套规则的"人话版",不是独立维护的另一份规则
```

这正是 §4.3 提过的"规则书双重身份"里"给人看的形态"。**纪律:A/B/C 是唯一事实源,D 类是基于它们编写的展示文本;规则变更时 D 类必须同步更新**,防止两边打架。(实践中 D 类通常需人工润色,因直接从引擎逻辑自动生成的文本太干,但事实源始终是 A/B/C。)

#### 四类内容总览(更新版)

| 类别           | 服务对象 | 形态                   | 是否喂 LLM |
| ------------ | ---- | -------------------- | ------- |
| **A 规则机制**   | 引擎   | Rule Module 代码/数据    | 否       |
| **B 结构化数据**  | 引擎查表 | 数据库                  | 否(按需检索) |
| **C 风格基调**   | LLM  | 提示词 + 范例             | 是       |
| **D 玩家可读引导** | 人类玩家 | 展示文本(投影自 A/B/C + 模组) | 否       |

```

## 5. 数据模型(Schema)设计

### 5.1 分层策略:Core + Module

**通用框架放 Core(所有规则系统共享),规则专有部分放 Module(按规则系统挂载)。**
即使当前只做 COC,也按此分层组织,以便将来加 DND 等时 Core 可复用、不必重构。
```

Core Schema(通用框架)
├── 地点 Location
├── 叙事 Narration(表象/真相分离)
├── 实体 Entity(属性块型 / 纯机制型)
├── 检定原语 Check(可配置参数)
└── 资源池 Resource(通用可增减数值)

COC Rule Module(COC 专有,对应 §4 的 A 类 + B 类核心)
├── 属性系统(百分比)
├── 检定规则(d100 ≤ 技能值,低优;困难骰/极难骰;奖励骰/惩罚骰)
├── 技能表(固定核心 + 可扩展槽位,见 §6)
├── SAN 理智系统(含 "X/YdZ" 损失语法)
└── 战斗 / 追逐等机制

```
### 5.2 Core 关键设计:检定原语可配置

检定做成"可配置原语",用参数描述,不写死方向。这样不同规则系统用同一引擎、不同参数:

```json
"check_primitive": {
  "dice": "d100",            // COC 用 d100(DND 会是 d20)
  "direction": "roll_under", // COC 低优:骰值 ≤ 目标即成功
  "target_source": "skill_value",
  "modifier_mechanism": ["bonus_die", "penalty_die"],
  "success_tiers": ["fumble", "fail", "success", "hard", "extreme", "critical"]
}
```

### 5.3 Core 关键设计:叙事的"表象/真相分离"

模组原文常把"玩家可见的表象"与"DM 才知道的真相"写在一起(因为写给人类 DM 看)。IR 必须在**数据结构上就分开**,否则 LLM 会剧透隐藏信息:

```json
"narration": {
  "on_first_enter": "只含玩家可见的表象",
  "style_hint": "生成提示,如'不要暗示这里有异常'"
},
"interactables": [
  {
    "hidden": true,
    "reveal_to_player": false,
    "truth": "只有 DM 知道的真相 / 机关 / 隐藏后果"
  }
]
```

### 5.3b Core 关键设计:静态场景 vs 时序场景(重要补充)

§5.3 的 `narration + interactables` 只能描述**静态场景**——"一个地点里固定藏着什么"。但实际游玩中大量出现**时序场景**:带时间顺序、带历史状态的动态规则。二者必须用不同结构表达。

|     | 静态场景                                         | 时序场景                                     |
| --- | -------------------------------------------- | ---------------------------------------- |
| 特征  | 地点里固定有什么                                     | 带"先后顺序"或"依赖历史"的动态规则                      |
| 例子  | 房间有把钥匙、墙上有幅画                                 | 先掷骰后揭示真相;同类恐怖只掉一次 SAN                    |
| 存放  | Location 的 `narration`/`interactables`(空间维度) | 触发块时序字段 + State 历史记录 + Rule Module(状态维度) |

**关键认知:时序场景不属于静态地点 IR。** 它由三者配合表达:①触发块的时序字段、②State 的历史记录、③Rule Module 的通用规则。LLM 在时序场景中的角色不变——**永远只在最后一步演绎已确定的结果,判断与记账都是引擎的活**。

下面用两个真实游玩场景说明。

#### 场景一:先掷骰、后揭示(悬念检定 suspense_check)

DM 说"全体过敏捷",玩家不知缘由,掷完才揭示"踩中陷阱,过了的闪开了"。特征:**检定在前、叙事揭示在后,检定时故意不告诉玩家原因**(制造紧张感)。

这不是"给叙事加突发效果类",而是**给交互块加一种'延迟揭示'的检定类型**:

```json
{
  "id": "trap_ambush_hall",
  "type": "suspense_check",          // 新类型:先检定后揭示
  "trigger": {
    "on": "enter_room",              // 触发时机
    "once": true                     // 只触发一次(依赖 State 记录,见场景二)
  },
  "check": {
    "scope": "all_players",          // 集体检定
    "attribute": "DEX",
    "difficulty": "regular",
    "prompt_to_player": "你们感到脚下传来异样,快过一个敏捷!"  // 故意不说原因
  },
  "reveal": {
    "after_check": true,             // 检定后才揭示
    "on_success": "你敏捷地跃开,寒光擦身而过——一排锈刺从地面弹出。",
    "on_failure": "你慢了半拍,尖刺刺入小腿。",
    "effect_on_failure": {
      "engine_call": "apply_damage",
      "params": { "damage": "1d6", "target": "failed_actors" }
    }
  }
}
```

**LLM 的两阶段信息隔离(防剧透的关键):**

```
阶段1:引擎检测触发 → 只把 check.prompt_to_player 交给 LLM
       → LLM 输出"快过敏捷!"  ← reveal/truth 此时对 LLM 不可见
玩家掷骰 → 引擎判成败 → 结算 effect
阶段2:引擎把"成败结果 + reveal 文本"交给 LLM → LLM 演绎揭示
       ← 此时才把真相给 LLM
```

核心:**`reveal` 内容在阶段1对 LLM 不可见,阶段2才可见。** LLM 喊"过敏捷"时它自己都"不知道"是陷阱,自然不剧透——信息隔离优于提示词约束(与 §5.3 同思路,加了时间维度)。

#### 场景二:同类恐怖只掉一次 SAN(全局去重规则)

第一次见血腥场面掉 SAN,之后再见**同等级同类型**场面就不掉了(COC 的"同源不重复"惯例)。特征:**贯穿整局、跨地点生效的全局规则,判断依赖玩家的历史经历**。

这类"全局设定"**不属于任何单个地点**,拆两层记录:

**第一层:规则本身 → COC Rule Module(A 类,通用规则)**

```json
// COC Rule Module → SAN 系统
"san_loss_rules": {
  "dedup_by_source": true,           // 开启同源去重
  "dedup_granularity": "source_tag"  // 按"恐怖来源标签"去重
}
```

**第二层:玩家经历过什么 → State(动态历史记录)**

```json
// State Store
"player_state": {
  "SAN": { "current": 48 },
  "encountered_san_sources": ["gore_scene_low", "corpse_normal"]  // 已见过的来源
}
```

**运行时逻辑(全在引擎,不经过 LLM):**

```
触发 SAN 检定,来源标签 = "gore_scene_low" →
  引擎查 encountered_san_sources:
    ├─ 已存在 → 跳过扣 SAN(仅叙事描述,不掉理智)
    └─ 不存在 → 正常掷骰扣 SAN → 把标签加进已见清单
```

**模组 IR 里只需标注来源标签和损失值,不写去重逻辑**(去重是引擎通用能力):

```json
"san_trigger": {
  "source_tag": "gore_scene_low",    // 来源标签(用于去重)
  "loss": "1/1d4",                   // 成功掉1,失败掉1d4
  "description_ref": "..."           // 给 LLM 演绎的场面描述
}
```

**三方分工:** 模组作者标"什么来源、掉多少";引擎管"见过没、要不要真扣"(查 State 去重);LLM 管"把场面演绎得多恐怖"(纯叙事,不碰 SAN 数值)。

#### 时序场景要求 Schema 补三样东西

1. **触发块增加类型与时序字段**:`type: suspense_check`(先掷后揭)、`trigger.once`(一次性)、`reveal.after_check`(揭示时机)
2. **State 增加"历史记录"区**(全局设定的落脚点——全局规则的"记忆"存 State,不存任何地点):
   - `triggered_events: []`(哪些一次性事件已触发)
   - `encountered_san_sources: []`(见过哪些恐怖来源)
3. **通用规则(去重、一次性判断)放进 Rule Module**:引擎读规则、查 State 历史、做确定性判断

### 5.4 Core 关键设计:实体可以是"纯机制型"

实体(NPC/怪物)**不强制有战斗属性块**。有的实体靠机制描述定义(如某些 COC 怪物没有标准 AC/HP,而是"碰触即造成特定后果")。Schema 要同时支持:

- **属性块型**:有生命值、伤害、护甲等结构化战斗数值
- **纯机制型**:无标准属性块,靠"触发条件 → 后果"的机制描述定义

### 5.5 剧本 IR:六类实体

模组解析产物(IR)至少含六类实体 + 实体间引用关系(构成场景图):

1. **元数据 Meta**:规则系统(COC)、模组标题、简介、人数/时长建议
2. **NPC/怪物 Entity**:剧情型(人格/动机/秘密,给 LLM 扮演)+ 属性型(数值,给引擎),可并存
3. **地点 Location**:描述、可交互元素、触发规则、遭遇、战利品、连接关系
4. **规则性触发块**:检定要求 `{触发条件, 检定类型, 失败后果}`、随机表 `{掷骰, 结果映射}`、伤害效果
5. **物品 Item**:名称、价值、效果、位置、获取条件
6. **美术资源 Asset**:插画/地图,**关联到对应的地点/NPC/怪物实体**(不孤立存储)

### 5.6 人物卡(调查员)Schema

人物卡是按钮系统和检定引擎的直接输入,字段要为二者服务:

```json
{
  "name": "调查员姓名",
  "occupation": "职业",
  "attributes": { "STR": 50, "DEX": 60, "INT": 70, "EDU": 65, "POW": 55, "...": "" },
  "resources": {
    "HP":  { "current": 11, "max": 11 },
    "SAN": { "current": 55, "max": 99, "start": 55 },
    "MP":  { "current": 11, "max": 11 },
    "LUCK": 50
  },
  "skills": [
    {
      "name": "侦查",
      "value": 65,
      "category": "调查",          // 给按钮系统分组用
      "check_ref": "coc_standard", // 给检定引擎用:怎么掷
      "growth_flag": false         // COC 成长标记
    },
    {
      "category_type": "科学",     // 可扩展槽位(见 §6)
      "specialization": "天文学",  // 玩家填
      "value": 1,
      "category": "知识",
      "check_ref": "coc_standard"
    }
  ]
}
```

**字段用途标注:** `value`/`check_ref` 给引擎;`category` 给按钮分组;`resources.SAN` 由 SAN 系统读写;`specialization` 需校验(见 §6)。

## 6. COC 技能表:固定核心 + 可扩展槽位

**技能框架全 COC 统一(固定核心,约 60+ 项),但部分技能是"类别 + 专精"结构,专精由玩家填。**

- **固定技能**:侦查、聆听、图书馆使用、话术、说服、医学、攀爬、潜行、闪避、克苏鲁神话……(名称 + 基础值固定)
- **可扩展槽位(需专精填空)**:艺术/手艺(___)、科学(___)、格斗/射击(___)、其他语言(___)、驾驶(___)……

### Schema 支持两种技能形态

```
固定技能:{ name:"侦查", value:25, category:"调查" }
专精技能:{ category_type:"科学", specialization:"天文学", value:1, category:"知识" }
                                    ↑ 玩家填,需校验
```

### 自填技能"是否符合常理"的校验(分层,大部分不需 LLM)

```
玩家填专精技能 →
  1. 类别是否在标准表内?     (查表,硬校验)   → 否则拒绝或走 3
  2. 专精是否符合年代/世界观? (年代约束,B类)  → 如"科学(核物理)"@1920 → 不合理
  3. 完全自创的新技能?       (软判断)         → 交 DM/规则裁决,LLM 可辅助
```

**大部分是情况 1、2(查表 + 年代约束,结构化、快而准),只有情况 3 才真需 LLM 软判断。** 再次符合"能结构化就别用 LLM"。

**务实提醒:** 校验放在**建卡/传卡环节**(运行时很少临时自创技能),前期可简化为"类别必须在标准表内"一条硬规则。标准技能表(60+ 固定项)是第一版必录的 B 类核心数据。

## 7. 大模型参与边界(明确 LLM 做什么、不做什么)

**这是本项目"确定性 + 创造性解耦"的落地。以 COC 的一次检定为例,清晰划出边界。**

### 7.1 LLM 负责什么(软、需智能、有创造性)

1. **叙事演绎(主战场)**:把引擎算出的结果(如"侦查成功")变成生动文字;推进场景、渲染氛围、扮演 NPC。
2. **自由行为的意图解析(枢纽)**:玩家打字"我想爬上房梁偷看"且无对应按钮时,LLM 判断"这需要一次潜行/攀爬检定",然后**回调确定性引擎**。
3. **临场裁决"该不该检定、用什么技能、难度多少"**:玩家说服 NPC,LLM 判断需不需要检定、用话术还是说服、难度如何——这是真人 KP 的临场判断。**判断完的执行交引擎。**
4. **C 类风格演绎**:以克苏鲁基调组织语言(靠内置知识 + 提示词)。

### 7.2 引擎负责什么(硬、确定性、瞬时)

1. **掷骰与判定**:d100 ≤ 技能值,判定成败等级(大成功/成功/困难/极难/失败/大失败)。
2. **数值结算**:HP/SAN/MP 增减,伤害计算,SAN 的 "X/YdZ" 损失。
3. **State 读写**:唯一真相源的更新。
4. **B 类事实查询**:技能基础值、装备属性、生物属性等查表。

### 7.3 一次检定的边界示例(COC)

```
玩家点【侦查】按钮
  │
  ├─[输入层] 生成结构化意图 { action:"check", skill:"侦查" }
  │
  ├─[引擎] 读人物卡"侦查=65" → 掷 d100 → 比较 ≤65 → 判定"成功"
  │        → (若涉及)结算 state
  │
  └─[LLM] 收到"侦查成功"这一确定结果 → 演绎:
          "你拨开积尘的书页,一行褪色的字迹映入眼帘……"

数值全程不经过 LLM。LLM 只决定"要做什么运算"和"如何叙述结果"。
```

### 7.4 边界对照总表

| 环节                 | 谁负责      | 类别     |
| ------------------ | -------- | ------ |
| 该不该检定 / 用什么技能 / 难度 | LLM      | 裁决     |
| 玩家自由行为的意图理解        | LLM      | 意图解析   |
| 掷骰、比较、判成败          | 引擎       | A      |
| HP/SAN 等数值结算       | 引擎       | A      |
| 技能值/装备/生物属性查询      | 引擎查表     | B      |
| 把结果演绎成叙事、扮演 NPC    | LLM      | C + 演绎 |
| 风格基调(克苏鲁氛围)        | LLM      | C      |
| 防止超纲(库外物品)         | 引擎(数据约束) | B      |

**一句话:LLM 决定"要做什么运算",引擎执行"运算本身"。按钮/快捷语的意义,是让玩家直接告诉系统"要做什么运算",从而省掉 LLM 的判断步骤(更快更准)。**

## 8. 第一版范围建议(务实优先级)

以 COC 为第一个规则系统,按"跑通自包含模组"排优先级:

**第一版必做:**

- COC Rule Module 核心(A 类):d100 检定、SAN 系统、基础数值结算
- 标准技能表(B 类核心):60+ 固定技能 + 可扩展槽位定义
- 风格提示词(C 类):克苏鲁基调,零成本
- Core Schema + COC Module Schema(§5)
- 人物卡 Schema + 按钮生成
- 单个自包含模组的 IR + 跑通

**可延后:**

- 装备/武器库、神话生物库、术语表(B 类扩展)——很多模组自带所需内容
- 规则书查阅界面(A 形态:给人看的文档)
- 完整的自创技能校验(先做"类别在表内"硬规则)
- 战斗/追逐等复杂 A 类机制(视首个模组是否用到)

**判断依据:模组是自包含的,先跑通一个模组,别被"要不要建完整克苏鲁库"卡住。**

## 附录 B:核心术语

- **DM/KP/守秘人**:游戏主持人,本项目由 AI 担任
- **调查员**:COC 中玩家角色的称呼
- **模组/Scenario**:自包含的冒险剧本
- **IR**:模组解析后的结构化数据,运行时读它
- **Rule Module**:某规则系统的形式化实现(A + B 核心),引擎加载执行
