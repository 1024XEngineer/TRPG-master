# 如何处理动态规则

## 问题

不同模组会有不同战斗规则：

- 追书人：火器/贯穿伤害减半，舍去小数
- 鬼屋：血肉防护 2d6，每受 1 伤减 1 甲，24h 或耗尽
- 死者：僵尸——枪弹只损 1 耐久 / 刀伤减半 / 无视重伤 / 不会闪避
- 蛙蛙村：长舌命中 → 下回合力量对抗拖拽，失败则直接吞食
- 复足：冷蛛 4 轮内未造成伤害则转换目标

为每条建一个字段（`ignores_major_wound: true`）是在用 JSON 写 if-else。第八个模组来了再加一个。

而且字段之间的**作用顺序表达不出来**——「血肉防护」和「火器减半」谁先算，结果差一倍。

---

## **规则在什么时刻被求值？**

战斗不是一次 `damage → hp` 的映射，它是一条**流水线**：

```
声明攻击
  → 决定用哪个技能           [复足: 闪躲/拳击/脚踢皆可]
  → 计算难度                 [死者: 人数→难度]
  → 掷攻击骰
  → 比较成功等级             [鬼屋: 六层分支]
  → 防守方能否闪避           [死者: 僵尸不能]
  → 掷闪避骰
  → 判定命中
  → 掷伤害骰
  → 伤害类型分流             [死者: 枪弹 vs 刀]
  → 护甲扣减                 [鬼屋: 血肉防护]
  → 写入 HP
  → 重伤判定                 [死者: 无视]
  → 触发后效                 [蛙蛙村: 拖拽/吞食]
```

动态规则并不是实体的固定属性，而是在游戏流程进行到特定阶段时，对默认行为进行修正。它可能影响行动声明、检定、伤害结算、状态判断，也可能产生跨回合效果。

因此，规则结构不仅要描述“产生什么效果”，还要表达“何时触发、在什么条件下触发”，以及多条规则同时生效时的执行顺序。

---

## 方案

导入的产物不是一个 JSON 对象，而是 `entity → rules[]` 的映射。

规则 = `(hook, when, then)` 三元组的数组。

```json
[
 {"hook":"on_dodge_declare","when":"always","then":{"forbid":true}},
 {"hook":"on_damage_roll","when":"damage.type=='firearm'","then":{"set":1}},
 {"hook":"on_damage_roll","when":"damage.type!='firearm'","then":{"scale":0.5,"round":"floor"}},
 {"hook":"on_major_wound","when":"always","then":{"forbid":true}}
]
```

引擎提前实现四张表，全部有限可枚举。LLM 只生成它们的组合。

---

## 四张表

### 1. Hook（挂载点，~12 个）

战斗是流水线，每步一个钩子：

```
on_attack_declare    on_difficulty_calc   on_attack_roll
on_dodge_declare     on_dodge_roll        on_hit_resolve
on_damage_roll       on_armor_apply       on_hp_write
on_major_wound       on_death             on_turn_end
```

### 2. 内建状态（可读变量，~15 个）

```
combat.round
self.rounds_without_damage
self.damage_taken_this_round
attack.type / attack.name
damage.type / damage.value
target.form
party.size / party.max_STR
```

**这份列表就是引擎能力的边界。** 规则引用了表外的东西（`self.mood`）→ 校验失败，打回重生成。

计数器类变量必须由引擎自动维护，不能指望规则自己累加。

### 3. 表达式语法（`when`）

```
字段引用   damage.type / attacker.STR
比较       == != < <= > >=
布尔       && || !
算术       + - * / floor() min() max()
聚合       count(party) / max(party.STR)
```

**无循环、无递归、无函数定义。** 纯表达式语言，二三百行写完，天然安全——不死循环，不越权。

六个模组的所有规则，没有一条需要循环。「每轮 1d6 力量伤害」是引擎的回合循环在驱动，规则只描述单轮。

### 4. 算子（`then`，~10 个）

```
set / scale / add / absorb / decrement
forbid / force / apply_condition / spawn / trigger
```

## 验证

**鬼屋·血肉防护**（有状态、会耗尽、有时限）

```json
{"hook":"on_armor_apply","when":"self.flesh_ward > 0",
 "then":{"absorb":"min(damage, self.flesh_ward)","decrement":"self.flesh_ward"}}
```

**蛙蛙村·长舌拖拽**（跨回合 + 后续对抗）

```json
{"hook":"on_hit_resolve","when":"attack.name=='长舌鞭笞'",
 "then":{"trigger":"pending_drag","expires_in_rounds":1}}

{"hook":"on_turn_end","when":"self.pending_drag",
 "then":{"oppose":{"a":"self.STR","b":"target.STR"},
         "on_win":{"trigger":"swallow"},"on_lose":{"clear":"pending_drag"}}}
```

跨回合靠 `trigger` + 状态槽实现，不需要新机制。

**复足·转换目标**

```json
{"hook":"on_turn_end","when":"self.rounds_without_damage >= 4",
 "then":{"force":"switch_target"}}
```

## 换规则系统

Core 定义 `(hook, when, then)` 这个结构本身。
Module 定义 hook 表和变量名。

COC → OSE，换的是内建状态的内容（`THAC0`、`AC`、`士气`），三元组结构不变。

---

## 三个坑

### 1. `when` 里不能放语义判据

```json
{"when": "玩家表述得很精彩"}              ❌ 求值器算不了
{"when": "roleplay_tier == 'excellent'"}  ✅
```

语义判据由 LLM 在**上游**求值，输出枚举写进状态槽，规则只读枚举。

**软判据和硬求值分在两个阶段。**

### 2. 同一 hook 上的执行顺序必须显式

「火器减半」和「护甲扣减」谁先？加 `priority: int`。

不加的话，LLM 生成的数组顺序就成了隐式语义——它不稳定。

### 3. LLM 会漏掉「禁止」类规则

`{"then":{"forbid":true}}` 的生成命中率明显低于 `{"then":{"scale":0.5}}`。LLM 不愿意生成限制自己的规则。

**「僵尸不会闪避」漏掉之后，系统不报错，僵尸只是变强了一点。**

静默失败。只能靠导入后定向自检兜底——针对每个怪物问一遍：

> 它有什么**不能做**的事？

---

## 结论

「动态生成规则」的准确含义：

**引擎提前实现 hook 表、内建状态表、表达式语法、算子表；LLM 生成这四张表的组合。**

不是让 LLM 发明字段名，也不是让引擎跑沙箱脚本。
