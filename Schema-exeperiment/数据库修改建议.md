我们要基于现有数据库设计搭 MVP。我已经用六个真实 COC 模组
（追书人、鬼屋、幸福蛙蛙村、银之锁、复足、死者的顿足舞）
逐条验证过当前 schema，请按下面的清单修改。

分成三部分：
  A. MVP 必改 —— 不改则至少三个模组无法导入
  B. MVP 可省 —— 本次搁置，先由大模型语义处理
  C. 后续扩展 —— 记录在案，暂不实现

每条修改都附了推翻它的具体模组和原文，请不要跳过理由部分——
理由决定了字段该长什么样。

────────────────────────────────────────────
A. MVP 必改（6 项）
────────────────────────────────────────────

【A0】worlds 新增 hooks、variables、world_rules 三个字段

  这一条排在最前面，因为它是 A1 的 rules 字段能不能用的前提，
  且它是架构而非字段——晚做的迁移成本远高于其余五条。

  worlds {
    id, name,
    definition   JSON,   -- 保留，一个字段够
    hooks        JSON,   -- ★ 新增
    variables    JSON,   -- ★ 新增
    world_rules  JSON    -- ★ 新增
  }

  ── hooks：引擎战斗流水线上的挂载点清单 ──

  战斗不是一次 damage → hp 的映射，它是一条流水线：

    声明攻击 → 决定用哪个技能 → 计算难度 → 掷攻击骰
    → 比较成功等级 → 防守方能否闪避 → 掷闪避骰 → 判定命中
    → 掷伤害骰 → 伤害类型分流 → 护甲扣减 → 写入 HP
    → 重伤判定 → 触发后效

  每一步都是一个可被外部规则拦截或改写的暂停点。COC 7e 的 12 个：

    on_attack_declare    on_difficulty_calc   on_attack_roll
    on_dodge_declare     on_dodge_roll        on_hit_resolve
    on_damage_roll       on_armor_apply       on_hp_write
    on_major_wound       on_death             on_turn_end

  每个 hook 需定义签名（接收什么、返回什么）。
  例：on_damage_roll 接收 damage、返回 damage，所以它能改写伤害值；
      on_dodge_declare 返回 bool，所以它能拦截整个闪避步骤。

  注意 hook 不是「事件」。事件是「已经发生了，通知你一声」，
  改不了既成事实。hook 是「即将发生，引擎在等你的返回值」。
  如果只做事件，死者的「僵尸被枪弹命中只损1点耐久」实现不了
  —— 伤害已经扣完了。

  ── variables：内建可读变量表 ──

    combat.round
    self.HP / self.HP_max / self.flesh_ward
    self.rounds_without_damage
    self.damage_taken_this_round
    attack.type / attack.name
    damage.type / damage.value
    target.form
    party.size / party.max_STR

  这份列表就是引擎能力的边界。规则引用表外的键（如 self.mood）
  → 导入校验失败，打回重生成。

  计数器类变量（rounds_without_damage）必须由引擎自动维护，
  不能指望规则自己累加。

  ── world_rules：COC 7e 自身的规则，写成三元组 ──

    [{"hook":"on_major_wound",
      "when":"damage >= self.HP_max / 2",
      "priority":100,
      "then":{"apply_condition":"major_wound"}}]

  关键：world_rules 与 module_entities.rules 必须是同一种数据结构。
  一旦结构不同，就必须写一个合并器，而合并器会成为整个系统最脏的地方。

  ── 规则的结构：(hook, when, then, mode, priority) ──

  when 是布尔表达式。求值器只需支持：
    字段引用 / 比较(== != < <= > >=) / 布尔(&& || !)
    / 算术(+ - * / floor min max) / 聚合(count max)

  无循环、无递归、无函数定义。这是个纯表达式语言，
  二三百行写完，且天然安全（不会死循环，不会越权）。
  六个模组的所有规则，没有一条需要循环
  —— 「每战斗轮1D6力量伤害」是引擎的回合循环在驱动，
  规则本身只描述单轮。

  then 的算子（约 10 个）：
    set / scale / add / absorb / decrement
    forbid / force / apply_condition / spawn / trigger

  mode 决定 world 与 module 规则冲突时如何消解，只需三个：
    append    我加一条，前面的照常执行
    override  我执行，同 hook 上 source=='world' 的规则跳过
    forbid    这个 hook 整个不执行

  引擎在每个 hook 上：
    1. 收集 world_rules + module rules + entity rules
    2. 按 priority 排序（module 默认优先于 world）
    3. 逐条执行；遇 forbid 跳过整个 hook；遇 override 跳过 world 规则

  ── 为什么不能用「一堆具名布尔字段」 ──

  死者的僵尸有四条规则：枪弹只损1耐久 / 刀伤减半 / 无视重伤 / 不会闪避。
  写成 ignores_major_wound: true 这类字段，等于用 JSON 写 if-else。
  第八个模组来了再加一个字段。

  更隐蔽的问题是作用顺序表达不出来：
  鬼屋的「血肉防护」和追书人的「火器伤害减半」如果同时存在，
  先减半还是先扣甲？结果差一倍。用字段的话这个顺序藏在引擎代码里，
  模组作者无法覆盖。用 (hook, priority) 就显式了。

  ── 六模组实证：算子集合已收敛 ──

  死者·僵尸：
    [{"hook":"on_dodge_declare","when":"always","mode":"forbid"},
     {"hook":"on_damage_roll","when":"damage.type=='firearm'","then":{"set":1}},
     {"hook":"on_damage_roll","when":"damage.type!='firearm'",
      "then":{"scale":0.5,"round":"floor"}},
     {"hook":"on_major_wound","when":"always","mode":"forbid"}]

  鬼屋·血肉防护（有状态、会耗尽、有时限）：
    [{"hook":"on_armor_apply","when":"self.flesh_ward > 0","priority":10,
      "then":{"absorb":"min(damage, self.flesh_ward)",
              "decrement":"self.flesh_ward"}}]

  复足·冷蛛转换目标（「4轮内未造成伤害则换目标」）：
    [{"hook":"on_turn_end","when":"self.rounds_without_damage >= 4",
      "then":{"force":"switch_target"}}]

  蛙蛙村·长舌拖拽（跨回合 + 后续对抗）：
    [{"hook":"on_hit_resolve","when":"attack.name=='长舌鞭笞'",
      "then":{"trigger":"pending_drag","expires_in_rounds":1}},
     {"hook":"on_turn_end","when":"self.pending_drag",
      "then":{"oppose":{"a":"self.STR","b":"target.STR"},
              "on_win":{"trigger":"swallow"},
              "on_lose":{"clear":"pending_drag"}}}]

  六个模组跑完，没有出现第 13 个 hook，也没有出现新算子。
  这就是为什么现在写它是安全的。

  ── 三个必须注意的地方 ──

  1. when 里不能放语义判据。
     {"when": "玩家表述得很精彩"}              ❌ 求值器算不了
     {"when": "roleplay_tier == 'excellent'"}  ✅
     语义判据由大模型在上游求值，输出枚举写进状态槽，规则只读枚举。
     软判据和硬求值必须分在两个阶段。
     （蛙蛙村：说服信使，不说=困难 / 合理=普通 / 精彩=普通+奖励骰）

  2. 同一 hook 上多条规则的执行顺序必须显式。
     不给 priority 的话，大模型生成的数组顺序就成了隐式语义，
     而它不稳定。

  3. 大模型会漏掉「禁止」类规则。
     {"then":{"forbid":true}} 的生成命中率明显低于 {"then":{"scale":0.5}}。
     它不愿意生成限制玩家的规则。
     「僵尸不会闪避」漏掉之后，系统不报错，僵尸只是变强了一点——静默失败。
     兜底方式是导入后对着 12 个 hook 逐个质询，尤其是空着的那些：
       「这个怪物，有什么不能做的事？」
     大模型回答这个问题很准。它只是不会主动想起来要问。

  ── 换规则系统时会发生什么 ──

  COC 的 hook 表：on_dodge_declare / on_major_wound / on_san_check
  OSE 的 hook 表：on_thac0_calc / on_saving_throw / on_morale_check

  hook 的名字变了、变量名变了（self.闪避 → self.AC），
  但 (hook, when, then) 这个结构、表达式求值器、算子表，一行没动。

  这就是 Core / Module 分层的落点。
  「哪些字段属于 Core」这个问法从一开始就错了——字段本身就是 Module 的产物。
    Core   定义规则如何被表达和求值
    Module 定义有哪些 hook、hook 上有哪些可读变量


【A1】新增 module_entities 表，吸收 module_clues 和 module_npcs

  module_entities {
    id, module_id, kind, name,
    content,          -- 线索内容 / 物品描述
    public_persona,   -- 保留原 module_npcs 的设计
    secrets,          -- 保留
    stats     JSON NULL,   -- 必须可空
    state     JSON,        -- ★ 见 A2
    rules     JSON         -- ★ (hook, when, then) 数组，见 A0
  }
  kind ∈ {npc, monster, item, clue, animal, object}

  原因一：现有设计把 module_clues 当作「最烧脑但也最关键的表」，
  这个假设只对调查向模组成立。六个模组里只有追书人和死者是这个形态。
  银之锁的核心是猫的生死，复足是梦境之石的携带与否，
  蛙蛙村是说服，鬼屋是战斗。围着线索网络生长 schema，
  这四个模组会被挤成畸形。

  原因二：猫、梦境之石、小号、羊毛布料，它们是同一类东西——
  「有状态的、可被表达式引用的实体」。线索只是其中一种 kind。

  原因三：stats 必须可空。追书人的邻居和看守、鬼屋的阿蒂、
  蛙蛙村原文写「其他游客守密人随便编就行」、银之锁的芭斯特，
  全都没有属性。字段不可空的话，解析器会逼大模型编数字。

  原因四：rules 挂在 entity 上，不是挂在 module 上。
  「僵尸不会闪避」是僵尸这个 entity 的规则，不是死者这个模组的规则。
  同一模组里，人类形态的特纳和僵尸形态的特纳规则不同。


【A2】module_entities.state 与 rooms.entity_states

  state 是初始值（内容库，只读），entity_states 是当前值（运行时）。

  必须存在的键（六模组实测）：
    cat.alive              银之锁
    pc.has_dreamstone      复足
    turner.form            死者（人类/僵尸两套数据）
    turner.death_count     死者（马努斯科被杀三次，alive 不能是单向 bool）

  原因：这些不是 NPC 的属性，是被结局表达式和 rules[].when 求值的变量。
  引擎读不了上下文，只能读变量。

  重要区分：另有一类状态可以不落库，只留在提示词里——
    npc_doctor.attitude     只影响大模型怎么演
    cat.trusts_player       只影响大模型怎么演

  判据（可以写成一个符号表检查自动跑）：
    这个键有没有出现在任何 rules[].when 或 win_conditions.expr 里？
      有  → 落 state
      没有 → 走提示词，不落库

  补充一层：被引擎读的状态，写入口必须是引擎校验过的 op。
  大模型只能提议 modify(cat.alive, true)，引擎校验后执行。
  不能让大模型直接改 state——它会救猫。


【A3】module_win_conditions 改为表达式

  现有：「找齐哪些核心线索就算通关」
  改为：expr TEXT  -- 引用 entity_states 的布尔表达式
        is_ending BOOL

  推翻它的四个模组：
    复足    携带梦境之石 × 离开宾馆 = 生；未携带 × 离开 = 死
    银之锁  救猫→猫活→走廊倒地→猫抓瞎NPC→NPC杀猫→锁解除
            （救猫是通关必要条件，猫死也是，且死法唯一）
    死者    阻止特纳而不杀他。杀了他会复活，且不再需要呼吸，更难杀
    蛙蛙村  闪避成功 = 失去永久幸运 +2d6

  is_ending 的必要性：银之锁「没救猫 → 被抓回房间 → 重来」
  是状态回滚不是终局。不加这个字段，大模型会把它演成 bad ending
  并结束游戏。


【A4】module_san_triggers 扩展 kind 与 source_tag

  + kind (enum)     SAN 有六种形态，现有设计只能表达第一种
  + source_tag      累计封顶用

  六种形态及出处：
    ① 检定式          0/1d6                    全部模组
    ② 无检定固定损失   读日记 −1d4              追书人、鬼屋
    ③ 精神伤害        对抗胜→直接扣1d6，不走检定  蛙蛙村
    ④ 上限扣减        +2%神话 → SAN_max 99→97   鬼屋
    ⑤ 回复           +1d6 / +2d6               四个模组
    ⑥ 累计封顶        同源合计 ≤ 6 / ≤ 8        追书人、死者

  ③ 尤其重要：大模型会习惯性插入一次 SAN 检定，
  因为绝大多数 SAN 损失都要检定。必须用字段名切断这个惯性。

  ⑥ 注意语义：是累计封顶，不是去重。追书人一场内触发三次同源 SAN，
  要求累加后与 6 比较。大模型会当成「见过就跳过」，语义完全不同。
  追书人和死者独立出现同一规则（习惯恐惧），说明它属于 worlds
  而非 module，但 tag 由模组提供。


【A5】characters 新增 conditions 与 ledger

  + conditions JSON   带调度器的状态机
  + ledger     JSON   计数器 + 时间窗
  + location   TEXT   从 rooms.current_scene_id 迁移过来

  conditions 不能是字符串数组。三个实例：
    鬼屋   科比特爪伤：命中→幸运检定→一天后→CON检定→分支→循环直到痊愈或死亡
    复足   感染阶段0-5，难度是阶段的函数（0幸运/1体质×5/2体质×4/…/5体质×1）
    蛙蛙村 变蛙四阶段。阶段2可逆、阶段3起不可逆；
           阶段2是 delta(+4d6)、阶段3是 set(=25)
           —— 大模型会把「APP变成25」误算成「APP−25」

  ledger 的实例：
    蛙蛙村 24小时内意志失败≥2次 → 此后惩罚骰
    复足   意志×1抵抗渴望10分钟，之后每10分钟必须重新鉴定
    鬼屋   MP 每小时回复1点（作者原文特别提醒 KP「小心记录 MP」）

  核心原因：这不是记忆问题，是调度问题。
  「一天后」需要有人在一天后把它叫醒。大模型是被动的，
  它只在玩家说话时被调用。玩家在游戏内第二天早上说「我去餐厅」，
  没有任何东西会提醒它该插入一次 CON 检定。

  location 迁移的原因：现有 rooms.current_scene_id 假设
  「当前所有玩家在同一个场景」。死者原文：
  「调查员们需要将队伍拆分开以增大其覆盖面」。

B. MVP 可省（本次搁置）
────────────────────────────────────────────

【B1】物品实体化 —— 本次完全由大模型语义处理

  inventory 保持字符串数组：["棍子", "煤油灯", "一把小钥匙"]
  不加 id，不加 locked_by，不建物品实体。

  已知会漂的地方，MVP 阶段接受：
    银之锁三把钥匙来源不同（挂画/床下/通风管）对应三个抽屉，
    大模型可能串。串了之后玩家打不开正确的抽屉，静默失败。
    → MVP 阶段先观察实际发生率，再决定是否加 id。

  但注意一个例外：复足的梦境之石不属于「物品」，
  它是结局表达式的输入（携带=生，不携带=死）。
  按 A2 的判据，它必须落 entity_states，
  只是不需要建「物品实体」——一个 bool 就够。

  这个区分很重要：落库的是「值」，不是「物品」。


【B2】rules 字段先建、可为空数组

  A0 定义了 hook 表和规则结构，A1 建了 rules 字段。
  但 MVP 阶段，模组导入时可以先不生成 rules，
  战斗规则暂时写进 secrets 或 public_persona，由大模型执行。

  这样做的前提是 A0 已经落地——表结构和 hook 清单先定死，
  后续接入规则引擎时不需要改表，只需要开始往 rules 里写东西。

  已知会漂的地方：这类「限制玩家」的规则，
  大模型的遗漏有系统性偏向——漏掉之后玩家都会更开心，
  且不会报错。MVP 阶段用 A0 里那个 12-hook 质询清单做人工复核。


【B3】三张场景中间表 —— 暂不合并

  module_scene_clues / module_scene_checkpoints / module_scene_san_triggers
  可以先留着。合并到 module_scenes.contents 是优化，不是阻塞。

  但请先删掉 module_clues.reachable_by 和 module_npcs.linked_clue_ids
  这两个字段（迁到 module_entities 时不要带过去）。

  原因：路径不是线索的属性，是场景的属性。
  中间表只存 (scene_id, clue_id)，表达不了 via——
  「在书房靠侦查，在门厅靠跟管家聊」。
  这正是 reachable_by 被迫诞生的原因，同一份信息存了两遍且可能对不上。

  MVP 阶段就用中间表 + 场景描述里的自由文本说明获取方式。


【B4】「核心线索至少两条路径」原则 —— 降级为警告

  现有：少于 2 条路径直接拒绝导入。
  改为：警告 + 允许 intentional_single_path 显式标记。

  原因：故意的单路径在六模组里很常见——
    银之锁  钢钳是救猫的唯一方法
    复足    感染只能通过切开囊肿取出幼蛛治疗
    死者    说服疯子需极限难度检定，「除此之外的任何结果都毫无作用」

  死者那句是作者在预防 KP 心软。硬拒绝的话这三个模组一个都进不来。


【B5】其余保持不动

  worlds.definition 一个字段够，不用拆
  （d100 / roll_under / 成功等级 / 技能表 留在这里；
    但 COC 7e 自身的可挂载规则要迁进 A0 的 world_rules）。
  module_pregens / module_assets / notes / users / blob_assets 全部保留。
  players.character_id 回指指针建议保留（分头行动时反查会成为热点）。

  C1  module_checkpoints 扩展
      + threshold      复足「撬门(力量20)」是固定阈值检定，
                       与 roll_under skill% 并列的第二种原语。
                       它在复足里出现四次（撬门/破窗/破门/上锁门），
                       是通用机制 → 应补进 worlds.definition 的原语列表，
                       而不是让复足用 override 打补丁。
                       判据：override 若反复出现，说明 World 定义得不够。
      + difficulty_expr 死者「1人=极限 / >3人=常规」
      + purpose         死者「敏捷检定确定抵达先后顺序」，返回排名而非成败
      + roll_representative 死者「让力量最高者掷，其他人不掷」

  C2  module_entities.forms
