# 数据模型设计

> **本文定位**：定义系统的**逻辑数据模型**——有哪些实体、每个实体有哪些字段、字段的语义与约束。
>
> 与 [[数据库设计导读]] 的分工：
> - 数据库设计文档回答 **「怎么存」**（建什么表、什么索引、JSONB 还是 TEXT）
> - 本文回答 **「存的是什么」**（`state JSONB` 这一列里，究竟装着什么结构）
>
> 两者是**投影关系**，不是对齐关系。逻辑模型里的一等公民（如 `Rule`）在物理模型里可能被折叠进一列 JSONB——这不矛盾，见 §7。
>
> **验证基础**：本模型的每一项设计都经过六个真实 COC 模组的逐条验证：
> 《追书人》《鬼屋》《幸福蛙蛙村》《银之锁》《复足》《死者的顿足舞》。
> 凡是标注「★」的字段，均为「不加则至少三个模组无法导入」的硬阻塞项。

---

## 一、分层

沿用现有三层划分。本文补充每层的**可变性契约**——这是后续所有约束的基础。

| 层 | 实体 | 可变性 | 谁能写 |
|---|---|---|---|
| **Content** | `World` / `ModulePack` / `Scene` / `Entity` / `Checkpoint` / `SanTrigger` / `WinCondition` / `Pregen` / `Asset` | 只读 | 仅模组导入流程 |
| **GameState** | `Room` / `Player` / `Character` / `Note` | 可变 | **仅引擎**（经校验的 op） |
| **EventLog** | `Event` | 只增 | 仅引擎 |

> **关键契约**：GameState 的写入口唯一。LLM 不得直接写状态，只能**提议** op，由引擎校验后执行。
> 详见 §6.3。

---

## 二、Content 层

### 2.1 World

一整套规则系统（如 COC 7e）。行数极少，几乎不变。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `WorldId` | 否 | 如 `"coc-7e"` |
| `name` | `string` | 否 | 「克苏鲁的呼唤 7版」 |
| `definition` | `RuleSystemDef` | 否 | 属性范围、技能表、检定原语、成功等级 |
| ★ `hooks` | `HookDef[]` | 否 | 引擎流水线上的可挂载点清单。见 §4.1 |
| ★ `variables` | `VariableDef[]` | 否 | 规则可读的内建变量表。见 §4.2 |
| ★ `world_rules` | `Rule[]` | 否 | 本规则系统自身的规则，写成三元组。见 §4.3 |

**为什么 `definition` 保留为单字段、而 `hooks`/`variables`/`world_rules` 要独立**

`definition` 装的是**声明式常量**（d100、roll_under、成功等级六档、技能基础值表）——读一次，之后按它算。没有代码需要遍历它的内部结构。

`hooks`/`variables`/`world_rules` 装的是**可执行单元**。引擎按 `hook` 分派、按 `variables` 校验、按 `world_rules` 求值。它们必须有结构。

**为什么 `world_rules` 与 `Entity.rules` 必须同构**

引擎在每个 hook 上会同时收集 world 规则与 entity 规则，排序后统一执行。若两者结构不同，就必须写一个合并器——**合并器会成为整个系统最脏的地方**。

### 2.2 ModulePack

一个具体剧本。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `ModuleId` | 否 | `builtin:core:shadow-over-arkham:1.0.0` |
| `title` | `string` | 否 | |
| `version` | `SemVer` | 否 | |
| `world_ref` | `WorldId` | 否 | → `World.id` |
| `authors` | `string` | 是 | |
| `players_min` / `players_max` | `int` | 否 | 见下方注 |
| `difficulty` | `string` | 是 | |
| `source` | `'builtin' \| 'imported'` | 否 | |
| `owner_user_id` | `UserId` | 是 | |

> **`players_min/max` 的用途被扩展了。** 它不只是「建议人数」——《复足》的模组内容是队伍规模的函数：
> 「梦境之石总数约为调查员总数的一半」「冷蛛数量 = 未携带石头的调查员数量」。
> 这两个字段必须能被 spawn 表达式引用（`party.size`，见 §4.2）。

|---|---|---|---|
| `id` | `SceneId` | 否 | |
| `module_id` | `ModuleId` | 否 | |

> 《追书人》《鬼屋》《蛙蛙村》《死者》都是城市或园区内自由移动，`exits` 恒为空。

### 2.4 ★ Entity

**本次改动中最核心的一张表。** 吸收原 `module_clues` 与 `module_npcs`。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `EntityId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| `kind` | `EntityKind` | 否 | `npc \| monster \| item \| clue \| animal \| object` |
| `name` | `string` | 否 | |
| `content` | `text` | 是 | 线索内容 / 物品描述。**自由文本** |
| `public_persona` | `text` | 是 | 🟢 玩家可见的表面人设。**自由文本** |
| `secrets` | `text` | 是 | 🔴 真实底牌。**自由文本，受隔离约束（§6.1）** |
| `stats` | `StatBlock` | **是** | 属性块 |
| ★ `state` | `Record<string, Primitive>` | 否 | 初始状态。见 §4.5 |
| ★ `rules` | `Rule[]` | 否 | 挂载在本实体上的规则。见 §4.3 |

**为什么合并 `clues` 与 `npcs`**

原设计将 `module_clues` 定位为「最烧脑但也最关键的表」，其隐含假设是**模组的核心是线索网络**。

该假设仅对调查向模组成立。六个模组的核心结构：

| 模组 | 核心结构 |
|---|---|
| 追书人 | 线索网络 ✓ |
| 死者的顿足舞 | 线索网络 ✓ |
| 银之锁 | **因果链**（猫的生死） |
| 复足 | **携带物开关**（梦境之石） |
| 幸福蛙蛙村 | **说服** |
| 鬼屋 | **战斗 + 孤注一骰** |

若 schema 围绕线索网络生长，后四个模组会被挤成畸形。

而猫、梦境之石、小号、羊毛布料——它们是同一类东西：**有状态的、可被表达式引用的实体**。线索只是其中一种 `kind`。

**为什么 `stats` 必须可空**

《追书人》的邻居与看守、《鬼屋》的阿蒂/佐利/罗斯/金、《银之锁》的芭斯特，全都没有属性。
《蛙蛙村》原文直接写：「其他游客守密人随便编就行」。

字段若不可空，解析器会**逼迫 LLM 编造数字**。

**为什么 `rules` 挂在 Entity 而非 Module**

「僵尸不会闪避」是僵尸这个实体的规则，不是《死者》这个模组的规则。
同一模组内，人类形态的特纳与僵尸形态的特纳，规则不同。

### 2.5 Checkpoint

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `CheckpointId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| `skill` | `SkillRef` | 否 | 见下方注 |
| `difficulty` | `Difficulty` | **是** | 见下方注 |
| `on_success` / `on_fail` | `Outcome` | 否 | |
| `hidden` | `bool` | 否 | 暗骰 |

```
SkillRef   = SkillName | "@" Category     // "侦查" 或 "@交涉"
Difficulty = 'regular' | 'hard' | 'extreme'
```

> **`skill` 必须允许类别值 `@交涉`。**
> 《鬼屋》：友好→魅惑 / 气势汹汹→恐吓 / 据理力争→说服 / 忽悠→话术。
> 这不是四选一，是**行为→技能的映射规则**——玩家会干出第五种。写死 `skill` 反而是 bug（作者故意留的开口）。
> 但候选集必须给出，否则 LLM 会选出玩家没有的技能。

> **`difficulty` 必须可空。**
> 《蛙蛙村》：说服信使——不阐述=困难 / 阐述合理=普通 / 阐述精彩=普通+奖励骰。
> 难度由玩家自然语言的质量决定，**运行时才知道**。
> 为空表示：由 LLM 在上游求值，输出枚举写入状态槽（见 §6.4）。

### 2.6 ★ SanTrigger

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `SanTriggerId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| ★ `kind` | `SanKind` | 否 | SAN 的六种形态 |
| ★ `source_tag` | `string` | 是 | 累计封顶的分组键 |
| `loss` | `SanExpr` | 是 | `"0/1d6"` 等 |
| `condition` | `text` | 是 | 触发条件（自由文本） |

```
SanKind =
  | 'check'         // ① 检定式        0/1d6
  | 'flat'          // ② 无检定固定损失  读日记 −1d4
  | 'direct'        // ③ 精神伤害      对抗胜 → 直接扣 1d6，不走 SAN 检定
  | 'max_reduce'    // ④ 上限扣减      +2% 神话 → SAN_max 99→97
  | 'gain'          // ⑤ 回复          +1d6 / +2d6
  | 'capped'        // ⑥ 累计封顶      同源合计 ≤ 6
```

**六种形态的出处**

| 形态 | 模组 | 原文 |
|---|---|---|
| ① `check` | 全部 | 「理智损失 0/1D6」 |
| ② `flat` | 追书人、鬼屋 | 「与道格拉斯交谈后损失 1d4 理智」 |
| ③ `direct` | 蛙蛙村 | 信使精神冲击：POW 对抗胜 → 直接扣 1d6 |
| ④ `max_reduce` | 鬼屋 | 读《伊波恩之书》：+2% 神话，SAN 上限 99→97 |
| ⑤ `gain` | 四个模组 | 「获得 1D6 的理智回复」 |
| ⑥ `capped` | 追书人、死者 | 「两次理智损失之和最大为 6 点」 |

> **③ 尤其重要。** LLM 会习惯性地为它插入一次 SAN 检定——因为绝大多数 SAN 损失都要检定。
> **必须用字段名切断这个惯性。**

> **⑥ 注意语义：是累计封顶，不是去重。**
> 《追书人》一场内触发三次同源 SAN，要求**累加后与 6 比较**。
> LLM 会当成「见过就跳过」——语义完全不同。
>
> 《追书人》与《死者》独立出现同一规则（COC 7e 的「习惯恐惧」），
> 说明该规则属于 `World.world_rules`，但 `source_tag` 由模组提供。

### 2.7 ★ WinCondition

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `WinConditionId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| ★ `expr` | `Expr` | 否 | 引用 `entity_states` 的布尔表达式。见 §5.2 |
| ★ `is_ending` | `bool` | 否 | `false` 表示状态回滚，非终局 |
| `text` | `text` | 否 | 结局文本 |

**为什么不能是「找齐哪些核心线索」**

| 模组 | 结局条件 |
|---|---|
| 复足 | `has_dreamstone && location == 'outside'`（未携带 × 离开 = 死） |
| 银之锁 | 救猫 → 猫活 → 走廊倒地 → 猫抓瞎 NPC → NPC 杀猫 → 锁解除 |
| 死者 | 阻止特纳**而不杀他**（杀了他会复活，且不再需要呼吸，更难杀） |
| 蛙蛙村 | 闪避**成功** = 失去永久幸运 +2d6 |

结局条件是**表达式**，地点与线索只是表达式的项。

**为什么需要 `is_ending`**

《银之锁》「没救猫 → 被抓回房间 → 重来」是**状态回滚**，不是终局。
《复足》「留守 = 消耗战」同理。

不加这个字段，LLM 会把它演成 bad ending 并结束游戏。

## 四、嵌套结构定义

> 本节展开所有「**有代码读取其内部键**」的结构。
>
> 判据：`public_persona` / `secrets` / `description` 等只被 LLM 读取的字段，
> 在物理层是 TEXT，在逻辑层就是「自由文本」，**不在此展开**。

### 4.1 HookDef

```
HookDef {
  name:      HookName
  signature: { accepts: VarName[], returns: 'void' | 'bool' | VarName }
  position:  int          // 在流水线中的次序，用于文档与校验
}
```

**COC 7e 的 12 个 hook**

战斗不是一次 `damage → hp` 的映射，而是一条流水线。每一步都是一个可被外部规则拦截或改写的暂停点：

```
声明攻击              on_attack_declare
决定用哪个技能         ─┐
计算难度              on_difficulty_calc     [死者: 人数→难度]
掷攻击骰              on_attack_roll
比较成功等级           ─┘                    [鬼屋: 六层分支]
防守方能否闪避         on_dodge_declare       [死者: 僵尸不能]
掷闪避骰              on_dodge_roll
判定命中              on_hit_resolve
掷伤害骰              on_damage_roll         [死者: 枪弹 vs 刀]
护甲扣减              on_armor_apply         [鬼屋: 血肉防护]
写入 HP               on_hp_write
重伤判定              on_major_wound         [死者: 无视]
死亡                  on_death
回合结束              on_turn_end            [蛙蛙村: 拖拽/吞食]
```

**Hook 不是「事件」**

| | 语义 | 能否改写 |
|---|---|---|
| 事件 | 「已经发生了，通知你一声」 | ❌ |
| Hook | 「即将发生，引擎在等你的返回值」 | ✅ |

```python
# 事件
target.HP -= damage;  emit("damaged")      # 已经扣了

# Hook
damage = fire("on_damage_roll", damage)    # 还没扣，规则说扣多少
target.HP -= damage
```

若只实现事件，《死者》的「僵尸被枪弹命中只损 1 点耐久」无法实现——伤害已经扣完了。

`signature` 决定了每个 hook 的干预能力：
- `on_damage_roll` 接收 `damage`、返回 `damage` → 能**改写**
- `on_dodge_declare` 返回 `bool` → 能**拦截**

---

### 4.2 VariableDef

规则的 `when` 表达式只能引用此表内的变量。**这份列表就是引擎能力的边界。**

```
combat.round
self.HP / self.HP_max / self.flesh_ward
self.rounds_without_damage
self.damage_taken_this_round
attack.type / attack.name
damage.type / damage.value
target.form
party.size / party.max_STR
```

> **计数器类变量必须由引擎自动维护。**
> `self.rounds_without_damage` 不能指望规则自己累加——规则只能读它。

> **导入校验**：规则引用表外的键（如 `self.mood`）→ 校验失败，打回重新生成。

---

### 4.3 ★ Rule

```
Rule {
  hook:     HookName
  when:     Expr                                    // 布尔表达式，见 §5.2
  then:     Op                                      // 见 §4.4
  mode:     'append' | 'override' | 'forbid'
  priority: int                                     // 默认 module 低于 world
}
```

**`mode` 的语义**

| mode | 含义 | 例子 |
|---|---|---|
| `append` | 我加一条，前面的照常执行 | 鬼屋·血肉防护（叠加在护甲之后） |
| `override` | 我执行，同 hook 上 `source == 'world'` 的规则跳过 | 复足·固定阈值检定 |
| `forbid` | 这个 hook 整个不执行 | 死者·僵尸无视重伤 |

**执行流程**

```
引擎在每个 hook 上：
  1. 收集 world_rules + module rules + entity rules
  2. 按 priority 排序
  3. 逐条执行
       遇 forbid   → 跳过整个 hook
       遇 override → 跳过所有 source == 'world' 的规则
```

> **不设 `invert` 模式。**
> 直觉上《死者》「INT 检定成功 → 更糟」像是「把输出反过来」，但实际不是：
>
> ```
> world:  INT 成功 → 保持清醒 ／ INT 失败 → 临时疯狂
> 死者:   INT 成功 → 完全理解 → 1D10 小时疯狂 ／ INT 失败 → 保持清醒
> ```
>
> 反转的是**语义**，不是布尔值。这是 `override`（替换后果），不是反转。
> `invert` 看似优雅，但会诱使建模者去表达「反过来」——而真实模组从不真的反转。

**为什么不能用「一堆具名布尔字段」**

《死者》的僵尸有四条规则：枪弹只损 1 耐久 / 刀伤减半 / 无视重伤 / 不会闪避。

写成 `ignores_major_wound: true` 这类字段，等于**用 JSON 写 if-else**。第八个模组来了再加一个。

更隐蔽的问题是**作用顺序表达不出来**：鬼屋的「血肉防护」与追书人的「火器伤害减半」若同时存在，先减半还是先扣甲？结果差一倍。
用字段的话，这个顺序藏在引擎代码里，模组作者无法覆盖。用 `(hook, priority)` 就显式了。

---

### 4.4 Op（算子表）

```
Op =
  | { set: Value }
  | { scale: number, round?: 'floor' | 'ceil' }
  | { add: Value }
  | { absorb: Expr, decrement: VarName }
  | { forbid: true }
  | { force: ActionName }
  | { apply_condition: ConditionId }
  | { spawn: { count: Expr, type: EntityId } }
  | { trigger: EventName, expires_in_rounds?: int }
  | { oppose: { a: Expr, b: Expr }, on_win: Op, on_lose: Op }
```

**六模组实证：算子集合已收敛**

```jsonc
// 死者·僵尸
[{"hook":"on_dodge_declare","when":"always","mode":"forbid"},
 {"hook":"on_damage_roll","when":"damage.type=='firearm'","then":{"set":1}},
 {"hook":"on_damage_roll","when":"damage.type!='firearm'",
  "then":{"scale":0.5,"round":"floor"}},
 {"hook":"on_major_wound","when":"always","mode":"forbid"}]

// 鬼屋·血肉防护（有状态、会耗尽、有时限）
[{"hook":"on_armor_apply","when":"self.flesh_ward > 0","priority":10,
  "then":{"absorb":"min(damage, self.flesh_ward)","decrement":"self.flesh_ward"}}]

// 复足·冷蛛转换目标（「4 轮内未造成伤害则换目标」）
[{"hook":"on_turn_end","when":"self.rounds_without_damage >= 4",
  "then":{"force":"switch_target"}}]

// 蛙蛙村·长舌拖拽（跨回合 + 后续对抗）
[{"hook":"on_hit_resolve","when":"attack.name=='长舌鞭笞'",
  "then":{"trigger":"pending_drag","expires_in_rounds":1}},
 {"hook":"on_turn_end","when":"self.pending_drag",
  "then":{"oppose":{"a":"self.STR","b":"target.STR"},
          "on_win":{"trigger":"swallow"},"on_lose":{"clear":"pending_drag"}}}]
```

> 六个模组跑完，**没有出现第 13 个 hook，也没有出现新算子**。
> 这是「现在就定死 hook 表」的依据——再等下去不会得到更多信息，只会积累更多需要迁移的数据。

### 4.5 ★ EntityState

```
state: Record<string, Primitive>
Primitive = boolean | number | string
```

**键名由模组导入时生成，不预先枚举。**

六模组实测必须存在的键：

| 键 | 模组 | 用途 |
|---|---|---|
| `cat.alive` | 银之锁 | 被 `WinCondition.expr` 引用 |
| `pc.has_dreamstone` | 复足 | 被 `WinCondition.expr` 引用（携带 = 生） |
| `turner.form` | 死者 | 被 `Rule.when` 引用（人类/僵尸两套规则） |
| `turner.death_count` | 死者 | 马努斯科被杀三次——`alive` 不能是单向 bool |

**判据：哪些状态该落库，哪些走提示词**

> **这个键有没有出现在任何 `Rule.when` 或 `WinCondition.expr` 里？**
> - 有 → 落 `state`
> - 没有 → 走 `public_persona` 自由文本，**不落库**

按此判据，以下状态**不应**落库：

```
npc_doctor.attitude      只影响 LLM 怎么演
cat.trusts_player        只影响 LLM 怎么演
```

原因：引擎读不了上下文，只能读变量。而 LLM 读不了变量之外的东西吗？读得了——**上下文它读得更好**。
态度是连续量，枚举成 `friendly | hostile` 反而失真。

> 该判据可实现为一个**符号表检查**，在导入流程中自动运行。

---

### 4.6 ★ Condition

```
Condition {
  id:            ConditionId
  timer?:        { after: Duration } | { every: Duration } | { at: Timestamp }
  check?:        SkillRef | Expr        // 复足: "CON * (6 - stage)"
  on_fail?:      Effect
  on_success?:   Effect
  stages?:       Stage[]                // 多阶段状态机
  reversible_until?: int
  repeat_until?: Expr                   // "healed || dead"
  desc:          text                   // ← 自由文本，不参与计算
}

Stage {
  n:      int
  check?: Expr                          // 难度可以是阶段的函数
  delta?: Record<AttrName, DiceExpr>    // 增量
  set?:   Record<AttrName, int>         // 赋值
  desc:   text
}

Effect {
  duration?:  DiceExpr
  permanent?: Record<AttrName, DiceExpr>
  apply?:     ConditionId
}
```

**为什么不能是字符串数组**

| 模组 | 机制 |
|---|---|
| 鬼屋 | 科比特爪伤：命中 → 幸运检定 → **一天后** → CON 检定 → 分支 → **循环直到痊愈或死亡** |
| 复足 | 感染阶段 0–5，难度 = 阶段的函数（0 幸运 / 1 体质×5 / 2 体质×4 / … / 5 体质×1） |
| 蛙蛙村 | 变蛙四阶段。阶段 2 可逆、阶段 3 起不可逆 |

**这不是记忆问题，是调度问题。**

「一天后」需要有人在一天后把它叫醒。LLM 是被动的，它只在玩家说话时被调用。
玩家在游戏内第二天早上说「我去餐厅」——**没有任何东西会提醒 LLM 该插入一次 CON 检定**。

它正在回答「餐厅长什么样」，不会自己想起「你昨天被抓伤了」。

**`delta` 与 `set` 必须分开建键**

《蛙蛙村》阶段 2 是 `跳跃 +4d6 / APP −2d6`（增量），阶段 3 是 `APP = 25 / 跳跃 = 85`（赋值）。

**LLM 会把「APP 变成 25」误算成「APP −25」。** 这是纯语义歧义，只能靠字段名消除。

**`desc` 是唯一的自由文本子字段**

引擎负责「何时」与「多少」，LLM 负责「像什么」：

```jsonc
{
  "id": "corbitt_infection",
  "timer": {"after": "24h"},
  "check": "CON",
  "on_fail":    {"duration": "1d10d", "permanent": {"CON": "-1d10"}},
  "on_success": {"duration": "1d6d"},
  "repeat_until": "healed || dead",
  "desc": "伤口周围皮肤泛灰白，触碰时有异物感"   // ← 不参与任何计算
}
```

七个键。**再简化就只能删掉 `timer`——然后科比特的爪伤永远不会发作。**

---

### 4.7 ★ LedgerEntry

```
LedgerEntry =
  | { count: int, window_h: int, then: Effect }       // 计数器 + 时间窗
  | { active: bool, expires_in_h: int, effect: Op }   // 单次事件 + 时长
  | { accumulated: int, cap: int }                    // 累计封顶
  | { regen: DiceExpr, per_h: int }                   // 周期回复
```

| 实例 | 模组 | 原文 |
|---|---|---|
| 计数器 + 时间窗 | 蛙蛙村 | 「24 小时内失败两次或以上意志检定 → 此后类似检定承受惩罚骰」 |
| 单次事件 + 时长 | 蛙蛙村 | 「陷入强烈幻觉 → 接下来几小时内再次接触池水承受惩罚骰」 |
| 累计封顶 | 追书人 | 「两次理智损失之和最大为 6 点」 |
| 周期回复 | 鬼屋 | 「MP 每小时回复 1 点」（作者原文特别提醒 KP「小心记录」） |

> LLM 数不清「24 小时内两次」——它需要回溯整个上下文数掷骰次数，并判断是否落在窗口内。长对话必错。
>
> 而《鬼屋》作者亲自提醒 KP「小心记录 MP」——**连人类都容易忘**。这是纯记账。

## 五、值对象与语法

### 5.1 骰式

```
DiceExpr  = "1d6" | "2d6" | "1d4+2" | "6+1d4+2" | "1d10+2"
SanExpr   = DiceExpr "/" DiceExpr      // "0/1d6" 成功/失败
```

> **作者已替 KP 展开的骰式，照抄。**
> 《鬼屋》的 `6+1d4+2` 是「穿刺武器极难成功造成最大伤害」规则的展开值。
> 引擎**不必理解它的来历**，照抄字符串解析即可。
>
> 反向还原成规则表达式是错误的方向。

---

### 5.2 ★ Expr（表达式语法）

用于 `Rule.when` 与 `WinCondition.expr`。

```ebnf
expr     = or_expr
or_expr  = and_expr { "||" and_expr }
and_expr = cmp_expr { "&&" cmp_expr }
cmp_expr = add_expr [ ("==" | "!=" | "<" | "<=" | ">" | ">=") add_expr ]
add_expr = mul_expr { ("+" | "-") mul_expr }
mul_expr = unary { ("*" | "/") unary }
unary    = [ "!" ] primary
primary  = number | string | bool
         | var_ref
         | func "(" [ expr { "," expr } ] ")"
         | "(" expr ")"

var_ref  = ident { "." ident }         // damage.type / self.HP / party.size
func     = "floor" | "ceil" | "min" | "max" | "count"
```

**无循环、无递归、无函数定义。**

这是一个纯表达式语言，约二三百行即可实现，且**天然安全**——不会死循环，不会越权。

> 六个模组的所有规则，**没有一条需要循环**。
> 「每战斗轮 1D6 力量伤害」是引擎的回合循环在驱动，规则本身只描述单轮。

**关键约束：`when` 里不能放语义判据**

```jsonc
{"when": "玩家表述得很精彩"}                 // ❌ 求值器算不了
{"when": "roleplay_tier == 'excellent'"}    // ✅
```

语义判据由 LLM 在**上游**求值，输出枚举写入状态槽，规则只读枚举。
**软判据与硬求值必须分在两个阶段。**（详见 §6.4）

---

### 5.3 其它值对象

```
Primitive   = boolean | number | string
Duration    = "24h" | "10m" | "1d10d"
SkillRef    = SkillName | "@" Category
Difficulty  = 'regular' | 'hard' | 'extreme'
Attributes  = Record<'STR'|'CON'|'SIZ'|'DEX'|'APP'|'INT'|'POW'|'EDU', int>
```

## 六、约束与不变式

> 本节是数据模型中**最容易被遗漏、也最有价值**的部分。
> 这些约束无法由类型系统表达，必须由代码或导入校验保证。

### 6.1 信息隔离

> **`Entity.secrets` 永不进入「生成旁白」的 LLM 请求上下文。**

这不是字段约束，是**提示词组装层的工程约束**。

理由：`secrets` 一旦进入上下文就泄漏，**事后收不回**。
这与「LLM 会不会算错」是两类不同的问题。

同理适用于 `Checkpoint.hidden`（暗骰）：让 LLM「假装不知道自己掷出了什么」不可靠。

---

### 6.2 EntityState 的引用完整性

> **`Entity.state` 的每个键，必须至少被一处 `Rule.when` 或 `WinCondition.expr` 引用。**
> 未被引用的状态应改为 `public_persona` 自由文本。

可实现为导入流程中的符号表检查。

---

### 6.3 ★ 状态写入口唯一

> **LLM 不得直接写 GameState。它只能提议 op，由引擎校验后执行。**

```jsonc
// LLM 输出
{
  "narration": "猫扑到男人脸上，指甲深深陷进他的眼睛。",
  "ops": [
    {"op": "modify", "path": "npc.kidnapper.blind", "set": true},
    {"op": "modify", "path": "cat.alive",           "set": false}
  ]
}
```

引擎收到后：
1. 校验 `path` 存在于 `entity_states` 的键空间
2. 校验该 op 在当前场景的 `allowed_ops` 内
3. 执行、记账（写 `Event`）、返回新状态

**为什么这条约束不可省**

六模组中有一类规则，LLM 的遗漏具有**系统性偏向**：

| 模组 | 规则 | 漏掉之后 |
|---|---|---|
| 死者 | 僵尸不会闪避 | 僵尸只是变强了一点 |
| 死者 | 特纳绝不出借小号 | 玩家拿到了小号 |
| 复足 | 手拉手无效 | 大家都活着 |
| 复足 | 蜘蛛海源源不绝 | 玩家杀退了蜘蛛 |
| 银之锁 | 猫必须死 | 玩家救下了猫 |
| 鬼屋 | INT 检定成功 → 更糟 | 玩家过检定，安全 |

**六条全是「限制玩家」的规则。六条漏掉之后玩家都会更开心。这不是巧合。**

且这类失败是**静默的**：

```
引擎约束失败：玩家拿了不该拿的钥匙 → 抽屉打不开 → 玩家立刻发现
提示词约束失败：猫活下来了          → 剧情继续 → 没有任何人发现
```

前者是崩溃，可以修。后者不能——因为你不知道它发生了。

> 因此：**凡是决定玩家能否通关的状态，不能交给一个想让玩家通关的东西来记。**

---

### 6.4 软判据与硬求值分离

> **`Rule.when` 只能引用 `VariableDef` 表内的变量，不能包含语义判据。**

《蛙蛙村》说服信使的三档难度：

```
玩家什么都不说，光掷骰      → 困难
讲了个合理的理由            → 普通
讲得精彩                   → 普通 + 奖励骰
```

正确的分工：

```
阶段 1  LLM 求值语义判据 → 输出枚举 roleplay_tier ∈ {none, reasonable, excellent}
阶段 2  引擎读枚举 → 查表得 difficulty → 掷骰
```

**枚举硬、判据软。** 若三档不写死，LLM 会自造第四档，或直接放水判成功。

---

### 6.5 World 与 Module 规则同构

> **`World.world_rules` 与 `Entity.rules` 必须是同一个 `Rule` 类型。**

一旦结构不同，就必须写一个合并器——**合并器会成为整个系统最脏的地方**。

**推论：`override` 若反复出现，说明 World 定义得不够。**

《复足》的「撬门（力量 20）」是固定阈值检定，在该模组中出现四次（撬门/破窗/破门/上锁门）。
若让它用四次 `override` 打补丁，是错的——**它是通用机制，应补进 `World.definition` 的检定原语列表**。

Module 的 `override` 应当是稀有的。

---

### 6.6 导入校验：路口清单

> **静默失败只能靠路口清单来兜。**

LLM 生成 `{"then": {"forbid": true}}` 的命中率，明显低于 `{"then": {"scale": 0.5}}`。
它不愿意生成限制玩家的规则。

「僵尸不会闪避」漏掉之后，系统不报错，玩家不知情。

**兜底方式**：导入完成后，对每个 `kind ∈ {npc, monster}` 的实体，逐个质询 12 个 hook——尤其是空着的那些：

```
on_dodge_declare    → 【空】   这个怪物真的能闪避吗？
on_damage_roll      → 火器减半
on_major_wound      → 【空】   这个怪物真的会受重伤吗？
```

> LLM 回答「它有什么**不能做**的事」时很准。**它只是不会主动想起来要问。**

**这正是 hook 表相对「一堆布尔字段」的隐性收益：分母是明确的 12。**
在「塞数据」的世界里，「这个模组我抽取了多少信息」这个问题**没有分母**。

---

### 6.7 数量的动态缩放

> **`ModulePack.players_min/max` 必须能被 `spawn` 表达式引用。**

《复足》：
- 「梦境之石总数约为调查员总数的一半」
- 「冷蛛数量 = 未携带石头的调查员数量」

若写死数字，2 人团与 5 人团的难度完全崩坏。

---

## 七、逻辑模型 → 物理模型的投影

**两者不是「对齐」关系，是投影关系。**

```
逻辑模型                     物理模型
─────────────────────       ──────────────────────
Entity {                     module_entities (
  id:    EntityId              id     TEXT PK,
  kind:  EntityKind            kind   TEXT,
  state: Record<...>           state  JSONB,    ← 一列吃掉整个 Record
  rules: Rule[]                rules  JSONB     ← 一列吃掉整个数组
}                            )

Rule {                       （不建 rules 表）
  hook:     HookName
  when:     Expr
  then:     Op
  mode:     ...
  priority: int
}
```

`Rule` 在逻辑模型里是**一等公民**，在物理模型里被折叠进一列 JSONB。

**这不矛盾。** 物理模型选择不建 `rules` 表，是因为当前没有「查询所有使用了 `on_dodge_declare` 的规则」这种需求。

若将来出现该需求（如「统计所有模组在哪些 hook 上挂了规则」），把 `rules` 抽成表即可——**逻辑模型一个字不用改**。

> **物理模型会因查询模式而变，逻辑模型不会。**
> 这就是两份文档必须分开的原因。

**折叠决策表**

| 逻辑结构 | 物理承载 | 折叠理由 |
|---|---|---|
| `Rule[]` | `entities.rules JSONB` | 无按规则检索的需求 |
| `Condition[]` | `characters.conditions JSONB` | 随角色整体读写 |
| `HookDef[]` | `worlds.hooks JSONB` | 行数 ~12，随 world 整体加载 |
| `Record<EntityId, ...>` | `rooms.entity_states JSONB` | 随房间整体读写 |
| `Event` | `events` 表 | ★ **不折叠**——需按 `(room_id, id)` 范围查询 |

---

## 八、MVP 范围

### 8.1 本期搁置：物品实体化

`Character.equipment` 保持 `string[]`，不加 `id`，不建物品实体。

**已知会漂的地方，本期接受**：

《银之锁》三把钥匙来源不同（挂画 / 床下 / 通风管），对应三个抽屉。
三把钥匙的 `name` 可能都是「一把小钥匙」——**语义上无法区分**。LLM 可能串。

> **但注意一个例外**：《复足》的梦境之石**不属于「物品」问题**。
>
> 它是 `WinCondition.expr` 的输入（携带 = 生，不携带 = 死）。
> 按 §4.5 的判据，它必须落 `entity_states`——但只需要一个 `bool`，**不需要建物品实体**。
>
> **落库的是「值」，不是「物品」。**
> 若这两件事被合并处理，§4.5 会被本条顺手废掉。

---

### 8.2 本期建字段、暂不填充

`Entity.rules` 与 `World.world_rules` 字段建出，MVP 阶段可为空数组。

战斗规则暂时写进 `secrets` / `public_persona`，由 LLM 执行。

**前提是 §4.1–4.4 已落地**——hook 清单与结构先定死，后续接入规则引擎时**不需要改表**，只需开始往 `rules` 里写东西。

> **为什么 hook 表不能一起搁置**
>
> 1. 它是 `rules` 字段的校验依据。没有 hook 清单，写进去的 JSON 无从校验
> 2. 它是架构而非字段。其余改动都是 `ALTER TABLE ADD COLUMN`；若 MVP 用了别的形式（如具名布尔开关），迁移时要重写所有已导入模组
> 3. 六模组已证明它收敛：12 hook / ~15 变量 / ~10 算子，跑完没有增加

### 8.4 已删除的字段

| 字段 | 处置 | 理由 |
|---|---|---|
| `module_clues.reachable_by` | **删除** | 见下 |
| `module_npcs.linked_clue_ids` | **删除** | 同上 |
| `rooms.current_scene_id` | 迁移至 `Character.location` | 分头行动 |

**`reachable_by` 为什么必须删**

线索的「内容」与「获取路径」是两个东西：

```
内容    「一小片深灰色羊毛布料」        唯一的，全局的
路径    在书房侦查 / 找管家聊天         多个，且属于场景
```

中间表 `module_scene_clues` 只存 `(scene_id, clue_id)`，**表达不了 `via`**——「在书房靠侦查，在门厅靠跟管家聊」。

于是 `reachable_by` 被迫诞生，把路径信息拉回线索身上。**同一份信息存了两遍，且可能对不上。**

正确归属：路径属于 `Scene.contents`（见 §8.3），内容属于 `Entity`。

**「核心线索至少两条路径」原则降级为警告**

原设计：少于 2 条路径直接拒绝导入。
改为：警告 + 允许 `intentional_single_path: true` 显式标记。

故意的单路径在六模组里很常见：

- 《银之锁》钢钳是救猫的唯一方法
- 《复足》感染只能通过切开囊肿取出幼蛛治疗
- 《死者》说服疯子需极限难度检定——「除此之外的任何结果都毫无作用」

> 《死者》那句是作者在**预防 KP 心软**。硬拒绝的话这三个模组一个都进不来。

## 九、换规则系统时会发生什么

```
COC 的 hook 表:  on_dodge_declare / on_major_wound / on_san_check
OSE 的 hook 表:  on_thac0_calc    / on_saving_throw / on_morale_check
```

变的是：
- `World.hooks` 的内容
- `World.variables` 的变量名（`self.闪避` → `self.AC`）

**没变的是**：`Rule` 结构、表达式求值器、算子表——**一行没动**。

> **「哪些字段属于 Core」这个问法从一开始就错了——字段本身就是 Module 的产物。**
>
> - **Core** 定义：规则如何被表达和求值
> - **Module** 定义：有哪些 hook、hook 上有哪些可读变量

## 附录 B：判断流程

面对任一 JSON 字段，依次回答：

```
Q1  有代码读它内部的键吗？
      没有 → 逻辑模型写「自由文本」，结束
      有   → 继续

Q2  键名能提前枚举吗？
      能   → 展开为具名结构（Rule / Condition / HookDef）
      不能 → 写 Record<string, T> + 约束条款

Q3  有跨字段的不变式吗？
      有   → 写入 §6「约束与不变式」
```

**实例**

| 字段 | Q1 | Q2 | 结果 |
|---|---|---|---|
| `World.hooks` | 有 | 能（12 个） | 展开 `HookDef` |
| `Entity.rules` | 有 | 能（5 键） | 展开 `Rule` + §6.5 |
| `Entity.state` | 有 | **不能** | `Record` + §6.2 / §6.3 |
| `Character.conditions` | 有 | 能（7 键） | 展开 `Condition`；`desc` 子字段标注「不参与计算」 |
| `Entity.public_persona` | **没有** | — | 自由文本 + §6.1 隔离约束 |
| `Event.payload` | 半 | 按 `type` 分 | discriminated union |
