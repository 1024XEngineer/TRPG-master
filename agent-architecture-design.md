# 多 Agent 架构设计方案

> **文档定位**：定义 AI 跑团主持系统的 Agent 划分、MVP 落地方案，以及游戏运行期采用单 Agent 或双 Agent 的取舍与演进路径。
>
> **适用阶段**：当前 MVP 设计与后续架构扩展。
>
> **核心结论**：顶层按生命周期划分模块，Agent 边界按上下文、权限、输出目标和失败模式决定。MVP 阶段采用“模组解析 Agent + 导入审查 Agent + 主持编排 Agent + 确定性规则引擎”；游戏运行期先由一个主持编排 Agent 前后贯穿，待系统跑通后再根据实际指标决定是否拆成“行动理解 Agent + 回复编排 Agent”。

---

## 1. 背景与问题

本项目希望实现一个能够主持 COC 等高自由度跑团游戏的 AI 系统。

系统必须同时满足两类目标：

1. **自由性**：玩家可以使用自然语言做出大量非预设行为，AI 需要理解、回应并维持沉浸感。
2. **确定性**：关键规则、检定、状态、计时、结局和资源变化不能只依赖大模型的记忆与临场发挥。

因此，系统不能简单理解为“一个大模型扮演主持人”，而应拆分为：

```text
LLM / Agent：理解自然语言、进行语义判断、组织叙事
确定性引擎：执行流水线、Hook、Rule、检定和状态校验
数据层：保存 Content、GameState 和 EventLog
```

在此基础上，团队提出了多 Agent 架构，但目前存在两种不同的划分思路：

### 1.1 按系统生命周期划分

```text
模组解析（游戏前）
对话互动（游戏中）
总结复盘（游戏后，可后置）
```

### 1.2 按运行期职责划分

```text
主持
剧情编排
角色行动处理
边界审查
```

这两种思路并不冲突。前者是在划分**系统阶段**，后者是在划分**阶段内部职责**。真正需要决定的是：哪些职责只需成为同一 Agent 内部的工作步骤，哪些职责值得成为独立 Agent。

---

## 2. Agent 划分原则

不应为了体现“多 Agent”而机械拆分。一个职责是否值得成为独立 Agent，建议从以下五个维度判断。

### 2.1 上下文是否不同

例如：

- 模组解析需要读取完整模组文本；
- 运行时主持只需要当前场景、角色状态和相关规则；
- 玩家可见叙事模型不应读取全部秘密和暗骰真相。

如果两个职责需要明显不同的上下文，适合拆分。

### 2.2 权限是否不同

例如：

- 模组解析 Agent 可以生成 Content 草稿；
- 运行时 Agent 不能直接修改 GameState；
- 叙事 Agent 不应读取或修改隐藏状态；
- 任何 Agent 都不应绕过引擎直接写关键游戏事实。

如果权限边界明显不同，适合拆分。

### 2.3 输出目标是否不同

例如：

- 行动理解强调严格、稳定的结构化 JSON；
- 回复编排强调自然语言、角色感和沉浸感；
- 导入审查强调找错、找漏，而不是继续创作。

目标越不同，拆分价值越高。

### 2.4 失败模式是否不同

例如：

- 模组解析可能漏掉规则；
- 行动理解可能选错流水线；
- 回复编排可能泄露秘密或擅自改写结果；
- 边界错误可能导致非法状态被写入。

如果拆分后能更清晰定位错误，适合拆分。

### 2.5 拆分收益是否大于成本

每新增一个 Agent，都会增加：

- 模型调用次数；
- 响应延迟；
- Token 成本；
- 中间协议设计成本；
- Agent 间上下文损失；
- 调试和一致性维护难度。

因此应遵循：

> **先细分职责，再决定是否物理拆成多个 Agent。**

---

## 3. 总体架构定稿

当前建议采用“两个生命周期模块、三个 Agent、一个确定性引擎”。

```text
阶段一：模组导入
  1. 模组解析 Agent
  2. 导入审查 Agent

阶段二：游戏运行
  3. 主持编排 Agent
  4. 确定性规则引擎（不是 Agent）

阶段三：总结复盘
  后续扩展，MVP 暂不实现
```

总体流程：

```text
模组文件
  ↓
模组解析 Agent
  ↓
结构化 Content 草稿
  ↓
导入审查 Agent
  ↓
校验通过的模组数据
  ↓
玩家输入
  ↓
主持编排 Agent
  ↓
ActionPlan / Check / Op 提议
  ↓
确定性规则引擎
  ↓
ConfirmedOutcome + GameState + EventLog
  ↓
主持编排 Agent
  ↓
玩家可见回复
```

## 4. Agent 1：模组解析 Agent

### 4.1 职责

模组解析 Agent 负责将模组原文转化为结构化 Content 数据，包括但不限于：

```text
ModulePack
Scene
Entity
Checkpoint
SanTrigger
WinCondition
Rule
Condition
初始 Entity.state
```

它类似“编译器前端”：

```text
模组 PDF / 文本
→ 识别内容与机制
→ 映射到系统数据模型
→ 生成结构化草稿
```

### 4.2 示例

原文：

```text
僵尸不会闪避，枪弹命中时只损失 1 点耐久。
```

输出示例：

```json
{
  "entity_id": "zombie",
  "rules": [
    {
      "hook": "on_dodge_declare",
      "when": "always",
      "mode": "forbid",
      "priority": 10
    },
    {
      "hook": "on_damage_roll",
      "when": "damage.type == 'firearm'",
      "then": {"set": 1},
      "mode": "append",
      "priority": 10
    }
  ]
}
```

### 4.3 权限边界

模组解析 Agent：

- 可以读取完整模组原文；
- 可以生成 Content 草稿；
- 不写 GameState；
- 不参与实时游戏；
- 不直接决定最终入库结果；
- 生成结果必须经过审查与程序校验。

### 4.4 为什么独立

它与运行期主持存在显著差异：

- 上下文不同：完整模组 vs 当前场景；
- 时效不同：离线长任务 vs 实时交互；
- 输出不同：结构化数据 vs 玩家可见叙事；
- 失败可重试：解析失败可重新生成，不影响即时体验。

---

## 5. Agent 2：导入审查 Agent

### 5.1 职责

导入审查 Agent 获取：

```text
原始模组文本
+
模组解析 Agent 的结构化结果
```

其目标不是继续创作，而是主动找错、找漏和找冲突。

主要检查：

1. 是否漏掉禁止类规则；
2. 是否漏掉条件满足后必然触发的事件；
3. 是否存在“成功反而更糟”的反直觉检定结果；
4. 哪些状态会被 Rule.when 或 WinCondition.expr 读取；
5. Rule 使用的 hook 是否存在；
6. Rule 引用的变量是否在 VariableDef 中；
7. 同一 hook 上是否缺少明确 priority；
8. Entity.state 是否包含无人引用的字段；
9. WinCondition 是否引用未定义状态；
10. secrets 是否错误进入公开字段；
11. Checkpoint 是否缺少成功、失败或 hidden 信息；
12. 当前引擎是否无法表达某个机制。

### 5.2 A/B/C/D 定向审查

建议审查 Agent 按四类问题主动反查：

```text
A 拒绝：这个实体有什么事情绝对不能做？
B 必然：条件满足后，有什么事件即使玩家不提也必须发生？
C 反转：哪些检定成功不一定是好事，或失败反而更安全？
D 值：哪些状态会影响规则、计时或结局？
```

### 5.3 为什么独立

同一个 Agent 自检容易产生确认偏差。解析 Agent 的目标是“完成生成”，审查 Agent 的目标是“证明生成结果可能有错”。两者目标天然不同。

推荐流程：

```text
生成
→ 审查
→ 自动修订或人工确认
→ 程序校验
→ 正式入库
```

## 6. 游戏运行期：主持编排 Agent

### 6.1 MVP 定稿

MVP 阶段先采用一个主持编排 Agent 前后贯穿：

```text
玩家输入
→ 主持编排 Agent 解析意图
→ 决定是否调用引擎
→ 引擎执行流水线和规则
→ 返回确定结果
→ 同一个主持编排 Agent 生成玩家可见叙事
```

这里的“一个 Agent”不代表内部没有细分，而是暂不将这些步骤物理拆成多个独立 Agent。

### 6.2 内部职责

主持编排 Agent 内部至少包含以下逻辑步骤：

```text
1. 读取当前可见上下文
2. 理解玩家意图
3. 拆分复合动作
4. 判断纯叙事行为或规则敏感行为
5. 匹配 Checkpoint 或构造临时检定
6. 选择一个或多个预设流水线
7. 生成 ActionPlan / Op 提议
8. 调用确定性引擎
9. 读取 ConfirmedOutcome
10. 扮演 NPC、描述场景、组织节奏
11. 输出玩家可见回复
```

### 6.3 纯叙事与规则敏感行为

不是所有玩家行为都调用引擎。

#### 纯叙事行为

特点：

- 不涉及严格数值结算；
- 不改变关键状态；
- 不影响结局；
- 不需要可追溯裁决。

示例：

```text
我看看窗外的雨。
我和酒保闲聊。
```

可由主持编排 Agent 直接回应。

#### 规则敏感行为

至少满足一项：

- 涉及掷骰；
- 涉及 HP、SAN、位置、物品或状态变化；
- 涉及关键线索或结局；
- 涉及特殊模组规则；
- 涉及时间推进、多人对抗或资源消耗。

这类行为必须进入引擎。

---

## 7. 确定性规则引擎不是 Agent

规则引擎负责：

```text
流水线
HookDef
VariableDef
Rule
Op
检定
GameState
Condition
Ledger
Invariant
EventLog
room_id 隔离
```

Agent 只能提议动作，不能直接修改关键状态。

正确流程：

```text
Agent 提议 Action / Op
→ 引擎检查合法性
→ 执行流水线和 Rule
→ 写 EventLog
→ 更新 GameState
→ 返回确定结果
```

边界审查不能主要依赖一个“边界审查 Agent”。以下约束必须由代码硬校验：

- refuse_ops；
- Rule.mode = forbid；
- 变量白名单；
- 状态不变式；
- GameState 唯一写入口；
- EventLog 一致性；
- room_id 隔离；
- 关键状态引用完整性。

原则：

> **语义判断可以交给 Agent，规则底线必须交给引擎。**

## 8. 游戏运行期：单 Agent 与双 Agent 两种方案

### 8.1 单 Agent 方案

```text
玩家输入
→ 主持编排 Agent
→ 引擎
→ 同一个主持编排 Agent
→ 玩家回复
```

同一个 Agent 同时负责：

- 引擎前的行动理解；
- 引擎后的叙事编排。

### 8.2 双 Agent 方案

```text
玩家输入
→ 行动理解 Agent
→ 确定性引擎
→ 回复编排 Agent
→ 玩家回复
```

#### 行动理解 Agent

负责：

- 识别玩家意图；
- 判断是否需要引擎；
- 匹配 Checkpoint；
- 选择流水线；
- 拆分复合动作；
- 生成结构化 ActionPlan。

#### 回复编排 Agent

负责：

- 根据引擎确定结果叙事；
- 扮演 NPC；
- 控制节奏和氛围；
- 隐藏玩家不应知道的信息；
- 不得重新裁决或修改结果。

---

## 9. 单 Agent 方案的优势与风险

### 9.1 优势

#### 语义连续性好

同一个 Agent 理解玩家原话，又读取引擎结果，能够保留：

- 玩家语气；
- 附带动作；
- 隐含意图；
- 表演细节；
- 前后叙事连续性。

#### 工程简单

- Agent 数量少；
- 中间协议较少；
- 调试链路较短；
- 更适合快速跑通 MVP。

#### 延迟和成本较低

即使工具调用前后需要两次推理，也可共享同一会话和上下文，不必维护两个独立 Agent 的信息交接。

### 9.2 风险

#### 上下文权限过大

同一个 Agent 可能同时看到：

- 隐藏 Checkpoint；
- NPC secrets；
- 暗骰结果；
- 尚未触发的剧情。

随后又由它直接生成玩家可见内容，存在泄密风险。

#### 可能推翻引擎结果

Agent 可能为了叙事流畅，把失败写成部分成功，或在语言上创造数据库中不存在的事实。

#### 职责过重

同一提示词需要同时约束：

```text
动作理解
流程选择
秘密保护
规则遵守
NPC 扮演
节奏控制
自然语言叙事
```

系统复杂后，稳定性可能下降。

---

## 10. 双 Agent 方案的优势与风险

### 10.1 优势

#### 职责和输出目标更纯粹

行动理解 Agent 只输出严格 JSON；回复编排 Agent 只生成自然语言。

```text
理解 Agent：准确、低温度、结构化
编排 Agent：自然、沉浸、角色一致
```

#### 信息权限可真正隔离

行动理解 Agent 可以读取动作判断所需的隐藏结构；回复编排 Agent 只读取经过脱敏的玩家可见结果。

例如暗骰失败后，回复编排 Agent 不需要知道：

```text
房间里确实有暗门，但玩家失败了。
```

它只需收到：

```text
玩家没有发现额外内容，请正常描述房间。
```

这比让同一个 Agent “知道但假装不知道”更可靠。

#### 更容易独立评测

行动理解 Agent 可评测：

- 意图分类准确率；
- 流水线选择准确率；
- Checkpoint 匹配准确率；
- 多动作拆解准确率。

回复编排 Agent 可评测：

- 是否忠实于引擎结果；
- 是否泄露秘密；
- NPC 人设一致性；
- 叙事质量。

#### 可使用不同模型

- 理解 Agent：便宜、快、结构化能力强；
- 回复 Agent：语言能力和角色扮演能力强。

### 10.2 风险

#### 上下文损失

理解 Agent 如果只输出最小动作结构，回复 Agent 可能丢失玩家语气、隐含意图和附带表演。

因此中间协议必须包含 `narrative_intent` 等信息。

#### 回复 Agent 仍可能重写事实

必须明确区分：

```text
confirmed_facts：必须遵守，不得修改
narrative_freedom：允许自由发挥
```

#### 成本和延迟增加

每轮通常需要两个独立模型调用，实时语音场景尤其敏感。

#### 协议复杂度增加

需要稳定定义：

```text
ActionPlan
ConfirmedOutcome
NarrationInput
```

否则两个 Agent 会互相丢失信息或产生矛盾。

## 11. MVP 选择：先采用单 Agent

当前阶段建议：

> **先采用一个主持编排 Agent 前后贯穿，优先跑通完整闭环。**

理由：

1. 当前首要目标是验证模组数据、流水线、规则和状态系统是否能协同运行；
2. 团队尚未积累足够的真实错误数据，无法判断拆分最能解决什么问题；
3. 单 Agent 能降低工程复杂度、延迟和调试成本；
4. 只要从一开始定义清晰的中间数据契约，后续仍可平滑拆分。

“先做一个”不等于将前后逻辑写死在一起。MVP 仍应保持两个清晰逻辑阶段：

```text
阶段 A：Action Interpretation
阶段 B：Narration Composition
```

初期可由同一个模型和同一个 Agent 服务承担。

---

## 12. 为未来拆分预留的数据契约

即使 MVP 使用单 Agent，也应从一开始定义以下中间结构。

### 12.1 ActionPlan

表示玩家行为经过理解后的结构化方案。

```json
{
  "route": "engine",
  "intent": "steal_item",
  "target": "guard_key",
  "pipeline": "check",
  "checkpoint_id": null,
  "check": {
    "skill": "sleight_of_hand",
    "difficulty": "regular"
  },
  "narrative_intent": {
    "approach": "假装喝醉接近守卫",
    "tone": "隐蔽、试探",
    "fallback": "被发现后推开守卫逃跑"
  }
}
```

建议字段：

```text
route
intent
target
pipeline
checkpoint_id
subactions
check
proposed_ops
narrative_intent
confidence
needs_clarification
```

### 12.2 ConfirmedOutcome

表示引擎已经确定、不可由叙事层修改的结果。

```json
{
  "facts": [
    "偷窃失败",
    "守卫发现玩家",
    "钥匙仍在守卫身上"
  ],
  "state_changes": [],
  "events": [],
  "next_required_action": "resolve_escape",
  "player_visible_constraints": [
    "不得宣称玩家已获得钥匙"
  ]
}
```

建议字段：

```text
confirmed_facts
state_changes
events
roll_results
visible_information
hidden_information
narration_constraints
next_required_action
```

### 12.3 Narration Output

```json
{
  "text": "你踉跄着靠近守卫，手指刚触到钥匙环，他便猛地扣住了你的手腕。",
  "state_claims": [],
  "suggested_actions": []
}
```

叙事层不得在 `state_claims` 中宣称未经引擎确认的关键事实。

## 13. 单 Agent 运行期建议流程

```text
Step 1：读取玩家输入与当前可见上下文
Step 2：生成 ActionPlan
Step 3：若 route == narrative，直接生成回复
Step 4：若 route == engine，调用确定性引擎
Step 5：引擎返回 ConfirmedOutcome
Step 6：基于玩家原话 + ActionPlan + ConfirmedOutcome 生成叙事
Step 7：检查叙事是否违反 confirmed_facts 或 narration_constraints
Step 8：向玩家输出
```

注意：虽然是同一个 Agent，Step 2 和 Step 6 应使用不同提示模板或不同状态节点，避免行动解析与文学表达互相干扰。

---

## 14. 后续拆成双 Agent 的触发条件

不应提前凭感觉拆分，而应基于运行数据。出现以下情况时，应优先考虑拆成两个 Agent。

### 14.1 信息泄露问题

- 多次泄露 NPC secrets；
- 暗骰失败后出现明显暗示；
- 提前暴露尚未触发的事件；
- 无法仅靠上下文裁剪解决。

### 14.2 行动解析问题

- 经常选错流水线；
- Checkpoint 匹配准确率低；
- 复合动作拆解错误；
- 自由叙事和规则行为分类不稳定；
- 需要使用专门的小模型降低成本。

### 14.3 叙事忠实度问题

- 经常推翻引擎失败结果；
- 在语言中创造未写入 GameState 的事实；
- 叙事自由度和确定事实难以同时约束。

### 14.4 评测和模型选型需求

- 希望分别评测解析与叙事；
- 希望前后阶段使用不同模型；
- 希望行动理解低延迟、回复生成高质量。

### 14.5 系统复杂度上升

- 多人同时行动；
- 单句包含多个条件动作；
- 大量重要 NPC 同场互动；
- 隐藏信息分层更加复杂。

---

## 15. 暂不建议独立的运行期 Agent

### 15.1 剧情编排 Agent

MVP 阶段，“主持”和“剧情编排”高度重叠。主持本身就负责：

- 场景描述；
- 节奏控制；
- NPC 反应；
- 暗线推进；
- 信息披露。

过早拆分可能导致“导演想强推剧情、主持想尊重玩家自由”的冲突，还需要额外协调器。

后续只有在以下情况下再考虑独立后台导演 Agent：

- 主持 Agent 长期忘记模组节奏；
- 多条暗线需要并行维护；
- 需要一个不直接对玩家说话的剧情规划器；
- 导演只提供建议，不直接修改状态。

### 15.2 角色行动处理 Agent

MVP 先作为主持编排 Agent 的内部步骤。只有当复杂动作拆解成为主要瓶颈时，再独立为 Action Interpreter。

### 15.3 边界审查 Agent

规则和状态边界应主要由确定性引擎负责，而不是 Agent：

```text
Agent 可以辅助发现风险
引擎必须拥有最终拒绝权
```

---

## 16. 后续扩展：总结复盘 Agent

总结复盘可在 MVP 完成后加入，负责：

- 局内摘要；
- 关键选择回顾；
- 角色表现总结；
- 规则争议解释；
- 模组完成度和遗漏线索分析；
- EventLog 转自然语言战报；
- 为下一局生成可复用经验。

它应主要读取 EventLog 和最终 GameState，而不是依赖对话全文重新猜测发生了什么。

---

## 17. 最终架构决策

### 17.1 MVP

```text
模组导入：
- 模组解析 Agent
- 导入审查 Agent

游戏运行：
- 主持编排 Agent（单 Agent，前后贯穿）
- 确定性规则引擎

赛后复盘：
- 暂不实现
```

### 17.2 长期演进方向

```text
玩家输入
→ 行动理解 Agent
→ 确定性规则引擎
→ 回复编排 Agent
→ 玩家
```

但是否拆分，应根据泄密率、意图解析准确率、叙事忠实度、延迟和成本等真实指标决定。

---

## 18. 一句话总结

> 顶层按生命周期划分模块，Agent 按上下文、权限、输出目标和失败模式划分。MVP 采用“解析 Agent + 审查 Agent + 主持编排 Agent + 确定性引擎”；游戏运行期先用一个主持编排 Agent 跑通闭环，同时通过 ActionPlan 和 ConfirmedOutcome 保留未来拆成双 Agent 的能力。
