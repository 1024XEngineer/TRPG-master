# 数据模型设计

> **本文定位**：定义系统的**逻辑数据模型**——有哪些实体、每个实体有哪些字段、字段的语义与约束。
>
> 与 [[数据库设计导读]] 的分工：
> - 数据库设计文档回答 **「怎么存」**（建什么表、什么索引、JSONB 还是 TEXT）
> - 本文回答 **「存的是什么」**（`state JSONB` 这一列里，究竟装着什么结构）
>
> 两者是**投影关系**，不是对齐关系。逻辑模型里的一等公民（如 `Rule`）在物理模型里可能被折叠进一列 JSONB——这不矛盾，见 §8。
>
> **当前框架输入边界**：运行时接受的模组 JSON 以
> `collaboration_framework/contracts.py` 的 Pydantic `ModuleContent` 为唯一事实源，
> 并以自动生成的 `schemas/module-content.schema.json` 校验。本文的 `ModulePack`、
> `SanTrigger` 等是导入与持久化内部逻辑模型；尚未映射进 `ModuleContent` 的字段不是
> 当前跨组件输入契约。
>
> **验证基础**：本模型的每一项设计都经过六个真实 COC 模组的逐条验证：
> 《追书人》《鬼屋》《幸福蛙蛙村》《银之锁》《复足》《死者的顿足舞》。
> 凡是标注「★」的字段，均为「不加则至少三个模组无法导入」的硬阻塞项。

---

## 一、分层

原设计为三层。本文补充**第四层 Reference**，并为每层补充**可变性契约**与**分片键**——这是后续所有约束的基础。

| 层 | 实体 | 分片键 | 可变性 | 谁能写 |
|---|---|---|---|---|
| **Reference** | `Skill` / `Occupation` / `Weapon` | `world_id` | 只读 | 规则系统录入（一次） |
| **Content** | `World` / `ModulePack` / `Scene` / `Entity` / `Checkpoint` / `SanTrigger` / `WinCondition` / `Pregen` / `Asset` | `module_id` | 只读 | 仅模组导入流程 |
| **GameState** | `Room` / `Player` / `Character` / `Note` | **`room_id`** | 可变 | **仅引擎**（经校验的 op） |
| **EventLog** | `Event` | **`room_id`** | 只增 | 仅引擎 |

### 1.1 为什么 Reference 层必须独立于 Content

「侦查」这个技能，《追书人》和《银之锁》都在用。它不属于任何一个模组，而属于**规则系统**。

原设计将技能列表塞进 `worlds.definition`（一个 JSONB）。这在**读取**上没问题，在**检索**上不行：

```sql
-- 车卡界面：列出所有「调查」类技能，按基础值排序
SELECT * FROM skills WHERE world_id = 'coc-7e' AND category = '调查' ORDER BY base DESC;
```

`definition JSONB` 做不到这件事（做得到，但没有索引、且难以维护）。

> **判据：前端需要按字段检索的东西，必须是列，不能是 JSON 内的键。**
>
> 这与 §8 的「折叠决策」是同一条原则的两面：
> `Rule` 可以折叠进 JSONB，因为没有「查询所有使用 `on_dodge_declare` 的规则」这种需求；
> `Skill` 不能折叠，因为车卡界面每次都要按 `category` 分组。

### 1.2 Reference 与 Content 的边界

同样是「武器」，落在哪一层取决于**它属于规则系统还是属于模组**：

| 例子 | 层 | 理由 |
|---|---|---|
| `.45 左轮`（1D10+2） | **Reference** | COC 7e 标准武器。任何 1920s 调查员都能带一把 |
| 科比特的「浮空匕首」 | **Content** | 《鬼屋》特有。它是一个挂了 `rules` 的 `Entity` |
| 「侦查」技能 | **Reference** | 规则手册内容 |
| 「一小片深灰色羊毛布料」 | **Content** | 《古宅幽影》的线索 |

### 1.3 GameState 的房间隔离

`room_id` 已贯穿 `players` / `characters` / `notes` / `events`。
本文新增的 `rooms.entity_states` / `rooms.timers` 挂在 `rooms` 上，**天然带 `room_id`**。

> **但求值器不会自动获得这个隔离。** 见 §7.9。

**单房间的可变状态量级**（不含 `events`）：

```
rooms.entity_states     ~20 实体 × 3 键      ≈  2 KB
rooms.timers            ~5 个                ≈ 0.5 KB
rooms.rolling_summary   摘要文本              ≈  4 KB
characters × 4          conditions + ledger  ≈ 12 KB
──────────────────────────────────────────────────
                                             ≈ 19 KB
```

**19 KB。一万个并发房间约 200 MB。** `entity_states` / `conditions` / `ledger` 的体积不是问题。

真正需要设计存储策略的是 `events`——它占单房间总量的 **92%**（约 250 KB / 局）。

### 1.4 ★ EventLog 是权威，GameState 是物化视图

> **决策：采用事件溯源（方案 B）。**

```
events               权威真相（append-only）
rooms.entity_states  物化视图（可从 events 重建）
characters.*         物化视图
players.pending_check 物化视图（等待玩家完成的检定请求）
```

**理由**

§7.3 规定「LLM 提议 op → 引擎校验 → 执行并记账」。若 `events` 只是日志，
这条约束就失去审计价值——**你无法回答「`cat.alive` 是在哪一条 event 上变成 `false` 的」。**

**代价**

1. 每个 op 是**两次写**：先 append `events`，再更新物化视图。**必须在同一事务内。**
2. `Event.payload` 必须**足以重建状态**：

```jsonc
// ❌ 不可重建
{"type": "cat_died"}

// ✅ 可重建
{"type": "state_modified",
 "payload": {"path": "cat.alive", "from": true, "to": false}}
```

**收益**

- 断线重连时，若物化视图损坏，可回放重建
- 调试时可追溯任一状态的变更来源
- §7.3 的「写入口唯一」获得可验证的证据链

> **一致性规则**：物化视图与 `events` 冲突时，**以 `events` 为准**。

## 二、Reference 层

> **全局只读，跨模组、跨房间共享。** 分片键 `world_id`。
>
> **当前仅支持 COC 7e**（`world_id = "coc-7e"`）。字段设计不为其它规则系统预留。

### 2.1 Skill

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `SkillId` | 否 | `"spot_hidden"` |
| `world_id` | `WorldId` | 否 | → `World.id` |
| `name` | `string` | 否 | 「侦查」 |
| `category` | `string` | 否 | 「调查」。**前端按此分组** |
| `base` | `int` | 否 | 25 |
| `kind` | `'fixed' \| 'specialized'` | 否 | |
| `parent_id` | `SkillId` | 是 | 专精技能指向类别 |
| `era` | `string` | 是 | 某些技能限定年代 |

**索引**：`(world_id, category)`、`(world_id, name)`

> **`parent_id` 的必要性**：《死者》有「艺术/手艺(小号)」这类专精技能。
> 专精技能的完整 `name` 在运行时拼出（`科学(天文学)`），不能写死为一行。
>
> `kind == 'specialized'` 的行是**类别**（科学、艺术/手艺），
> `parent_id` 指向它的行是**具体专精**。

> **`base` 是 `int`，不是表达式。**
> COC 7e 的技能基础值全部是常数。六版的「灵感 = INT×5」不在本期支持范围内。

### 2.2 Occupation

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `OccupationId` | 否 | `"antiquarian"` |
| `world_id` | `WorldId` | 否 | |
| `name` | `string` | 否 | 「古董商」 |
| `era` | `string` | 是 | `"1920s"` |
| `credit_rating_min` / `max` | `int` | 否 | 信用评级范围 |
| `skill_points_expr` | `Expr` | 否 | `"EDU*4"` |
| `skill_refs` | `SkillId[]` | 否 | 本职技能 |
| `optional_count` | `int` | 否 | 从备选池中再选几个 |
| `optional_pool` | `SkillId[]` | 是 | 备选池 |
| `description` | `text` | 是 | |

> **`skill_points_expr` 是表达式，不是数字。**
> COC 7e 的职业点数公式因职业而异：`EDU*4` / `EDU*2 + STR*2` / `EDU*2 + (DEX*2 或 APP*2)`。
>
> **它复用 §6.2 的 `Expr` 求值器。** 不需要新语法。

### 2.3 Weapon

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `WeaponId` | 否 | `"colt_45"` |
| `world_id` | `WorldId` | 否 | |
| `name` | `string` | 否 | 「.45 左轮」 |
| `category` | `WeaponCategory` | 否 | 供 `attack.type` 读取 |
| `skill_ref` | `SkillId` | 否 | → `Skill` |
| `damage` | `DiceExpr` | 否 | `"1d10+2"` |
| `base_range` | `string` | 是 | 「15 码」 |
| `attacks_per_round` | `string` | 是 | `"1"` / `"1(3)"` |
| `ammo` | `int` | 是 | |
| `malfunction` | `int` | 是 | 故障值 |
| `era` | `string` | 是 | |

```
WeaponCategory = 'firearm' | 'melee' | 'thrown' | 'explosive' | 'unarmed'
```

> **★ 武器不适用 §6.4 的来源约定。**
>
> `Character.weapons` 引用 `WeaponId`，**不是字符串**。
>
> 理由：战斗流水线要读 `attack.type`（来自 `Weapon.category`）：
>
> ```jsonc
> {"hook":"on_damage_roll", "when":"damage.type == 'firearm'", "then":{"set":1}}
> ```
>
> 字符串「一把左轮」没有 `type`。
> 按 §5.8.7 的 Q4，武器是 **D 类**（被 `Rule.when` 引用），不是门。

**为什么这与「不建武器表」的旧判断不冲突**

旧判断（「武器只在 `desc` 里写清伤害骰，`impale`/`ammo`/`malfunction` 全砍」）是针对 **Content 层**的。

```
Content 层的武器    科比特的「浮空匕首」  → Entity，挂 rules，无需 ammo
Reference 层的武器  「.45 左轮」          → 前端要检索、要装备，需完整字段
```

**判据：这把武器属于模组，还是属于规则系统？**

## 三、Content 层

### 3.1 World

一整套规则系统（如 COC 7e）。行数极少，几乎不变。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `WorldId` | 否 | 如 `"coc-7e"` |
| `name` | `string` | 否 | 「克苏鲁的呼唤 7版」 |
| `definition` | `RuleSystemDef` | 否 | 属性范围、技能表、检定原语、成功等级、默认 Checkpoint |
| ★ `hooks` | `HookDef[]` | 否 | 引擎流水线上的可挂载点清单。见 §5.1 |
| ★ `variables` | `VariableDef[]` | 否 | 规则可读的内建变量表。见 §5.2 |
| ★ `world_rules` | `Rule[]` | 否 | 本规则系统自身的规则，写成三元组。见 §5.3 |

**为什么 `definition` 保留为单字段、而 `hooks`/`variables`/`world_rules` 要独立**

`definition` 装的是**声明式常量**（d100、roll_under、成功等级六档、默认 Checkpoint）——读一次，之后按它算。
引擎会读取其中的已知键，但前端和数据库没有按这些键跨 World 检索的需求，因此物理层仍保留为一个 JSONB 字段。

`hooks`/`variables`/`world_rules` 装的是**可执行单元**。引擎按 `hook` 分派、按 `variables` 校验、按 `world_rules` 求值。它们必须有结构。

**为什么 `world_rules` 与 `Entity.rules` 必须同构**

引擎在每个 hook 上会同时收集 world 规则与 entity 规则，排序后统一执行。若两者结构不同，就必须写一个合并器——**合并器会成为整个系统最脏的地方**。

### 3.2 ModulePack

一个具体剧本。

`ModulePack` 是内部逻辑实体，不等同于运行时输入对象 `ModuleContent`。导入流程负责
将这里的逻辑实体投影为当前 Pydantic 契约；在契约扩展前，不得把本节字段直接塞入
`ModuleContent` JSON。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `ModuleId` | 否 | `builtin:core:shadow-over-arkham:1.0.0` |
| `title` | `string` | 否 | |
| `version` | `SemVer` | 否 | |
| `world_ref` | `WorldId` | 否 | → `World.id` |
| `authors` | `string` | 是 | |
| `players_min` / `players_max` | `int` | 否 | 见下方注 |
| `difficulty` | `string` | 是 | |
| `source` | `'builtin' \| 'imported'` | 否 | |
| `owner_user_id` | `UserId` | 是 | |

> **`players_min/max` 的用途被扩展了。** 它不只是「建议人数」——《复足》的模组内容是队伍规模的函数：
> 「梦境之石总数约为调查员总数的一半」「冷蛛数量 = 未携带石头的调查员数量」。
> 这两个字段必须能被 spawn 表达式引用（`party.size`，见 §5.2）。

### 3.3 Scene

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `SceneId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| `title` | `string` | 否 | |
| `description` | `text` | 否 | 场景描述。**自由文本，LLM 可演绎** |
| `map_ref` | `AssetRef` | 是 | |
| `exits` | `SceneId[]` | **是** | 相邻场景 |
| `checkpoint_ids` | `CheckpointId[]` | 否 | 当前场景可路由的模组 Checkpoint。默认 `[]` |

> **`exits` 必须可空。** 六个模组里只有《银之锁》（房间→长廊→门）和《复足》（10 层楼真实拓扑）有邻接约束。
> 《追书人》《鬼屋》《蛙蛙村》《死者》都是城市或园区内自由移动，`exits` 恒为空。

### 3.4 ★ Entity

**本次改动中最核心的一张表。** 吸收原 `module_clues` 与 `module_npcs`。

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `EntityId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| `kind` | `EntityKind` | 否 | `npc \| monster \| item \| clue \| animal \| object` |
| `name` | `string` | 否 | |
| `content` | `text` | 是 | 线索内容 / 物品描述。**自由文本** |
| `public_persona` | `text` | 是 | 🟢 玩家可见的表面人设。**自由文本** |
| `secrets` | `text` | 是 | 🔴 真实底牌。**自由文本，受隔离约束（§7.1）** |
| `stats` | `StatBlock` | **是** | 属性块 |
| `locked` | `bool` | 是 | 门的状态。LLM 可提议改写 |
| ★ `refuse_ops` | `OpName[]` | 否 | **A 类**：引擎必须拒绝的 op。默认 `[]`。见 §5.8.3 |
| ★ `state` | `Record<string, Primitive>` | 否 | **D 类**初始值。见 §5.5 / §5.8.6 |
| ★ `rules` | `Rule[]` | 否 | **B/C 类**规则。见 §5.3 / §5.8 |

**为什么合并 `clues` 与 `npcs`**

原设计将 `module_clues` 定位为「最烧脑但也最关键的表」，其隐含假设是**模组的核心是线索网络**。

该假设仅对调查向模组成立。六个模组的核心结构：

| 模组 | 核心结构 |
|---|---|
| 追书人 | 线索网络 ✓ |
| 死者的顿足舞 | 线索网络 ✓ |
| 银之锁 | **因果链**（猫的生死） |
| 复足 | **携带物开关**（梦境之石） |
| 幸福蛙蛙村 | **说服** |
| 鬼屋 | **战斗 + 孤注一骰** |

若 schema 围绕线索网络生长，后四个模组会被挤成畸形。

而猫、梦境之石、小号、羊毛布料——它们是同一类东西：**有状态的、可被表达式引用的实体**。线索只是其中一种 `kind`。

**为什么 `stats` 必须可空**

《追书人》的邻居与看守、《鬼屋》的阿蒂/佐利/罗斯/金、《银之锁》的芭斯特，全都没有属性。
《蛙蛙村》原文直接写：「其他游客守密人随便编就行」。

字段若不可空，解析器会**逼迫 LLM 编造数字**。

**为什么 `rules` 挂在 Entity 而非 Module**

「僵尸不会闪避」是僵尸这个实体的规则，不是《死者》这个模组的规则。
同一模组内，人类形态的特纳与僵尸形态的特纳，规则不同。

### 3.5 Checkpoint

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `CheckpointId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| `match_hint` | `text` | 否 | Intent 路由用的简短匹配提示 |
| `priority` | `int` | 否 | 当前场景有多个候选时的排序依据 |
| `skill` | `SkillRef` | 否 | 见下方注 |
| `difficulty` | `Difficulty` | **是** | 见下方注 |
| `on_success` / `on_fail` | `Outcome` | 否 | |
| `hidden` | `bool` | 否 | 暗骰 |
| `roll_mode` | `RollMode` | 否 | `'auto'` 引擎静默掷；`'prompt'` 要求客户端展示掷骰 UI |

```
SkillRef   = SkillName | "@" Category     // "侦查" 或 "@交涉"
Difficulty = 'regular' | 'hard' | 'extreme'
```

> **`skill` 必须允许类别值 `@交涉`。**
> 《鬼屋》：友好→魅惑 / 气势汹汹→恐吓 / 据理力争→说服 / 忽悠→话术。
> 这不是四选一，是**行为→技能的映射规则**——玩家会干出第五种。写死 `skill` 反而是 bug（作者故意留的开口）。
> 候选集同时是 **UI 契约**：模组路由使用
> `expand(checkpoint.skill) ∩ character.skills`；默认路由使用
> `normalize(intent.proposed_skills) ∩ character.skills`。结果由引擎固化，客户端不能扩充。

> **`difficulty` 必须可空。**
> 《蛙蛙村》：说服信使——不阐述=困难 / 阐述合理=普通 / 阐述精彩=普通+奖励骰。
> 难度由玩家自然语言的质量决定，**运行时才知道**。
> 为空表示：由 LLM 在上游求值，输出枚举写入状态槽（见 §7.4）。

Intent 分别返回 `ExecutionRoute = narrative | engine` 与
`CheckRoute = none | module | default`。前者决定是否调用确定性引擎，后者只决定是否检定
以及检定来源；`module` 加载模组节点，`default` 加载
`World.definition.default_checkpoint`，不在运行时生成 Content。
完整路由、掷骰与 Narrator 边界见 [[checkpoint流程设计文档]]。

`roll_mode='prompt'` 创建 `pending_check`；`roll_mode='auto'` 静默结算。
硬约束：`hidden = true ⇒ roll_mode = 'auto'`。

### 3.6 ★ SanTrigger

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `SanTriggerId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| ★ `kind` | `SanKind` | 否 | SAN 的六种形态 |
| ★ `source_tag` | `string` | 是 | 累计封顶的分组键 |
| `loss` | `SanExpr` | 是 | `"0/1d6"` 等 |
| `condition` | `text` | 是 | 触发条件（自由文本） |

```
SanKind =
  | 'check'         // ① 检定式        0/1d6
  | 'flat'          // ② 无检定固定损失  读日记 −1d4
  | 'direct'        // ③ 精神伤害      对抗胜 → 直接扣 1d6，不走 SAN 检定
  | 'max_reduce'    // ④ 上限扣减      +2% 神话 → SAN_max 99→97
  | 'gain'          // ⑤ 回复          +1d6 / +2d6
  | 'capped'        // ⑥ 累计封顶      同源合计 ≤ 6
```

**六种形态的出处**

| 形态 | 模组 | 原文 |
|---|---|---|
| ① `check` | 全部 | 「理智损失 0/1D6」 |
| ② `flat` | 追书人、鬼屋 | 「与道格拉斯交谈后损失 1d4 理智」 |
| ③ `direct` | 蛙蛙村 | 信使精神冲击：POW 对抗胜 → 直接扣 1d6 |
| ④ `max_reduce` | 鬼屋 | 读《伊波恩之书》：+2% 神话，SAN 上限 99→97 |
| ⑤ `gain` | 四个模组 | 「获得 1D6 的理智回复」 |
| ⑥ `capped` | 追书人、死者 | 「两次理智损失之和最大为 6 点」 |

> **③ 尤其重要。** LLM 会习惯性地为它插入一次 SAN 检定——因为绝大多数 SAN 损失都要检定。
> **必须用字段名切断这个惯性。**

> **⑥ 注意语义：是累计封顶，不是去重。**
> 《追书人》一场内触发三次同源 SAN，要求**累加后与 6 比较**。
> LLM 会当成「见过就跳过」——语义完全不同。
>
> 《追书人》与《死者》独立出现同一规则（COC 7e 的「习惯恐惧」），
> 说明该规则属于 `World.world_rules`，但 `source_tag` 由模组提供。

### 3.7 ★ WinCondition

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `id` | `WinConditionId` | 否 | |
| `module_id` | `ModuleId` | 否 | |
| ★ `expr` | `Expr` | 否 | 引用 `entity_states` 的布尔表达式。见 §5.2 |
| ★ `is_ending` | `bool` | 否 | `false` 表示状态回滚，非终局 |
| `text` | `text` | 否 | 结局文本 |

**为什么不能是「找齐哪些核心线索」**

| 模组 | 结局条件 |
|---|---|
| 复足 | `has_dreamstone && location == 'outside'`（未携带 × 离开 = 死） |
| 银之锁 | 救猫 → 猫活 → 走廊倒地 → 猫抓瞎 NPC → NPC 杀猫 → 锁解除 |
| 死者 | 阻止特纳**而不杀他**（杀了他会复活，且不再需要呼吸，更难杀） |
| 蛙蛙村 | 闪避**成功** = 失去永久幸运 +2d6 |

结局条件是**表达式**，地点与线索只是表达式的项。

**为什么需要 `is_ending`**

《银之锁》「没救猫 → 被抓回房间 → 重来」是**状态回滚**，不是终局。
《复足》「留守 = 消耗战」同理。

不加这个字段，LLM 会把它演成 bad ending 并结束游戏。

### 4.3 Player / Note / User

`Player` 沿用原有身份与连接字段，并新增一个**单槽、仅引擎可写**的交互状态：

| 字段 | 类型 | 可空 | 语义 |
|---|---|---|---|
| `pending_check` | `PendingCheck` | 是 | 唯一挂起态；所有 `prompt` 检定都必须先创建它 |

## 五、嵌套结构定义

> 本节展开所有「**有代码读取其内部键**」的结构。
>
> 判据：`public_persona` / `secrets` / `description` 等只被 LLM 读取的字段，
> 在物理层是 TEXT，在逻辑层就是「自由文本」，**不在此展开**。

### 5.1 HookDef

```
HookDef {
  name:      HookName
  signature: { accepts: VarName[], returns: 'void' | 'bool' | VarName }
  position:  int          // 在流水线中的次序，用于文档与校验
}
```

**COC 7e 的三张流水线，共 19 个 hook**

战斗不是一次 `damage → hp` 的映射，而是一条流水线。每一步都是一个可被外部规则拦截或改写的暂停点。

同样地，检定与场景进入也各是一条流水线。三张表的 `Rule` 结构完全相同，只是挂载点不同。

**① 战斗流水线（12 个）**

```
声明攻击              on_attack_declare
决定用哪个技能         ─┐
计算难度              on_difficulty_calc     [死者: 人数→难度]
掷攻击骰              on_attack_roll
比较成功等级           ─┘                    [鬼屋: 六层分支]
防守方能否闪避         on_dodge_declare       [死者: 僵尸不能]
掷闪避骰              on_dodge_roll
判定命中              on_hit_resolve
掷伤害骰              on_damage_roll         [死者: 枪弹 vs 刀]
护甲扣减              on_armor_apply         [鬼屋: 血肉防护]
写入 HP               on_hp_write
重伤判定              on_major_wound         [死者: 无视]
死亡                  on_death
回合结束              on_turn_end            [蛙蛙村: 拖拽/吞食]
```

**② 检定流水线（4 个）**

```
声明检定              on_check_declare
计算难度              on_difficulty_calc     （与战斗共用）
掷骰                  on_check_roll
求值成功等级 → 分派    on_check_resolve       ★ 见 §5.8「C 反转」
```

`on_check_roll` 只有一个取值来源：**CheckResolver 在检定实际执行时现掷并生成服务端权威 `raw`**。
`roll_mode` 只决定流水线是否在掷骰前暂停：

| `roll_mode` | 暂停点 | 恢复方式 |
|---|---|---|
| `prompt` | 引擎完成候选校验与难度计算后，创建 `pending_check` | 玩家提交 `request_id + skill_id`，校验后恢复到 `on_check_roll` |
| `auto` | 不暂停、不创建 `pending_check` | 引擎直接进入 `on_check_roll` |

两者都严格执行 `on_check_declare → on_difficulty_calc → on_check_roll → on_check_resolve`。
客户端不能提交 `raw` 或 `tier`，Narrator 也不能改写 ActionResult；否则《鬼屋》与《蛙蛙村》的 C 类规则会被绕过。
交互时机的统一不增加或删除 hook，因此 19 个 hook 的边界不变。

**③ 场景流水线（3 个）**

```
进入场景              on_scene_enter         ★ 见 §5.8「B 必然」
离开场景              on_scene_exit
交互                  on_interact
```

> **为什么 `on_scene_enter` 必须存在**
>
> 《银之锁》：玩家走进走廊，猫必须跳出去抓瞎绑匪，随后被杀。
> 这件事**不由玩家发起**——玩家只是说了「我往前走」。
>
> LLM 没有任何理由在这一刻回忆起第 8 轮救过的那只猫。它正在描述走廊。
>
> 若只有战斗流水线，这个机制无处挂载。

**Hook 不是「事件」**

| | 语义 | 能否改写 |
|---|---|---|
| 事件 | 「已经发生了，通知你一声」 | ❌ |
| Hook | 「即将发生，引擎在等你的返回值」 | ✅ |

```python
# 事件
target.HP -= damage;  emit("damaged")      # 已经扣了

# Hook
damage = fire("on_damage_roll", damage)    # 还没扣，规则说扣多少
target.HP -= damage
```

若只实现事件，《死者》的「僵尸被枪弹命中只损 1 点耐久」无法实现——伤害已经扣完了。

`signature` 决定了每个 hook 的干预能力：
- `on_damage_roll` 接收 `damage`、返回 `damage` → 能**改写**
- `on_dodge_declare` 返回 `bool` → 能**拦截**

---

### 5.2 VariableDef

规则的 `when` 表达式只能引用此表内的变量。**这份列表就是引擎能力的边界。**

```
combat.round
self.HP / self.HP_max / self.flesh_ward
self.rounds_without_damage
self.damage_taken_this_round
attack.type / attack.name
damage.type / damage.value
target.form
party.size / party.max_STR
check.id / check.tier / check.roleplay_tier
```

> **计数器类变量必须由引擎自动维护。**
> `self.rounds_without_damage` 不能指望规则自己累加——规则只能读它。

> **导入校验**：规则引用表外的键（如 `self.mood`）→ 校验失败，打回重新生成。

`check.raw` 与 `check.initiator` 暂不加入 `VariableDef`。六个验证模组没有规则需要按原始骰值或发起方分支；
它们是 CheckResolver / Event 的审计数据，不等于规则表达式的可读变量。未来只有在真实规则需要时，才分别增加
`check.raw` 或 `check.initiator ∈ {'player','engine'}`，守住「变量表就是引擎能力边界」。

### 5.3 ★ Rule

```
Rule {
  hook:     HookName
  when:     Expr                                    // 布尔表达式，见 §5.2
  then:     Op                                      // 见 §5.4
  mode:     'append' | 'override' | 'forbid'
  priority: int                                     // 默认 module 低于 world
}
```

**`mode` 的语义**

| mode | 含义 | 例子 |
|---|---|---|
| `append` | 我加一条，前面的照常执行 | 鬼屋·血肉防护（叠加在护甲之后） |
| `override` | 我执行，同 hook 上 `source == 'world'` 的规则跳过 | 复足·固定阈值检定 |
| `forbid` | 这个 hook 整个不执行 | 死者·僵尸无视重伤 |

**执行流程**

```
引擎在每个 hook 上：
  1. 收集 world_rules + module rules + entity rules
  2. 按 priority 排序
  3. 逐条执行
       遇 forbid   → 跳过整个 hook
       遇 override → 跳过所有 source == 'world' 的规则
```

> **不设 `invert` 模式。**
> 直觉上《死者》「INT 检定成功 → 更糟」像是「把输出反过来」，但实际不是：
>
> ```
> world:  INT 成功 → 保持清醒 ／ INT 失败 → 临时疯狂
> 死者:   INT 成功 → 完全理解 → 1D10 小时疯狂 ／ INT 失败 → 保持清醒
> ```
>
> 反转的是**语义**，不是布尔值。这是 `override`（替换后果），不是反转。
> `invert` 看似优雅，但会诱使建模者去表达「反过来」——而真实模组从不真的反转。

**为什么不能用「一堆具名布尔字段」**

《死者》的僵尸有四条规则：枪弹只损 1 耐久 / 刀伤减半 / 无视重伤 / 不会闪避。

写成 `ignores_major_wound: true` 这类字段，等于**用 JSON 写 if-else**。第八个模组来了再加一个。

更隐蔽的问题是**作用顺序表达不出来**：鬼屋的「血肉防护」与追书人的「火器伤害减半」若同时存在，先减半还是先扣甲？结果差一倍。
用字段的话，这个顺序藏在引擎代码里，模组作者无法覆盖。用 `(hook, priority)` 就显式了。

### 5.4 Op（算子表）

```
Op =
  | { set: Value }
  | { scale: number, round?: 'floor' | 'ceil' }
  | { add: Value }
  | { absorb: Expr, decrement: VarName }
  | { forbid: true }
  | { force: ActionName }
  | { apply_condition: ConditionId }
  | { spawn: { count: Expr, type: EntityId } }
  | { trigger: EventName, expires_in_rounds?: int }
  | { oppose: { a: Expr, b: Expr }, on_win: Op, on_lose: Op }
```

**六模组实证：算子集合已收敛**

```jsonc
// 死者·僵尸
[{"hook":"on_dodge_declare","when":"always","mode":"forbid"},
 {"hook":"on_damage_roll","when":"damage.type=='firearm'","then":{"set":1}},
 {"hook":"on_damage_roll","when":"damage.type!='firearm'",
  "then":{"scale":0.5,"round":"floor"}},
 {"hook":"on_major_wound","when":"always","mode":"forbid"}]

// 鬼屋·血肉防护（有状态、会耗尽、有时限）
[{"hook":"on_armor_apply","when":"self.flesh_ward > 0","priority":10,
  "then":{"absorb":"min(damage, self.flesh_ward)","decrement":"self.flesh_ward"}}]

// 复足·冷蛛转换目标（「4 轮内未造成伤害则换目标」）
[{"hook":"on_turn_end","when":"self.rounds_without_damage >= 4",
  "then":{"force":"switch_target"}}]

// 蛙蛙村·长舌拖拽（跨回合 + 后续对抗）
[{"hook":"on_hit_resolve","when":"attack.name=='长舌鞭笞'",
  "then":{"trigger":"pending_drag","expires_in_rounds":1}},
 {"hook":"on_turn_end","when":"self.pending_drag",
  "then":{"oppose":{"a":"self.STR","b":"target.STR"},
          "on_win":{"trigger":"swallow"},"on_lose":{"clear":"pending_drag"}}}]
```

> 六个模组跑完，**没有出现第 13 个 hook，也没有出现新算子**。
> 这是「现在就定死 hook 表」的依据——再等下去不会得到更多信息，只会积累更多需要迁移的数据。

### 5.5 ★ EntityState

```
state: Record<string, Primitive>
Primitive = boolean | number | string
```

**键名由模组导入时生成，不预先枚举。**

六模组实测必须存在的键：

| 键 | 模组 | 用途 |
|---|---|---|
| `cat.alive` | 银之锁 | 被 `WinCondition.expr` 引用 |
| `pc.has_dreamstone` | 复足 | 被 `WinCondition.expr` 引用（携带 = 生） |
| `turner.form` | 死者 | 被 `Rule.when` 引用（人类/僵尸两套规则） |
| `turner.death_count` | 死者 | 马努斯科被杀三次——`alive` 不能是单向 bool |

**判据：哪些状态该落库，哪些走提示词**

> **这个键有没有出现在任何 `Rule.when` 或 `WinCondition.expr` 里？**
> - 有 → 落 `state`
> - 没有 → 走 `public_persona` 自由文本，**不落库**

按此判据，以下状态**不应**落库：

```
npc_doctor.attitude      只影响 LLM 怎么演
cat.trusts_player        只影响 LLM 怎么演
```

原因：引擎读不了上下文，只能读变量。而 LLM 读不了变量之外的东西吗？读得了——**上下文它读得更好**。
态度是连续量，枚举成 `friendly | hostile` 反而失真。

> 该判据可实现为一个**符号表检查**，在导入流程中自动运行。

### 5.6 ★ Condition

```
Condition {
  id:            ConditionId
  timer?:        { after: Duration } | { every: Duration } | { at: Timestamp }
  check?:        SkillRef | Expr        // 复足: "CON * (6 - stage)"
  on_fail?:      Effect
  on_success?:   Effect
  stages?:       Stage[]                // 多阶段状态机
  reversible_until?: int
  repeat_until?: Expr                   // "healed || dead"
  desc:          text                   // ← 自由文本，不参与计算
}

Stage {
  n:      int
  check?: Expr                          // 难度可以是阶段的函数
  delta?: Record<AttrName, DiceExpr>    // 增量
  set?:   Record<AttrName, int>         // 赋值
  desc:   text
}

Effect {
  duration?:  DiceExpr
  permanent?: Record<AttrName, DiceExpr>
  apply?:     ConditionId
}
```

**为什么不能是字符串数组**

| 模组 | 机制 |
|---|---|
| 鬼屋 | 科比特爪伤：命中 → 幸运检定 → **一天后** → CON 检定 → 分支 → **循环直到痊愈或死亡** |
| 复足 | 感染阶段 0–5，难度 = 阶段的函数（0 幸运 / 1 体质×5 / 2 体质×4 / … / 5 体质×1） |
| 蛙蛙村 | 变蛙四阶段。阶段 2 可逆、阶段 3 起不可逆 |

**这不是记忆问题，是调度问题。**

「一天后」需要有人在一天后把它叫醒。LLM 是被动的，它只在玩家说话时被调用。
玩家在游戏内第二天早上说「我去餐厅」——**没有任何东西会提醒 LLM 该插入一次 CON 检定**。

它正在回答「餐厅长什么样」，不会自己想起「你昨天被抓伤了」。

**`delta` 与 `set` 必须分开建键**

《蛙蛙村》阶段 2 是 `跳跃 +4d6 / APP −2d6`（增量），阶段 3 是 `APP = 25 / 跳跃 = 85`（赋值）。

**LLM 会把「APP 变成 25」误算成「APP −25」。** 这是纯语义歧义，只能靠字段名消除。

**`desc` 是唯一的自由文本子字段**

引擎负责「何时」与「多少」，LLM 负责「像什么」：

```jsonc
{
  "id": "corbitt_infection",
  "timer": {"after": "24h"},
  "check": "CON",
  "on_fail":    {"duration": "1d10d", "permanent": {"CON": "-1d10"}},
  "on_success": {"duration": "1d6d"},
  "repeat_until": "healed || dead",
  "desc": "伤口周围皮肤泛灰白，触碰时有异物感"   // ← 不参与任何计算
}
```

七个键。**再简化就只能删掉 `timer`——然后科比特的爪伤永远不会发作。**

### 5.7 ★ LedgerEntry

```
LedgerEntry =
  | { count: int, window_h: int, then: Effect }       // 计数器 + 时间窗
  | { active: bool, expires_in_h: int, effect: Op }   // 单次事件 + 时长
  | { accumulated: int, cap: int }                    // 累计封顶
  | { regen: DiceExpr, per_h: int }                   // 周期回复
```

| 实例 | 模组 | 原文 |
|---|---|---|
| 计数器 + 时间窗 | 蛙蛙村 | 「24 小时内失败两次或以上意志检定 → 此后类似检定承受惩罚骰」 |
| 单次事件 + 时长 | 蛙蛙村 | 「陷入强烈幻觉 → 接下来几小时内再次接触池水承受惩罚骰」 |
| 累计封顶 | 追书人 | 「两次理智损失之和最大为 6 点」 |
| 周期回复 | 鬼屋 | 「MP 每小时回复 1 点」（作者原文特别提醒 KP「小心记录」） |

> LLM 数不清「24 小时内两次」——它需要回溯整个上下文数掷骰次数，并判断是否落在窗口内。长对话必错。
>
> 而《鬼屋》作者亲自提醒 KP「小心记录 MP」——**连人类都容易忘**。这是纯记账。

### 5.8 ★ 四类引擎介入（A / B / C / D）

> 本节是整份文档的**判断中枢**。
> 它回答一个问题：**什么东西该纯文本，什么东西必须由引擎接管。**

#### 5.8.1 两个正交的维度

模组里的每一样东西，落在这张表的某一格：

| | **LLM 会偏离** | **LLM 不会偏离** |
|---|---|---|
| **会卡关** | ∅ *(空)* | 钢钳、囊肿 → **提示系统** |
| **不会卡关** | A / B / C / D → **引擎介入** | 柜子、钥匙、棍子 → **纯文本** |

**左上角是空的，这不是巧合。**

会卡关的东西，LLM 反而不会偏离——因为它想帮玩家找到钢钳，不会隐瞒。

而 LLM 会偏离的地方，恰好都不卡关——**因为它偏离的方向就是「让玩家继续前进」。**

> **推论：提示系统与防偏离系统服务于完全不相交的两批机制。**
> 它们不需要联动，也不该共用字段。

#### 5.8.2 右下角：门

**门 = 不会卡关 + LLM 不会偏离。**

第二项成立，是因为 LLM 的偏离方向（让玩家过去）与作者的意图重合。

**在这个方向上，LLM 的偏向是免费的助力。**

```jsonc
// 门：纯文本。没有 rules，没有 refuse_ops，不进 entity_states
{
  "kind": "object",
  "name": "书房的柜子",
  "secrets": "上锁。需要从画框后取出的那把钥匙。",
  "locked": true
}
```

玩家用铁撬撬、用砖砸、请 NPC 帮忙——**LLM 全权判断，怎么放水都行。**
打开它的后果只是「玩家拿到了里面的东西」。

> **`locked` 甚至不是必需的**（信息已在 `secrets` 里）。
> 保留它只是为了让引擎知道「这里有一个状态可以被改」。

**但门有两种。判据只有一条：**

> **这扇门开了之后，有没有别的规则要读这个结果？**
>
> - **没有 → 门 I**：纯文本，LLM 全权。（柜子、抽屉、上锁的房门）
> - **有 → 门 II**：**门本身仍是纯文本，但门后的「值」必须落库。**（梦境之石）

门 II 的正确写法见 §5.8.6（D 类）。

#### 5.8.3 A 类：拒绝

> **玩家发起一个 op → 引擎必须拒绝。**

**特征**：由玩家的动作触发。玩家不动，什么都不会发生。

```jsonc
{
  "kind": "item",
  "name": "特纳的小号",
  "secrets": "特纳不会把它卖或借给任何人，无论怎么劝说或威胁都不会改变心意。",
  "refuse_ops": ["transfer", "give", "sell"]
}
```

LLM **仍然读得到 `secrets`**，仍然能演绎「特纳把小号抱得更紧了」。
它只是不能提议 `modify(trumpet.holder, "pc_001")`——引擎会拒。

**为什么 LLM 不行**：它会被说服。玩家的理由说得很好，而它想让对话舒服地继续。

**为什么这不是卡关**：玩家永远拿不到小号，故事照样走完。小号不是通关必需。
**A 是边界，不是推动力。**

#### 5.8.4 B 类：必然

> **某个条件满足 → 引擎必须主动触发。**

**特征**：**不需要玩家发起任何 op。它自己会发生。**

```jsonc
// 银之锁：玩家走进走廊，猫必须跳出去
{"hook":"on_scene_enter",
 "when":"scene == 'corridor' && cat.alive",
 "then":{"trigger":"cat_sacrifice"}}
```

**为什么 LLM 不行**：它不会**主动想起来**。

玩家在第 41 轮说「我往前走」，LLM 正在描述走廊。第 8 轮救下的那只猫，没有任何东西提醒它。

**这不是记忆问题，是调度问题**——与 §5.6 `Condition` 的定时器同源。

**为什么这不是卡关**：**它是推动剧情的力量，不是阻碍。**

> **B 类是「无论如何，这件事会发生」。**
> 银之锁的核心悲剧（猫必须死，且只能死于绑匪之手）完全由 B 类承载。
>
> 若无 B 类，LLM 会在玩家拼命保护猫时写下「你紧紧抱着它，它的心跳贴着你的胸口」，
> 然后整个模组的结局条件永远无法满足。

#### 5.8.5 C 类：反转

> **引擎求值出 `success` → 在成功分支上挂坏结果。**

**特征**：由**引擎自己的求值结果**触发，与玩家意图无关。

```jsonc
// 鬼屋：INT 检定成功 → 完全理解真相 → 临时疯狂
{"hook":"on_check_resolve",
 "when":"check.id == 'corbitt_int' && check.tier == 'success'",
 "then":{"apply_condition":"temp_insanity", "duration":"1d10h"}}

// 蛙蛙村：闪避成功 = 失去永久幸运 +2d6
{"hook":"on_dodge_roll",
 "when":"check.id == 'messenger_kiss' && check.tier != 'fail'",
 "then":{"nothing":true}}
```

**为什么 LLM 不行**：**它的成败先验完全相反。**

LLM 必然把 `success` 演绎成好结果。给它一条「成功反而更糟」的规则，
它会在生成时漏掉，或在执行时把成败两侧读反。

**为什么这不是卡关**：玩家过了检定，游戏继续——只是变糟了。
**C 是代价，不是阻碍。**「你可以这么做，但你会后悔。」

#### 5.8.6 D 类：值

> **它出现在任何 `Rule.when` 或 `WinCondition.expr` 里。**

**D 不是「机制」，是「数据」。** 它没有触发时机，它只是躺在那里被读。

```jsonc
// 门：纯文本，LLM 全权判断玩家怎么拿到石头
{
  "kind": "item",
  "name": "梦境之石",
  "secrets": "半透明的石头，触手温热。带着它的人不会被冷蛛看见。"
}

// 门后的值：落库，引擎保护
"entity_states": {
  "pc_001.has_dreamstone": true
}

// 唯一写入口
{"op": "modify", "path": "pc_001.has_dreamstone", "set": true}
```

**LLM 可以随便演绎玩家怎么拿到石头**——从桌上捡起、从同伴手里接过、在废墟里翻出。

它只能通过一个 op 改那个 bool。而引擎会检查 D 类的**写入不变式**：

```
石头总数 = floor(party.size / 2)
has_dreamstone == true 的人数 不得超过石头总数
```

**《复足》「手拉手无效」的实现，不是禁止 LLM 演绎，而是这个 op 被拒。**

LLM 仍然可以写：「你抓着他的手冲进夜色。」
然后引擎在结局求值时读到 `pc_002.has_dreamstone == false`。**他死了。**
而 LLM 从头到尾没被禁止说任何话。

**为什么 LLM 不行**：**它是唯一能改这个值的人，而它有动机改错。**

这与 A/B/C 都不同——不是「会被说服」「不会想起」「先验相反」，
而是**利益冲突**。详见 §7.3。

#### 5.8.7 判断流程

面对模组里的任何一样东西，依次问：

```
Q1  玩家发起某个 op 时，引擎必须拒绝吗？
      是 → A：Entity.refuse_ops

Q2  某个条件满足时，引擎必须主动触发吗？
      是 → B：Rule on on_scene_enter / on_turn_end

Q3  引擎求值出 success 时，要挂坏结果吗？
      是 → C：Rule on on_check_resolve

Q4  它出现在任何 Rule.when 或 WinCondition.expr 里吗？
      是 → D：entity_states 的一个键，写入口唯一

四个都否 → 纯文本。
```

> **六个模组跑一遍，绝大多数东西四个都否。**
> 柜子、抽屉、钥匙、棍子、煤油灯、日记、羊毛布料、大部分 NPC——**全是纯文本。**

#### 5.8.8 四类的真正差别

| | 需要什么 | 为什么 LLM 不行 | 作者意图 |
|---|---|---|---|
| **A 拒绝** | 一个 op 黑名单 | 它会被说服 | 「你不能拿到这个」——设定边界 |
| **B 必然** | 挂在场景 hook 上的规则 | 它不会主动想起 | 「无论如何这件事会发生」——推动剧情 |
| **C 反转** | 挂在检定 hook 上的规则 | 它的成败先验相反 | 「你可以这么做，但你会后悔」——设置代价 |
| **D 值** | 受保护的变量 + 写入不变式 | 它是唯一写入者，且有动机写错 | ——（不是机制，是数据） |
| **门** | **什么都不需要** | *(它的偏向与作者意图一致)* | 「你该过去，方法随意」 |

> **A/B/C/D 的共同点：LLM 的偏向与作者的意图相反。**
> **门的特殊性：LLM 的偏向与作者的意图一致。**

#### 5.8.9 交叉：一个实体可以同时属于多类

```
猫       B（走廊必须触发牺牲）+ D（cat.alive 被 WinCondition.expr 读）
梦境之石  D（且有写入不变式）。不是 A——引擎不需要 refuse_ops，
          只需要「持有者人数 ≤ 石头总数」这条不变式
特纳的小号 仅 A。trumpet.holder 不出现在任何表达式里，
          拿到不通关，拿不到不卡关
```

> **注意「手拉手无效」的归类。**
> 直觉上它像 A（拒绝一个「分享」op），但更准确的是 **D 的写入校验**。
>
> 引擎不需要 `refuse_ops: ["share"]` 这个字段。
> 它只需要知道：`has_dreamstone == true` 的人数不能超过石头总数。

#### 5.8.10 三个反直觉的结论

**① `impassable: bool` 这样的字段不该存在。**

「作者希望你撞墙」这个描述同时套住了 A、B、C——**而它们需要三种完全不同的机制**：
A 是拒绝一个 op，B 是主动触发，C 是接管分支。

一个 bit 装不下三种东西。

**② 提示池不需要排除 A/B/C/D。**

因为它们**压根不会卡关**，提示系统的触发条件（长时间无新 flag）永远不会碰到它们。

小号、猫、INT 检定——玩家该干嘛干嘛，flag 照常涨。

> **`hintable` / `hint_priority` 字段因此不必存在。**
> 提示池 = 当前场景中所有未被发现的、纯文本的实体。

**③ 「保留其他通过方式」只对门成立。**

真人 DM 之所以能同时做到「不让玩家卡关」和「不让玩家蒙混过关」，
是因为他的自由度**不是均匀的**：

```
玩家找不到钥匙          → DM 松了     （门）
玩家极难说服骰不过三次   → DM 不松     （门的门槛）
玩家一心要杀特纳        → DM 让他杀，然后特纳爬起来  （C）
```

**LLM 没有这条线。它必须由 A/B/C/D 四个显式标记画出来。**

### 6.1 骰式

```
DiceExpr  = "1d6" | "2d6" | "1d4+2" | "6+1d4+2" | "1d10+2"
SanExpr   = DiceExpr "/" DiceExpr      // "0/1d6" 成功/失败
```

> **作者已替 KP 展开的骰式，照抄。**
> 《鬼屋》的 `6+1d4+2` 是「穿刺武器极难成功造成最大伤害」规则的展开值。
> 引擎**不必理解它的来历**，照抄字符串解析即可。
>
> 反向还原成规则表达式是错误的方向。

### 6.2 ★ Expr（表达式语法）

用于 `Rule.when` 与 `WinCondition.expr`。

```ebnf
expr     = or_expr
or_expr  = and_expr { "||" and_expr }
and_expr = cmp_expr { "&&" cmp_expr }
cmp_expr = add_expr [ ("==" | "!=" | "<" | "<=" | ">" | ">=") add_expr ]
add_expr = mul_expr { ("+" | "-") mul_expr }
mul_expr = unary { ("*" | "/") unary }
unary    = [ "!" ] primary
primary  = number | string | bool
         | var_ref
         | func "(" [ expr { "," expr } ] ")"
         | "(" expr ")"

var_ref  = ident { "." ident }         // damage.type / self.HP / party.size
func     = "floor" | "ceil" | "min" | "max" | "count"
```

**无循环、无递归、无函数定义。**

这是一个纯表达式语言，约二三百行即可实现，且**天然安全**——不会死循环，不会越权。

> 六个模组的所有规则，**没有一条需要循环**。
> 「每战斗轮 1D6 力量伤害」是引擎的回合循环在驱动，规则本身只描述单轮。

**关键约束：`when` 里不能放语义判据**

```jsonc
{"when": "玩家表述得很精彩"}                 // ❌ 求值器算不了
{"when": "check.roleplay_tier == 'excellent'"}    // ✅
```

语义判据由 LLM 在**上游**求值，输出枚举写入状态槽，规则只读枚举。
**软判据与硬求值必须分在两个阶段。**（详见 §7.4）

### 6.4 ★ 物品字符串的来源约定

> **`Character.equipment` 的每个字符串，必须在拾取时就把来源固化进名字。**

```jsonc
// ✅ 正确
"equipment": [
  "从画框后取出的小钥匙",
  "从床底摸到的小钥匙",
  "从通风管掏出的小钥匙"
]

// ❌ 错误：来源分开存
"equipment": ["小钥匙", "小钥匙", "小钥匙"],
"origins":   {"key_1": "画框后", ...}
```

**为什么这样就够了，不需要 `id`**

《银之锁》三把钥匙的原文名称都是「一把小钥匙」——**语义上无法区分**。
这曾是主张引入 `id` 的唯一理由。

写进来源之后，它们成为三个**不同的字符串**。id 的作用（同一性）被自然语言承担了。

**而且玩家看得懂。** `key_painting` 是给引擎看的，「从画框后取出的小钥匙」是给人看的——
**而 LLM 恰好读人话。**

顺带解决了一个 id 方案解决不了的问题：玩家说「我用刚才在画后面找到的那把」，
字符串**直接匹配**。id 方案还得先做一次语义解析。

**锁的一侧同样写来源**

```jsonc
{
  "kind": "object",
  "name": "书房的柜子",
  "secrets": "上锁。需要从画框后取出的那把钥匙。",
  "locked": true
}
```

**约束**

> 来源必须在拾取时就固化进字符串。一旦分开存，就又回到需要 `id` 关联的老路。

LLM 生成这个字符串时会自然带上来源——它刚刚描述完玩家从画框后掏出钥匙。
这几乎不需要额外提示。

**边界**

本约定只适用于**门 I**（见 §5.8.2）。

若物品属于 **D 类**（出现在 `Rule.when` 或 `WinCondition.expr` 里），
字符串本身仍是自由文本，但**门后的值必须落 `entity_states`**。

> 《复足》的梦境之石：`"梦境之石"` 是字符串，`pc.has_dreamstone` 是受保护的 bool。
> **落库的是「值」，不是「物品」。**

## 七、约束与不变式

> 本节是数据模型中**最容易被遗漏、也最有价值**的部分。
> 这些约束无法由类型系统表达，必须由代码或导入校验保证。

### 7.1 信息隔离

> **`Entity.secrets` 永不进入「生成旁白」的 LLM 请求上下文。**

这不是字段约束，是**提示词组装层的工程约束**。

理由：`secrets` 一旦进入上下文就泄漏，**事后收不回**。
这与「LLM 会不会算错」是两类不同的问题。

同理适用于 `Checkpoint.hidden`（暗骰）：让 LLM「假装不知道自己掷出了什么」不可靠。
暗骰必须由 CheckResolver 静默执行，不能创建客户端可见的 `pending_check`；弹出骰子界面本身就会泄露「这里值得检查」。

---

### 7.2 EntityState 的引用完整性

> **`Entity.state` 的每个键，必须至少被一处 `Rule.when` 或 `WinCondition.expr` 引用。**
> 未被引用的状态应改为 `public_persona` 自由文本。

可实现为导入流程中的符号表检查。

---

### 7.3 ★ 状态写入口唯一

> **LLM 不得直接写 GameState。它只能提议 op，由引擎校验后执行。**

```jsonc
// LLM 输出
{
  "narration": "猫扑到男人脸上，指甲深深陷进他的眼睛。",
  "ops": [
    {"op": "modify", "path": "npc.kidnapper.blind", "set": true},
    {"op": "modify", "path": "cat.alive",           "set": false}
  ]
}
```

引擎收到后：
1. 校验 `path` 存在于 `entity_states` 的键空间
2. 校验该 op 在当前场景的 `allowed_ops` 内
3. 执行、记账（写 `Event`）、返回新状态

**为什么这条约束不可省**

六模组中有一类规则，LLM 的遗漏具有**系统性偏向**：

| 模组 | 规则 | 漏掉之后 |
|---|---|---|
| 死者 | 僵尸不会闪避 | 僵尸只是变强了一点 |
| 死者 | 特纳绝不出借小号 | 玩家拿到了小号 |
| 复足 | 手拉手无效 | 大家都活着 |
| 复足 | 蜘蛛海源源不绝 | 玩家杀退了蜘蛛 |
| 银之锁 | 猫必须死 | 玩家救下了猫 |
| 鬼屋 | INT 检定成功 → 更糟 | 玩家过检定，安全 |

**六条全是「限制玩家」的规则。六条漏掉之后玩家都会更开心。这不是巧合。**

且这类失败是**静默的**：

```
引擎约束失败：玩家拿了不该拿的钥匙 → 抽屉打不开 → 玩家立刻发现
提示词约束失败：猫活下来了          → 剧情继续 → 没有任何人发现
```

前者是崩溃，可以修。后者不能——因为你不知道它发生了。

> 因此：**凡是决定玩家能否通关的状态，不能交给一个想让玩家通关的东西来记。**

**D 类的写入不变式**

对每个 `state` 键，除「写入口唯一」外，还可声明**跨实体的不变式**，由引擎在执行 op 时校验：

```jsonc
// 复足
{
  "key": "*.has_dreamstone",
  "invariant": "count(has_dreamstone == true) <= floor(party.size / 2)"
}
```

《复足》「手拉手无效」的实现，**不是禁止 LLM 演绎**，而是这条不变式让 op 被拒。

LLM 仍然可以写「你抓着他的手冲进夜色」。
引擎在结局求值时读到 `pc_002.has_dreamstone == false`——他死了。
**而 LLM 从头到尾没被禁止说任何话。**

> 这是 §5.8.6 与本节的交汇点：
> **A 类靠拒绝 op 的类型（`refuse_ops`），D 类靠拒绝 op 的结果（不变式）。**

---

### 7.4 软判据与硬求值分离

> **`Rule.when` 只能引用 `VariableDef` 表内的变量，不能包含语义判据。**

《蛙蛙村》说服信使的三档难度：

```
只说「我说服他」，不阐述理由  → 困难
讲了个合理的理由            → 普通
讲得精彩                   → 普通 + 奖励骰
```

正确的分工：

```
阶段 1  LLM 求值语义判据 → 输出枚举 check.roleplay_tier ∈ {none, reasonable, excellent}
阶段 2  引擎读枚举 → 查表得 difficulty → 掷骰
```

**枚举硬、判据软。** 若三档不写死，LLM 会自造第四档，或直接放水判成功。

---

### 7.5 World 与 Module 规则同构

> **`World.world_rules` 与 `Entity.rules` 必须是同一个 `Rule` 类型。**

一旦结构不同，就必须写一个合并器——**合并器会成为整个系统最脏的地方**。

**推论：`override` 若反复出现，说明 World 定义得不够。**

《复足》的「撬门（力量 20）」是固定阈值检定，在该模组中出现四次（撬门/破窗/破门/上锁门）。
若让它用四次 `override` 打补丁，是错的——**它是通用机制，应补进 `World.definition` 的检定原语列表**。

Module 的 `override` 应当是稀有的。

---

### 7.6 导入校验：路口清单

> **静默失败只能靠路口清单来兜。**

LLM 生成 `{"then": {"forbid": true}}` 的命中率，明显低于 `{"then": {"scale": 0.5}}`。
它不愿意生成限制玩家的规则。

「僵尸不会闪避」漏掉之后，系统不报错，玩家不知情。

**兜底方式**：导入完成后，对每个 `kind ∈ {npc, monster}` 的实体，逐个质询 19 个 hook——尤其是空着的那些：

```
on_dodge_declare    → 【空】   这个怪物真的能闪避吗？
on_damage_roll      → 火器减半
on_major_wound      → 【空】   这个怪物真的会受重伤吗？
```

> LLM 回答「它有什么**不能做**的事」时很准。**它只是不会主动想起来要问。**

**这正是 hook 表相对「一堆布尔字段」的隐性收益：分母是明确的 19。**
在「塞数据」的世界里，「这个模组我抽取了多少信息」这个问题**没有分母**。

**A/B/C/D 四类的质询清单**

除 hook 空位外，导入流程还应对每个实体逐条追问（对应 §5.8.7 的四个问题）：

```
A  □ 玩家索要/夺取它时，是否无论如何都不能得手？
      → refuse_ops

B  □ 是否存在某个时刻，无论玩家做什么，这件事都必须发生？
      → Rule on on_scene_enter / on_turn_end

C  □ 是否存在某个检定，成功反而导致更坏的结果？
      → Rule on on_check_resolve

D  □ 它是否出现在任何结局条件或规则条件里？
      → entity_states
```

> **B 与 C 的漏报率最高。**
> B 需要 LLM 想象「玩家什么都没做时会发生什么」——这与它的响应式本能相反。
> C 需要它承认「成功是坏事」——这与它的成败先验相反。
>
> 这两类必须由**人工复核**，不能只靠 LLM 自检。

---

### 7.7 数量的动态缩放

> **`ModulePack.players_min/max` 必须能被 `spawn` 表达式引用。**

《复足》：
- 「梦境之石总数约为调查员总数的一半」
- 「冷蛛数量 = 未携带石头的调查员数量」

若写死数字，2 人团与 5 人团的难度完全崩坏。

---

### 7.8 ★ 事件溯源的一致性

> **`events` 是权威，`rooms.entity_states` 与 `characters.*` 是物化视图。**

**执行顺序（必须在同一事务内）**

```
1. 校验 op（路径存在 / 不在 refuse_ops / 不违反 D 类不变式）
2. append events   ← 权威写入
3. 更新物化视图     ← 派生写入
```

**`Event.payload` 的可重建性要求**

每条状态变更事件必须携带**足以重建**的信息：

```jsonc
// ❌ 语义事件，不可重建
{"type": "cat_died"}

// ✅ 状态事件，可重建
{"type": "state_modified",
 "payload": {"path": "cat.alive", "from": true, "to": false},
 "cause": "rule:cat_sacrifice"}
```

> `cause` 字段记录变更来源（`rule:<id>` / `op:<player_id>` / `timer:<id>`），
> 使「`cat.alive` 是在哪一条 event 上变成 `false` 的」成为可回答的问题。

**冲突时以 `events` 为准。** 物化视图损坏时可回放重建。

统一检定请求要求 `Event.payload` 的 discriminated union 至少覆盖以下事件；这些事件不仅用于展示，
还必须能重建 `Player.pending_check`：

| `type` | 必需 payload | 状态效果与可见性 |
|---|---|---|
| `check_requested` | `request_id` `checkpoint_ref` `player_id` `character_id` `options` `difficulty` `targets` `bonus_penalty` `requested_at` `expires_at` | 创建 `pending_check`；向目标客户端私有投递 `check.request` |
| `check_resolved` | `check_id` `request_id?` `checkpoint_ref` `skill_id` `raw` `difficulty` `target` `bonus_penalty` `tier` `roll_mode` `rng_seed/rng_cursor` | `prompt` 时消费 `pending_check`；记录最终结算并生成 ActionResult |
| `check_timeout` | `request_id` `policy` `timed_out_at` `selected_skill_id?` `next_expires_at?` | 记录挂起检定的超时处置：自动掷、取消或重新提醒 |

`check_resolved.raw` 与随机源审计信息只能由 CheckResolver 写入，客户端回传的同名字段一律不作为事实来源。
`rng_seed` / `rng_cursor` 是引擎审计字段，不进入玩家事件投影；`ActionResult` 也只在宿主
内部流转。玩家只收到允许公开的骰面结果、`NarrationOutput` 和 ViewProjector 投影。

`check_timeout` 不能只是通知事件：其 payload 必须足以重建处置后的挂起状态。`policy='cancel'` 时清空槽位；
`policy='reprompt'` 时携带新的 `next_expires_at`；`policy='auto_roll'` 时必须携带按确定性策略选出的
`selected_skill_id`，随后产生关联的 `check_resolved`。
具体采用哪一种默认策略仍需在待办中确定。

---

### 7.9 ★ 求值上下文的房间隔离

> **所有 `Rule.when` / `WinCondition.expr` 的求值，必须在显式 `EvalContext` 内进行。**

```
EvalContext {
  room_id:       RoomId              // 必需
  entity_states: Record<...>         // 来自 rooms.entity_states
  self:          Character | Entity
  party:         Character[]         // WHERE room_id = ?
  combat?:       CombatState
}
```

**引擎不得从任何全局作用域读取实体状态。**

> **这不是数据库问题，是求值器的问题。**
>
> `room_id` 已贯穿所有 GameState 表。但 `party.size` 若实现成
> 「查所有 `characters`」而非「查 `WHERE room_id = ?`」，两局游戏会互相污染。
>
> **《复足》正好会触发这个 bug**：梦境之石数量 = `floor(party.size / 2)`。
> 两个 4 人房间并发时，`party.size` 会算成 8，石头数变成 4。
>
> 而且**它不会报错**——只是所有人都活着走出了宾馆。

---

### 7.10 ★ 统一检定请求与掷骰权威

> **Intent 分离执行路由与检定来源；引擎确定机械事实；Narrator 只展开既成结果。**

1. **两个维度独立**：Intent 阶段输出 `ExecutionRoute` 与 `CheckRoute`。
   `narrative / engine` 决定是否进入 `AtomicActionEngine`；`none / module / default`
   回答是否检定及检定来源。`narrative` 只允许搭配 `none`，`engine` 可以搭配三者；
   不再维护独立的 `requires_check`
2. **来源必须可校验**：`module.checkpoint_id` 必须属于当前 `Scene.checkpoint_ids` 候选集；
   `default` 加载 `World.definition.default_checkpoint`。系统不在运行时生成或写回新的 Content Checkpoint
3. **候选与难度必须在弹窗前硬化**：模组路由使用
   `expand(checkpoint.skill) ∩ character.skills`；默认路由使用
   `normalize(intent.proposed_skills) ∩ character.skills`。难度来自所加载的 Checkpoint；为空时由
   `check.roleplay_tier` 查确定性映射。`difficulty / targets / bonus_penalty` 只写入引擎求出的最终快照
4. **所有可见检定只有一个时机**：`roll_mode='prompt'` 必须先创建 `pending_check`，再接受
   `request_id + skill_id` 完成命令。`pending_check != null` 时不得覆盖式创建第二个请求；
   服务端必须再次校验玩家、房间、角色、请求和技能候选，并保证同一 `request_id` 最多结算一次
5. **服务端权威、事件可重放**：玩家点击「投掷」只表示授权继续；每次骰值均由 CheckResolver 当场生成，
   `raw` 与 `rng_seed` / `rng_cursor` 一同落 Event。客户端不得生成、覆盖或固定结果
6. **暗骰与低打断检定走 `auto`**：`hidden=true ⇒ roll_mode='auto'`，不得创建 `pending_check`，
   也不得向玩家或旁白 LLM 暴露发生了检定或 `raw`。不值得打断节奏的检定可按 World / Checkpoint 策略走 `auto`，
   但 `SanTrigger.kind='direct'` 等非检定效果不得伪造一条检定流水线
7. **完整结算不可绕过**：`prompt` 与 `auto` 都必须经过 `on_check_roll → on_check_resolve`。
   写 `check_resolved`、执行 C 类后果和生成 ActionResult 必须在同一事务内；仅 `prompt` 清空 `pending_check`。
   默认节点只固定“玩家行动成功 / 玩家行动失败”，Narrator 不得改写该分支或产生未声明的机械效果
8. **超时必须显式落事件**：`pending_check.expires_at` 到期后必须写 `check_timeout`；自动掷、取消或重新提醒
   三种策略都要可回放，不能让槽位永久悬挂或由客户端本地清除

完整的路由、默认分支与 Narrator 输入见 [[checkpoint流程设计文档]]。

## 八、逻辑模型 → 物理模型的投影

**两者不是「对齐」关系，是投影关系。**

```
逻辑模型                     物理模型
─────────────────────       ──────────────────────
Entity {                     module_entities (
  id:    EntityId              id     TEXT PK,
  kind:  EntityKind            kind   TEXT,
  state: Record<...>           state  JSONB,    ← 一列吃掉整个 Record
  rules: Rule[]                rules  JSONB     ← 一列吃掉整个数组
}                            )

Rule {                       （不建 rules 表）
  hook:     HookName
  when:     Expr
  then:     Op
  mode:     ...
  priority: int
}
```

`Rule` 在逻辑模型里是**一等公民**，在物理模型里被折叠进一列 JSONB。

**这不矛盾。** 物理模型选择不建 `rules` 表，是因为当前没有「查询所有使用了 `on_dodge_declare` 的规则」这种需求。

若将来出现该需求（如「统计所有模组在哪些 hook 上挂了规则」），把 `rules` 抽成表即可——**逻辑模型一个字不用改**。

> **物理模型会因查询模式而变，逻辑模型不会。**
> 这就是两份文档必须分开的原因。

**折叠决策表**

| 逻辑结构 | 物理承载 | 折叠 / 不折叠的理由 |
|---|---|---|
| `Rule[]` | `entities.rules JSONB` | 无按规则检索的需求 |
| `Condition[]` | `characters.conditions JSONB` | 随角色整体读写 |
| `PendingCheck` | `players.pending_check JSONB` | 唯一挂起检定，随玩家整体读写、由事件重建 |
| `HookDef[]` | `worlds.hooks JSONB` | 行数 ~19，随 world 整体加载 |
| `Record<EntityId, ...>` | `rooms.entity_states JSONB` | 随房间整体读写 |
| `Skill` / `Occupation` / `Weapon` | **独立表** | ★ **不折叠**——前端需按 `category` 检索、分组、排序 |
| `Event` | **独立表** | ★ **不折叠**——需按 `(room_id, id)` 范围查询；且它是权威真相 |

> **`Rule` 折叠、`Skill` 不折叠**，这两个决策看似矛盾，实则同一条原则：
>
> **前端需要按字段检索的东西必须是列；只随宿主整体读写的东西可以是 JSON。**

---

### 8.1 数据表清单

**共 20 张表，分四层。**

#### Reference 层（3 张 · 全局只读 · 分片键 `world_id`）

| 表 | 主键 | 关键列 | 索引 | 前端用途 |
|---|---|---|---|---|
| `skills` | `id` | `world_id` `name` `category` `base` `kind` `parent_id` | `(world_id, category)` `(world_id, name)` | 车卡、技能面板 |
| `occupations` | `id` | `world_id` `name` `skill_points_expr` `skill_refs` `optional_pool` | `(world_id, era)` | 选职业 |
| `weapons` | `id` | `world_id` `name` `category` `skill_ref` `damage` `ammo` `malfunction` | `(world_id, category)` | 装备、战斗 |

#### Content 层（10 张 · 模组只读 · 分片键 `module_id`）

| 表 | 主键 | 关键列 | JSONB 列 | 备注 |
|---|---|---|---|---|
| `worlds` | `id` | `name` | `definition` `hooks` `variables` `world_rules` | 行数极少 |
| `module_packs` | `id` | `world_ref` `title` `version` `players_min/max` `source` | `rules` | `rules` 装全局定时器与因果链 |
| `module_scenes` | `id` | `module_id` `title` `description` `map_ref` | `exits` `checkpoint_ids` | `exits` 可空；`checkpoint_ids` 默认 `[]` |
| ★ `module_entities` | `id` | `module_id` `kind` `name` `content` `public_persona` `secrets` `locked` | `stats` `state` `rules` `refuse_ops` | 吸收 `clues` + `npcs` |
| `module_checkpoints` | `id` | `module_id` `match_hint` `priority` `skill` `difficulty` `hidden` `roll_mode` | `on_success` `on_fail` | `difficulty` 可空；`hidden ⇒ auto` |

## 九、MVP 范围

### 9.1 物品：不建实体，不加 id

`Character.equipment` 保持 `string[]`，采用 §6.4 的**来源约定**。

**这不再是「本期搁置、将来补上」的妥协，而是最终设计。**

理由见 §5.8.2：绝大多数物品是**门 I**——不会卡关，且 LLM 的偏向与作者意图一致。
它们不需要 `id`，来源字符串已承担同一性。

原先主张 `id` 的唯一场景（《银之锁》三把同名钥匙）已被 §6.4 解决。

> **但注意边界**：本条只适用于门 I。
>
> 《复足》的梦境之石属于 **D 类**——它出现在 `WinCondition.expr` 里（携带 = 生，不携带 = 死）。
>
> 石头本身仍是字符串 `"梦境之石"`，但 `pc.has_dreamstone` **必须落 `entity_states`**。
>
> **搁置的是「门」，不是「值」。**
> 判据只有一条：**它出现在任何 `when` 或 `expr` 里吗？**
>
> 若把「物品全部纯文本」当作一刀切规则，《复足》会**静默失败**——
> 三个人手拉手活着走出宾馆，而没人知道哪里错了。

---

### 9.2 规则引擎：分批填充

**hook 清单与 `Rule` 结构（§5.1–5.4）必须在 MVP 就落地。**

但 `rules` 数组的填充可以分批：

| 类别 | 挂载点 | MVP | 理由 |
|---|---|---|---|
| **B 必然** | `on_scene_enter` / `on_turn_end` | ✅ **必须** | 《银之锁》的核心悲剧完全由它承载。缺失 → 结局条件永远无法满足 |
| **C 反转** | `on_check_resolve` | ✅ **必须** | LLM 的成败先验相反，无法用提示词补救 |
| **战斗规则** | `on_damage_roll` / `on_armor_apply` / … | ⏸ 可延后 | 暂写进 `secrets`，由 LLM 执行；用 §7.6 的 19-hook 质询清单人工复核 |

> **B 与 C 不能搁置的原因，与战斗规则不同。**
>
> 战斗规则漏了，僵尸只是变强一点——**有损，可观察，可修**。
> B 漏了，猫活着走完全场——**模组的结局条件永不满足，游戏无法通关**。
> C 漏了，反直觉设计被静默抹除——**玩家永远不知道自己错过了什么**。

> **为什么 hook 表不能一起搁置**
>
> 1. 它是 `rules` 字段的校验依据。没有 hook 清单，写进去的 JSON 无从校验
> 2. 它是架构而非字段。其余改动都是 `ALTER TABLE ADD COLUMN`；若 MVP 用了别的形式（如具名布尔开关），迁移时要重写所有已导入模组
> 3. 六模组已证明它收敛：19 hook / ~15 变量 / ~10 算子，跑完没有增加

### 9.3 后续扩展

| 项 | 触发条件 | 出处 |
|---|---|---|
| `Checkpoint.threshold` | 需支持固定阈值检定 | 复足「撬门（力量 20）」 |
| `Checkpoint.difficulty_expr` | 难度依赖队伍规模 | 死者「1 人 = 极限 / >3 人 = 常规」 |

### 9.4 已删除的字段

| 字段 | 处置 | 理由 |
|---|---|---|
| `module_clues.reachable_by` | **删除** | 见下 |
| `module_npcs.linked_clue_ids` | **删除** | 同上 |
| `rooms.current_scene_id` | 迁移至 `Character.location` | 分头行动 |

**`reachable_by` 为什么必须删**

线索的「内容」与「获取路径」是两个东西：

```
内容    「一小片深灰色羊毛布料」        唯一的，全局的
路径    在书房侦查 / 找管家聊天         多个，且属于场景
```

中间表 `module_scene_clues` 只存 `(scene_id, clue_id)`，**表达不了 `via`**——「在书房靠侦查，在门厅靠跟管家聊」。

于是 `reachable_by` 被迫诞生，把路径信息拉回线索身上。**同一份信息存了两遍，且可能对不上。**

正确归属：路径属于 `Scene.contents`（见 §9.3），内容属于 `Entity`。

**「核心线索至少两条路径」原则降级为警告**

原设计：少于 2 条路径直接拒绝导入。
改为：警告 + 允许 `intentional_single_path: true` 显式标记。

故意的单路径在六模组里很常见：

- 《银之锁》钢钳是救猫的唯一方法
- 《复足》感染只能通过切开囊肿取出幼蛛治疗
- 《死者》说服疯子需极限难度检定——「除此之外的任何结果都毫无作用」

> 《死者》那句是作者在**预防 KP 心软**。硬拒绝的话这三个模组一个都进不来。

## 十、换规则系统时会发生什么

> **本期仅支持 COC 7e。** 本节说明该设计为何不会在换系统时崩塌。

```
COC 的 hook 表:  on_dodge_declare / on_major_wound / on_san_check
OSE 的 hook 表:  on_thac0_calc    / on_saving_throw / on_morale_check
```

变的是：

| 层 | 变化 |
|---|---|
| **Reference** | 新增一批 `world_id = 'ose'` 的行。**表结构不变** |
| **Content** | `worlds.hooks` / `variables` / `world_rules` 的内容 |
| **GameState** | `characters.skills` 的键换一套（本就是开放键） |

**没变的是**：`Rule` 结构、表达式求值器、算子表、四层分片、A/B/C/D 分类——**一行没动**。

> **Reference 层天然支持多规则系统**，因为它以 `world_id` 分片。
> 这也是它必须独立于 `worlds.definition` 的第二个理由（第一个是前端检索，见 §1.1）。

**已知的两处会破**

1. `skills.base` 是 `int`。COC 6e 的「灵感 = INT×5」需要放宽成 `Expr`
2. `SanTrigger` 整表是 COC 特有的。OSE 没有 SAN，该表恒空

> 前者是一行 `ALTER COLUMN`。后者无害——空表不影响任何查询。

> **「哪些字段属于 Core」这个问法从一开始就错了——字段本身就是 Module 的产物。**
>
> - **Core** 定义：规则如何被表达和求值
> - **Module** 定义：有哪些 hook、hook 上有哪些可读变量

## 附录 B：两个判断流程

### B.1 建模：这个东西该不该由引擎接管？（§5.8.7）

```
Q1  玩家发起某个 op 时，引擎必须拒绝吗？        → A：refuse_ops
Q2  某个条件满足时，引擎必须主动触发吗？        → B：Rule / on_scene_enter
Q3  引擎求值出 success 时，要挂坏结果吗？       → C：Rule / on_check_resolve
Q4  它出现在任何 Rule.when 或 WinCondition.expr 里吗？
                                              → D：entity_states

```

### B.2 写文档：这个 JSON 字段该展开吗？

```
Q1  有代码读它内部的键吗？
      没有 → 逻辑模型写「自由文本」，结束
      有   → 继续

Q2  键名能提前枚举吗？
      能   → 展开为具名结构（Rule / Condition / HookDef）
      不能 → 写 Record<string, T> + 约束条款

Q3  有跨字段的不变式吗？
      有   → 写入 §7「约束与不变式」
```

| `Entity.public_persona` | **没有** | — | 自由文本 + §7.1 隔离约束 |
