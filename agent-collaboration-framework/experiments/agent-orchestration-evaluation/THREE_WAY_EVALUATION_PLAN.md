# Agent 编排三方案同条件评估计划

> 状态：待执行
> 日期：2026-07-21
> 目的：在供应商 Agent Runtime、显式 Python 和 LangGraph 三个实现之间建立可复现证据；本文件不是选型结论。

## 1. 前置：先确认是否需要 Agent runtime

当前 `Orchestrator.run()` 没有模型工具调用或多轮工具循环。开始实现第三个实验前，先为当前与近期已承诺功能填写工具清单：

| 能力/用例 | 当前还是规划 | 调用者 | 模型能否选择不调用 | 最大连续轮数 | 是否并行 | 安全/费用边界 |
| --- | --- | --- | --- | ---: | --- | --- |
| 读取/投影 `PlayerView` | 当前 | 应用编排 | 否 | 0 | 否 | 只返回玩家可见投影 |
| `ActionExecutor.execute()` | 当前 | 应用编排 | 否 | 0 | 否 | 唯一权威命令边界 |
| 网络搜索 | 待产品确认 | 待定 | 待定 | 待定 | 待定 | 域名、次数、费用和内容安全待定义 |
| 页面抓取/引用核验 | 待产品确认 | 待定 | 待定 | 待定 | 待定 | SSRF、内容注入与引用策略待定义 |
| 工具调用审核 Agent | 设想，未承诺 | 模型/策略层 | 待定 | 待定 | 否 | 确定性策略门不能被 LLM 替代 |
| fan-out Worker | 设想，未承诺 | Planner | 待定 | 待定 | 是 | 私有上下文与部分失败待定义 |

这里的“轮数”只统计“模型请求工具 → 工具结果回填 → 模型再次决策”的循环。应用代码固定调用一个端口不计为模型工具轮次。

如果经需求确认仍然没有模型自主工具调用，停止三种 Agent runtime 的比较，直接评估基础模型 SDK 的结构化输出即可。

## 2. 三个独立实现

三个项目不得互相导入，也不得依赖当前 TRPG 的提示词、契约或工具：

1. `plain-python-tool-agent`：基础模型 SDK + 手写消息、tool-call 拼接、执行和循环；
2. `langgraph-tool-agent`：LangChain/LangGraph 推荐高层 Agent API；复杂场景需要时再下沉 `StateGraph`；
3. `provider-agent-runtime`：供应商打包的 Agent loop，例如 OpenAI Agents SDK 或 Anthropic tool runner。

第三个项目必须记录：

- runtime 名称、版本和稳定性标记；
- 支持的模型 API，以及 Qwen/OpenAI-compatible endpoint 是否真实可用；
- tracing、session、托管工具或远端执行的默认数据行为；
- 关闭 tracing 或敏感内容采集的方法；
- 哪些循环、错误处理和重试由 SDK 隐式完成。

如果要同时比较 OpenAI 与 Anthropic runtime，应作为两个 provider 子实验报告，不应把它们合并成一个模糊的“SDK 方案”。

## 3. 公平比较约束

在 runtime 允许的情况下固定：

- 同一模型和模型参数；
- 同一业务无关系统提示词；
- 同一工具名称、描述、参数 schema 和返回值；
- 同一输入集与重复次数；
- 同一 `TextDelta`、`ToolCall`、`ToolResult`、`Completed` 外部事件；
- 同一超时、最大轮数、错误样本和终止条件；
- 独立的新虚拟环境，记录锁定版本和安装体积。

若供应商 Runtime 不能接入同一模型，这是选型数据，不应通过偷偷更换模型消除。允许使用该 Runtime 支持的模型验证代码路径，但报告必须分开：

1. **兼容性结果**：是否满足项目模型约束；
2. **实现复杂度结果**：在其支持模型上的代码量；
3. **模型行为结果**：不同模型之间不做工具选择正确率的直接归因。

## 4. 测试场景

### S0：无工具、结构化输出

输入要求模型返回固定 schema 的对象。验证基础模型 SDK 是否已经足够，以及 Agent runtime 带来的额外代码/依赖。

### S1：单轮工具调用

提供 `add_numbers(a, b)`，要求模型自主调用一次后回答。验证参数流拼接、schema 校验、工具执行、结果回填和结构化事件。

### S2：多轮工具调用

提供两个必须串联的工具，例如：

```text
find_timezone(city) → timezone_name
get_current_time(timezone_name) → local_time
```

输入只提供城市，要求至少连续两轮工具调用后回答。验证工具结果能否自然触发下一轮选择、最大轮数与停止条件。

### S3：失败路径

至少覆盖：

- 非法参数类型；
- 未注册工具；
- 工具抛出异常；
- 工具超时；
- 模型持续重复同一工具；
- 达到最大轮数仍未结束。

验证错误是否以可观察、可恢复、不会泄漏内部信息的事件回填。

### S4：工具执行前审核（条件场景）

只有近期需求确认需要第二个 LLM 审核时才纳入选型权重。验证：确定性策略门、审核 Agent、批准/拒绝分支、拒绝原因回填、重试上限和工具未执行保证。

### S5：fan-out/fan-in（条件场景）

只有近期需求确认需要并行 Worker 时才纳入选型权重。固定 Worker 和动态 Worker 分开测试；记录聚合顺序、部分失败、取消、超时、共享状态 reducer 和上下文隔离。

## 5. 数据采集

每个场景至少记录：

| 维度 | 指标 |
| --- | --- |
| 正确性 | 工具选择正确率、参数正确率、任务完成率、越权调用数 |
| 循环 | 模型调用轮数、工具调用轮数、重复调用数、终止原因 |
| 流式 | 首事件时间、事件类型/顺序、不可展示内容泄漏数 |
| 性能 | 端到端耗时；本地 fake model 下的 runtime 开销另测 |
| 代码 | Agent/工具/事件适配 SLOC、圈复杂度、测试 SLOC |
| 依赖 | 直接/传递依赖数、虚拟环境体积、版本冲突 |
| 可靠性 | 超时、失败回填、重试、恢复和幂等表现 |
| 运维 | tracing 数据、可定位性、provider 锁定、升级面 |
| 高级能力 | 审核、checkpoint、interrupt、fan-out/fan-in 的增量代码 |

真实模型端到端计时至少交替随机化实现顺序并增加样本量；公网、排队和模型推理不能被表述为纯框架开销。本地 runtime 开销使用相同 fake model/tool latency 单独测量。

## 6. 决策输出模板

实验结束后单独提交 ADR，至少包含：

1. 当前与近期已承诺需求，以及不在范围内的设想；
2. 三方案版本、可复现命令和原始数据；
3. 每项需求对应的证据权重；
4. 被选方案满足需求的证据；
5. 另外两个方案未被选择的具体反证；
6. provider、tracing、安全与迁移成本；
7. 重新评估触发条件。

在 ADR 合并前，任何文档、Issue 或 PR 都不得把“继续普通 Python”“改用 LangGraph”或“直接采用供应商 SDK”写成项目最终决定。
