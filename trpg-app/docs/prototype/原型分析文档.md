 # 前端原型分析文档
 
 **版本**：v1.0
 **日期**：2026-07-09
 **原型文件**：`docs/prototype/index.html`
 **设计主题**：Arkham Case Files — 1920s 复古案件档案美学
 
 ---
 
 ## 一、原型概述
 
 ### 1.1 范围
 
 本原型覆盖 AI DM 产品的核心用户流程，从进入应用到完成一次游戏交互的完整链路，共计 **7 个页面 + 1 个骰子交互层 + 4 个信息面板**。
 
 ### 1.2 完整用户流程
 
 ```
 登录 ──→ 选择游戏 ──→ 选择世界 (CoC) ──→ 背景故事
    ↓                                                   ↓
 聊天室 ←── 等待大厅 ←── 创建角色 ←────────────────────┘
    ↓
 掷骰交互 (D% / D20 / D6) ←→ 查看角色卡 / 技能 / 地图 / 速记
 ```
 
 ### 1.3 原型文件结构
 
 单一 HTML 文件（1247 行），自包含、不依赖外部构建工具，只需浏览器即可运行。
 
 | 资源 | 说明 |
 |---|---|
 | `index.html` | 完整原型，含 CSS (~500 行) + HTML (~300 行) + JS (~250 行) |
 | Lucide Icons（CDN） | 图标库，用于界面中的按钮和标识 |
 | 无其他外部依赖 | 纯 CSS + 原生 JS，零框架 |

 ## 二、页面分析
 
 ### 2.1 登录页（Login）
 
 **文件位置**：`#s-login`
 
 | 元素 | 设计 | 说明 |
 |---|---|---|
 | 品牌区域 | 金色渐变图标 + 产品名 "ARKHAM CASE FILES" | 强化品牌记忆 |
 | 副标题 | "AI 守秘人 · 智能跑团主持人" | 明确产品定位 |
 | 标签 | "克苏鲁的呼唤 7th 相容" | 快速告知规则系统 |
 | 主操作 | 「扫码加入房间」— 高优先级按钮 | 预期为主要使用场景 |
 | 次操作 | 「创建新房间」| 次要入口 |
 | 浏览入口 | 「浏览已有案件」| 低优先级，文字按钮 |
 
 **设计意图**：
 - 降低决策负担：用户只有三个选择，主按钮视觉最突出
 - "扫码"放在第一个按钮，呼应线上聚会场景（一扫码就上车）
 - 品牌区域占屏幕约 50%，让新用户第一眼就知道这是什么
 
 **建议改进**：
 - MVP 阶段可考虑去掉「浏览已有案件」，减少维护成本
 - 后续可加入房间码快捷入口（已创建房间的用户直接输码进入）
 
 ### 2.2 游戏选择页（Game Selection）
 
 **文件位置**：`#s-games`
 
 | 元素 | 设计 | 说明 |
 |---|---|---|
 | 布局 | 2×2 宫格卡片 | 四款游戏，一目了然 |
 | 跑团卡片 | 底部蓝色边框 + "推荐"徽章 | 高亮主推内容 |
 | 其他卡片 | 灰色边框 + "开发中"徽章 | 明确告知不可用 |
 | 导航 | 左上角返回箭头 | 保持导航一致性 |
 
 **设计意图**：
 - 明确传达产品 Roadmap：四款游戏都在规划中，但目前只有跑团可用
 - "开发中"徽章管理用户预期，防止进入后发现不能用
 - 宫格布局适合触屏，点击区域大
 
 **建议改进**：
 - 可点击的游戏卡片添加微动效（hover 轻微上浮）
 - 长期规划中，此页面将接入动态配置（后端返回可用游戏列表）
 
 ### 2.3 世界选择页（System Selection）
 
 **文件位置**：`#s-system`
 
 两款游戏并排展示为列表卡片，左侧彩色竖条区分：
 
 | 卡片 | 左侧色 | 状态 |
 |---|---|---|
 | 龙与地下城 (DND) | 橙色 `#c08050` | 展示中（不可点击）|
 | 克苏鲁的呼唤 (CoC) | 紫色 `#7050a0` | 可点击，带"推荐新手 · 已就绪"徽章 |
 
 **设计意图**：
 - DND 卡片仅展示不跳转，暗示后续会支持，但不是 MVP 范围
 - CoC 卡片带推荐标签，引导新用户选择
 - 明确告知"目前仅开放克苏鲁"，降低认知负担
 
 ### 2.4 背景故事页（Story Intro）
 
 **文件位置**：`#s-story`
 
 这是最具氛围感的一个页面，与前面页面的浅色复古风形成强烈反差：
 
 | 设计决策 | 实现方式 |
 |---|---|
 | 深色氛围 | `background: linear-gradient(180deg, #1a1620, #0d0b10)` |
 | 紫光晕 | `radial-gradient(ellipse at 50% 30%, rgba(112,80,160,0.08))` |
 | 打字机标签 | `font-family: 'Courier New'` + 字母间距 0.15em |
 | 渐入动画 | 文本 `opacity: 0` → `fadeIn 2s`，按钮 4s 后出现 |
 
 **设计意图**：
 - 用视觉反差告诉用户：故事要开始了
 - 文字渐入 + 按钮延迟出现，让玩家有一段时间沉浸阅读，而不是跳过
 - 案件编号 #1927-03 统一世界观，暗示这是一个系列
 
 **建议改进**：
 - 后续可加入"跳过"按钮（目前需要等 4 秒动画）
 - 可考虑加入打字机逐字出现效果增强沉浸感
 - 背景可加入动态粒子/迷雾效果（使用 Canvas 或 CSS）
 
 ### 2.5 角色创建页（Character Creation）
 
 **文件位置**：`#s-character`
 
 角色创建拆分为三个逻辑步骤，用进度条可视化管理：
 
 | 区块 | 内容 | 交互 |
 |---|---|---|
 | 步骤进度 | 4 段进度条（第 2 步高亮） | 视觉反馈当前进度 |
 | ① 选择职业 | 6 格宫格：侦探/记者/教授/医生/警察/古董商 | 点击选中，金色边框高亮 |
 | ② 基本信息 | 姓名 + 年龄 + 出生地 | 输入框编辑 |
 | ③ 基础属性 | 8 项属性的 ± 按钮 (STR/CON/SIZ/DEX/INT/POW/APP/EDU) | 点击调整，范围 15-99 |
 
 **设计意图**：
 - 拆步骤降低认知负荷：新人不会被 8 项属性 + 技能 + 装备同时吓到
 - 职业选择优先，因为职业决定了核心技能组
 - 属性使用 ± 按钮而非直接输入，更适合移动端触屏操作
 - 属性值范围限制 15-99，符合 CoC 7th 规则
 
 **建议改进**：
 - 技能点分配界面尚未实现（原型中只有属性）
 - 角色卡预览缺失（玩家看不到最终角色长什么样）
 - 可加入"随机创建"按钮降低新手上手门槛
 - 职业选择后可自动推荐预设属性分配

 ## 三、骰子系统分析（核心交互）
 
 ### 3.1 功能定位
 
 骰子系统是产品的核心交互组件，也是 AI 无法替代的**确定性规则执行**环节。原型将掷骰设计为一个**独立的全屏交互页面**，而非简单的按钮点击。
 
 ### 3.2 三种骰子模式
 
 | 模式 | 物理表现 | 判定逻辑 | 适用场景 |
 |---|---|---|---|
 | **D%** | 两个 d10 十面骰（十位+个位） | CoC 7th 规则：大成功(1) / 极限(≤skill/5) / 困难(≤skill/2) / 普通(≤skill) / 失败 / 大失败(100) | 主骰子，所有技能检定 |
 | **D20** | 六边形二十面体骰 | 显示数值，无规则判定 | DND 检定 / 自由投 |
 | **D6** | 方形六面骰 | 显示数值，无规则判定 | 伤害骰 / 自由投 |
 
 ### 3.3 交互流程
 
 ```
 触发掷骰 ──→ 全屏骰子页面 ──→ 手指按住骰子拖动 ──→ 松手投出
   │                                         │
   │  · AI 消息中的「拿起骰子」按钮           │  · 持续摇晃动画
   │  · 底部快捷栏的「🎲 掷骰」               │  · 摇动计 0→5
   │  · 快捷行动中的「掷骰」                  │  · 金色光环反馈
   │                                         │
   ▼                                         ▼
 确认结果 ──── 结果显示 ──── 骰子飞出 ──── 动画翻滚
   │                · 大数字显示
   │                · 判定等级标签
   │                · 详细公式
   ▼
 结果发送到聊天
 ```
 
 ### 3.4 物理交互实现分析
 
 原型中实现的拖曳-摇动-投掷系统包含以下关键技术点：
 
 | 机制 | 实现方式 | 效果 |
 |---|---|---|
 | **位置跟踪** | `touchmove` / `mousemove` 事件 + 阻尼位移 (`dx * 0.3`) | 骰子跟随手指移动，但带有惯性感 |
 | **摇晃检测** | 方向变化计数器 (每 25ms 检测一次 `dirX`/`dirY` 改变) | 真实检测"来回拖动"动作 |
 | **持续震动** | `requestAnimationFrame` 循环 (约 60fps) | 拖曳时骰子持续抖动，幅度随摇动强度递增 |
 | **双骰独立抖动** | 正弦波相位偏移 (`Date.now() * 0.02` vs `* 0.85`) | 两个 d10 各自抖，看起来更自然 |
 | **发光反馈** | `box-shadow` 强度动态计算 (`0.08 + intensity * 0.35`) | 摇动越强骰子越亮 |
 | **投掷物理** | 反方向抛出 + 三次贝塞尔缓动 + 旋转圈数随力度变化 | 松手后骰子沿拖动反方向飞出 |
 | **数字闪烁** | `setInterval 70ms` 快速切换随机数字 | 模拟骰子滚动过程中的数字变化 |
 
 ### 3.5 摇动计设计
 
 5 段进度条，从 0 → 5 反映摇动强度：
 
 | 等级 | 触发条件 | 视觉反馈 |
 |---|---|---|
 | 1 | 轻轻拖动 | 骰子跟随手指 |
 | 2 | 2-3 次方向变化 | 金色光环出现，轻微震动 |
 | 3 | 4-5 次方向变化 | 明显震动，提示"再用力一点" |
 | 4 | 6-7 次方向变化 | 强光 + 快速抖动 |
 | 5 | 8+ 次方向变化 | 最强光效 + "松手投出！" |
 
 ### 3.6 结果展示
 
 D% 模式结果展示两个骰子的值及总和：
 
 ```
    30   +   2   =   32
  ┌────┐    ┌────┐    ┌────┐
  │ 30 │    │ 2  │    │ 32 │  ← 金色大数字
  └────┘    └────┘    └────┘
        成功           ← 绿色标签
  侦察 65% · 30+2=32 · 需求 ≤65  ← 详细公式
 ```

 ## 六、技术实现分析
 
 ### 6.1 页面路由
 
 使用简单的 `display: none/block` 切换 + CSS animation 实现页面过渡，无路由库：
 
 ```javascript
 function goTo(id) {
   document.querySelectorAll('.screen').forEach(s => s.classList.remove('active'));
   document.getElementById(id).classList.add('active');
   window.scrollTo({ top: 0, behavior: 'smooth' });
 }
 ```
 
 **特点**：
 - 所有页面在同一个 DOM 中，切换无网络请求
 - 页面切换带 `opacity + translateY` 动画（0.4s ease）
 - 适合原型阶段，生产环境应替换为 SPA 路由
 
 ### 6.2 骰子系统架构
 
 ```
 ┌─────────────────────────────────────┐
 │          事件层                       │
 │  mousedown / touchstart → onGrab()  │
 │  mousemove / touchmove → onMove()   │
 │  mouseup / touchend → onRelease()   │
 └──────────────┬──────────────────────┘
                │
 ┌──────────────▼──────────────────────┐
 │          状态层                       │
 │  isGrabbed, shakeLevel, direction-  │
 │  Changes, baseX/Y, lastResult       │
 └──────────────┬──────────────────────┘
                │
 ┌──────────────▼──────────────────────┐
 │          动画层                       │
 │  shakeLoop() → requestAnimation-   │
 │  Frame → 独立抖动 + 发光           │
 │  throwDice() → CSS transition +    │
 │  setInterval 数字闪烁               │
 └──────────────┬──────────────────────┘
                │
 ┌──────────────▼──────────────────────┐
 │          渲染层                       │
 │  CSS transform: translate + rotate  │
 │  3D shapes via clip-path           │
 │  box-shadow glow                    │
 └─────────────────────────────────────┘
 ```
 
 ### 6.3 动画性能分析
 
 | 动画 | 方式 | GPU 加速 | 性能影响 |
 |---|---|---|---|
 | 页面切换 | `opacity + transform` | ✅ 是 | 低 |
 | 骰子抖动 | `transform: translate + rotate` | ✅ 是 | 低（~60fps） |
 | 骰子发光 | `box-shadow` 变化 | ❌ 否 | 中（每帧重绘） |
 | 数字闪烁 | `textContent` 更新 | ❌ 否 | 极低 |
 
 **建议优化**：
 - 发光效果生产环境可改为 `filter: drop-shadow()` 或叠加层
 - 抖动频率可降至 30fps（每两帧一次），人眼感受不到差异但性能翻倍
 
 ### 6.4 响应式设计
 
 原型采用 Mobile-first 策略：
 
 - 默认宽度 100%，最大宽度 390px（iPhone 标准宽度）
 - 桌面端（≥500px 视口）增加手机外壳效果（32px 圆角 + 8px 边框 + 阴影）
 - 所有尺寸使用 `px` 单位，适合原型阶段
 - 生产环境应转为 `rem` + 媒体查询

 ## 七、用户体验分析
 
 ### 7.1 设计亮点
 
 1. **一致性**：所有页面共享同一套色彩、圆角、字体体系，品牌感强
 2. **氛围感**：故事页的深色渐变 + 渐入动画，骰子页的桌面纹理 + 发光效果，沉浸感好
 3. **物理反馈**：骰子拖曳的抖动、摇动计、松手飞出的抛物线，让数字操作有了物理实感
 4. **信息层级**：每个页面只做一件事，不堆砌
 5. **容错设计**：每个关键操作都有兜底（AI 消息中的掷骰按钮 + 快捷栏 + 快捷行动）
 
 ### 7.2 待改进点

 |---|---|---|
 | 故事页 4 秒动画不可跳过 | 中 | 加入"点击跳过"或"点击继续" |
