## 一、范围声明

**本次模块拆分只覆盖后端**（`core` + `server`），不包含前端 `web` 包。

- 前端由团队另一位成员负责（`TRPG-master` 仓库），本次不代为规划其内部模块结构。
- 前后端的边界契约是 [[API接口对齐规范]]，不是本文档。
- **架构以后端为准**：前端现有代码（`TRPG-master/trpg-app`）经实地核查，发现协议假设与当前锁定的后端架构存在多处分歧（账号模型、掷骰权威归属、WS 协议形状等，详见 [[2026-07-12 架构规范自查与前端代码实际核对]]）。这些分歧由前端向后端架构对齐，不反过来迁就前端现状。

## 二、包结构总览

技术栈落地按 ADR-9：后端 Python。划分判据按 [[架构设计参考 1]]"模块拆分"一节原文三条：①表的边界决定模块边界；②代码量大的功能值得独立成模块；③耦合度——一个模块大量依赖另一个模块私有 API 说明拆分不合理。

| 包 | 对应架构层 | 职责 | 接口（详见 master §4.5，不重复定义） | 划分理由 |
|---|---|---|---|---|
| `core/content` | Content（只读） | 加载模组包 | `ContentRepo` | 叶子模块，独立表域（worlds/module_packs/…），不需要再拆 |
| **`core/state`** 🆕 | 运行时状态访问层 | 跨房间隔离唯一入口、事件溯源 | `GameStateRepo`、`EventLog` | 🔧 从初版 `core/rules` 拆出——两者操作同一组表（rooms/players/characters/notes/events/entity_states），master 原文自己说 `EventLog`"实现上是 `GameStateRepo` 管辖范围内的一张表"，按表边界该是同一模块，但职责（数据访问）跟"规则怎么算"完全不同，混装违反耦合判据 |
| `core/rules` | L4 领域核心·确定性（系统心脏） | 动作裁决、检定、Hook/Rule/Op/Expr 规则引擎 | `RulesEngine`、`CheckResolver` | 全系统代码量最大最复杂的一块（12个hook+表达式语言+Condition/Ledger调度），按"代码量大独立成模块"单列；依赖 `core/state` 读写状态，不再跟数据访问层混装 |
| `core/view` | 权限唯一出口 | 从 GameState 裁剪出 PlayerView / SceneActionMenu | `ViewProjector` | 不变 |
| `core/ai` | L5 AI 适配层（实时角色） | 意图理解、表达生成、答疑、SAN临场判定、复盘摘要 | `IntentParser`、`Narrator`、`QAResponder`、`SanJudge`、`SummaryGenerator`、`ModelRoleConfig`（按角色查表选供应商/型号，无独立 `ModelAdapter` 接口） | 内部按文件（不再拆子包）：`intent.py`/`narrator.py`/`qa.py`/`san_judge.py`/`summary.py`/`adapters.py`(STT+ImageGen)/`config.py`(ModelRoleConfig)——每个角色体量是一个类+一两个方法，拆子目录是形式主义 |
| **`core/moduleimport`** 🆕 | L5 AI 适配层（离线角色，独立） | 模组导入解析 | `ModuleImportAgent`（桩） | 🔧 从初版 `core/ai` 独立出来——AI 编排详细设计原文反复强调这是"离线异步、不进实时回合循环、耦合度远低于核心游戏循环的独立模块"，这个判断本身就是"该独立成模块"的理由，混在 `core/ai` 里会模糊它跟 Intent/Narrator 等实时角色的本质区别 |
| `core/orchestrator` | L3 编排层 | 串起一回合的 Intent→Rules→View→Narrator 全流程 | `Orchestrator` | 不变 |
| `server` | L2 接入层 | WS 会话、房间生命周期、REST 端点 | `Gateway`（对应 API 对齐规范全部端点） | 不变；内部可再按 rest/ws/auth 分文件，属实现细节，不影响模块级设计 |

**Python 里"暴露接口、藏实现"的具体做法**：每个包是一个目录（如 `core/rules/__init__.py`），`__init__.py` 只导出该包对外该公开的类型（如 `RulesEngine`/`Rule`/`Op`），内部 Hook 分发逻辑、Expr 求值器实现细节都是包私有，外部只能 `from core.rules import RulesEngine`，拿不到内部实现——这是单体架构里对应"每个模块暴露一个接口"的做法，不需要网络服务边界也能做到。

## 三、模块依赖关系（依赖倒置，避免循环耦合）

```
core/content      ← core/rules  ← core/orchestrator ← server
core/state        ← core/rules
core/state        ← core/view   ← core/orchestrator ← server
core/ai           ← core/orchestrator ← server（Intent/Narrator）
core/ai           ← server（QAResponder，qa.ask 不占回合、不经 Orchestrator）
core/ai.SanJudge  ⇠ core/rules（依赖倒置，见下方"⚠️ 发现的真实依赖"）
core/moduleimport ← server（模组导入端点，独立分支，不进回合循环）
core/content      ← core/moduleimport（导入产出最终落进 ContentRepo 管的表）
```

- `core/content`/`core/state` 都是叶子模块（纯数据访问，不依赖其他 core 包）。`core/content` 被 `rules`（规则/Hook注入）和 `moduleimport`（导入产出落库）共同依赖；`core/state` 被 `rules`/`view` 共同依赖。
- `core/rules` 依赖 `core/content`（只读加载模组定义）+ `core/state`（`GameStateRepo.load/save`、`EventLog.append`），不依赖 `core/view`。
- `core/view` 依赖 `core/state` 暴露的 `GameState` 类型（只读投影），不依赖 `core/ai`/`core/rules`。
- `core/ai`（实时角色）只依赖 `core/view`/`core/rules` **暴露的数据类型**（`PlayerView`、`ActionResult`、`SceneActionMenu`），不反向依赖其内部实现——这是通信铁律一（类型级唯一出口）在模块依赖层面的体现。
- `core/moduleimport` 只依赖 `core/content`（产出最终落库的地方），不依赖 `rules`/`view`/`ai`/`orchestrator`——它是离线管线，跟实时回合循环完全解耦。
- `core/orchestrator` 是唯一同时依赖 `core/rules`/`core/view`/`core/ai` 的模块，对下游做进程内同步调用（§2.1 通信风格）。
- `server` 依赖 `core/orchestrator`（`handleTurn`）、`core/ai`（`QAResponder`）、`core/content`（模组浏览端点）、`core/state`（`GameStateRepo`/`EventLog`，供 replay/summary 等 REST 读取）、`core/moduleimport`（导入端点）。

**⚠️ 发现的真实依赖（初版遗漏，本轮修正）**：master §4.5 `SanJudge` 接口注释明确写着"仅在没有预设 SanTrigger 命中时**由 RulesEngine 调用**"——也就是说 `core/rules`（确定性核心）运行时真的要调用 `core/ai` 的一个接口，这跟"领域核心可脱离 AI 独立单测"（§二末尾">领域核心（L4）是纯确定性的心脏，可脱离AI与网络独立单测"）这条设计原则有张力，必须用**依赖倒置**处理：`core/rules` 不 import `core/ai` 的具体实现，只依赖 `SanJudge` 这个**接口类型**（可以让这个类型定义留在 `core/rules` 自己的模块里，`core/ai` 实现它并在启动时注入进 `RulesEngine`，而不是反过来 `core/rules` 主动 import `core.ai.san_judge`）；单测 `RulesEngine` 时注入一个假的 `SanJudge` 桩即可脱离真实 AI 跑通。上图 `⇠` 记号表示这条是"通过接口注入"的依赖，不是直接 import，方向上仍然是 `core/ai` 实现、`core/rules` 定义契约。

不存在循环依赖。

## 四、MVP 实现程度

| 模块 | MS1 状态 |
|---|---|
| `core/content` | 真实现（内置 1 个最小模组） |
| `core/state` | 真实现（`GameStateRepo`/`EventLog` 是跨房间隔离唯一入口，不能是桩） |
| `core/rules`（含战斗规则引擎字段） | 真实现；战斗规则字段建出、暂不填充（走 LLM 自由文本，ADR-11 末尾口径）；`SanJudge` 依赖以接口桩注入，真实模型接入进度跟 `core/ai` 一致 |
| `core/view` | 真实现 |
| `core/ai` | 先桩后接模型（各角色接口按 `ModelRoleConfig` 查表选供应商/型号，见 §六.3） |
| `core/moduleimport` | 桩（`ModuleImportAgent.parse()` 粗粒度接口，内部四阶段管线明确推迟到本轮之后，见 AI 编排详细设计待办） |
| `core/orchestrator` | 真实现（逻辑简单但是全流程枢纽） |
| `server` | 最小可回显 |

## 五、落地目录结构

```
packages/
  core/
    content/            # ContentRepo
    state/              # 🆕 GameStateRepo, EventLog
    rules/              # RulesEngine, CheckResolver, Hook/Rule/Op/Expr 引擎
    view/                # ViewProjector
    ai/
      intent.py          # IntentParser
      narrator.py        # Narrator（旁白+NPC共用，persona 参数区分）
      qa.py               # QAResponder
      san_judge.py        # SanJudge（供 core/rules 依赖倒置注入）
      summary.py          # SummaryGenerator
      adapters.py         # STTAdapter, ImageGenAdapter
      config.py           # ModelRole, ModelRoleConfig
    moduleimport/        # 🆕 ModuleImportAgent（桩）
    orchestrator/         # Orchestrator
  server/                # Gateway，暴露 REST/WS（对应 API 对齐规范），内部可再按 rest/ws/auth 分文件
```

按 ADR-9，落地时用 Python 包结构（`core`/`server` 换成 Python 包，字段 `camelCase`→`snake_case`，接口用 Pydantic/typing 表达 master §4.5 的 TS 记法）。每个包的 `__init__.py` 只导出该包的公开接口类型，内部实现细节不对外暴露（见 §二末尾说明）。
